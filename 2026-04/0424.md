📑  Zustand & React Query 활용

1. 프로젝트 개요
- 목적: Zustand를 활용한 클라이언트 상태 관리와 TanStack Query(React Query)를 이용한 서버 상태 관리 기법 학습 및 비교
- 주요 기능: 기본적인 상태 변경, 비동기 데이터 로딩, 로컬 스토리지 연동(Persist), 서버 데이터 캐싱 및 CRUD 테스트
1. 기술 스택
- Framework: Next.js 16 (App Router), React 19
- State Management: Zustand 5
- Data Fetching: TanStack Query 5 (React Query)
- Mock Backend: JSON Server (db.json)

---

1. 주요 학습 내용 (Zustand)

① Basic Store (useCartStore, useUIStore)

- 기능: 객체/배열 상태 변경, 모달/사이드바 등 UI 상태 관리
- 특징: 불변성 유지를 위해 기존 배열을 복사하여 업데이트하는 패턴 학습 (...state.items)

② Async Store (useUserStore)

- 기능: Store 내부에서 async/await를 사용하여 비동기 통신 로직(fetch)을 직접 처리
- 특징: loading 상태를 직접 관리하며 통신 전후로 set을 호출하는 패턴

③ Persist Middleware (usePersistStore)

- 기능: 상태를 브라우저의 localStorage에 자동으로 저장하고 불러오기
- 활용: 다크모드/라이트모드 테마 설정 등 브라우저를 새로고침해도 유지되어야 하는 데이터에 활용

---

1. 주요 학습 내용 (React Query)

① Provider 설정 (Providers.js)

- QueryClientProvider를 통한 전역 설정
- staleTime 옵션을 활용하여 데이터의 "신선도" 관리 및 불필요한 재요청 방지(캐싱)

② Declarative Fetching (useQuery)

- queryKey를 통한 데이터 고유 관리 (캐싱 키)
- isLoading, isError 등 통신 상태에 따른 선언적 UI 렌더링 방식 학습

---

1. 핵심 포인트: Zustand vs React Query

| 구분 | Zustand (Client State) | React Query (Server State) |
| --- | --- | --- |
| 관리 대상 | UI 상태 (모달, 토글, 장바구니 등) | 서버 데이터 (DB, API 응답) |
| 비동기 처리 | 직접 로딩/에러 상태 관리해야 함 | 로딩/에러/성공 상태 자동 관리 |
| 데이터 유지 | 새로고침 시 사라짐 (persist 없으면) | 캐싱으로 데이터 유지 및 재사용 |
