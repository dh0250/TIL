## 1. `@Query` / `@NativeQuery` — 직접 쿼리 작성

쿼리 메소드 명명 규칙으로 표현하기 어려운 복잡한 쿼리는 `@Query`로 직접 작성

```java
public interface CategoryRepository extends JpaRepository<Category, Integer> {

    // 방법 1 — JPQL (엔티티/필드명 기준)
    @Query("SELECT c FROM Category c ORDER BY c.categoryCode")
    List<Category> findAllCategory();

    // 방법 2 — Native Query (실제 SQL, 테이블/컬럼명 기준)
    @Query(
        value = "SELECT category_code, category_name, ref_category_code FROM tbl_category ORDER BY category_code",
        nativeQuery = true
    )
    List<Category> findAllCategory();

    // 방법 3 — @NativeQuery (Spring Data JPA 3.x 신규 어노테이션, nativeQuery=true 축약형)
    @NativeQuery("SELECT category_code, category_name, ref_category_code FROM tbl_category ORDER BY category_code")
    List<Category> findAllCategory();
}
```

| 방식 | 어노테이션 | 쿼리 언어 | 기준 |
| --- | --- | --- | --- |
| JPQL | `@Query` | JPQL | 엔티티 클래스명 / 필드명 |
| Native Query | `@Query(nativeQuery = true)` | SQL | 실제 테이블명 / 컬럼명 |
| Native Query (단축) | `@NativeQuery` | SQL | 실제 테이블명 / 컬럼명 (3.x+) |

> JPQL은 DB 벤더 독립적이고, Native Query는 DB 종속적이지만 복잡한 SQL 그대로 사용 가능

## 2. Dirty Checking (변경 감지) 기반 수정

`@Transactional` 안에서 조회한 엔티티는 **영속 상태**이므로 `save()` 호출 없이도 필드를 바꾸면 트랜잭션 종료 시 자동으로 UPDATE 발생

```java
@Transactional
public MenuResponseDTO modifyMenu(int menuCode, MenuRequestDTO requestDTO) {

    // 1. 영속 상태로 조회
    Menu foundMenu = menuRepository.findById(menuCode)
            .orElseThrow(() -> new IllegalArgumentException("수정할 메뉴가 존재하지 않습니다."));

    Category newCategory = categoryRepository.findById(requestDTO.getCategoryCode())
            .orElseThrow(() -> new IllegalArgumentException("존재하지 않는 카테고리 입니다."));

    // 2. 엔티티 내부 메소드로 상태 변경 (setter 노출 X)
    foundMenu.modify(
            requestDTO.getMenuName(),
            requestDTO.getMenuPrice(),
            newCategory,
            requestDTO.getOrderableStatus()
    );

    // 3. save() 호출 없음 — 트랜잭션 커밋 시 변경 감지로 자동 UPDATE
    return modelMapper.map(foundMenu, MenuResponseDTO.class);
}
```

> 핵심: `@Transactional` 범위 안에서 영속 엔티티의 필드가 바뀌면 JPA가 스냅샷과 비교해서 커밋 전 자동 UPDATE

## 3. `save()` 내부 동작 — INSERT vs UPDATE 분기

```java
Menu savedMenu = menuRepository.save(newMenu);  // INSERT or UPDATE?
```

| 상황 | 동작 | 내부 호출 |
| --- | --- | --- |
| PK가 `0` (int) 또는 `null` (Integer) | INSERT | `EntityManager.persist()` |
| PK에 값이 있음 | UPDATE (merge) | `EntityManager.merge()` |

- `@GeneratedValue`를 사용하면 새 엔티티는 PK가 `0`/`null` → 자동으로 `persist()`
- PK를 직접 세팅한 엔티티를 `save()` 하면 → `merge()` (SELECT 후 UPDATE)

## 4. 삭제 — `existsById()` + `deleteById()` 패턴

```java
@Transactional
public void deleteMenu(int menuCode) {

    // 존재 확인 후 명시적 예외 처리
    if (!menuRepository.existsById(menuCode)) {
        throw new IllegalArgumentException("삭제할 메뉴가 존재하지 않습니다.");
    }

    menuRepository.deleteById(menuCode);
}
```

> `delete(entity)` 대신 `deleteById(id)`를 쓰면 엔티티를 먼저 조회하지 않아도 됨
> `existsById()` → COUNT 쿼리 1회 / `findById()` → SELECT 쿼리 1회, 용도에 맞게 선택

## 5. CRUD REST API — HTTP 상태코드 설계

```java
// 조회 — 200 OK
@GetMapping("/{menuCode}")
public ResponseEntity<MenuResponseDTO> findMenuByCode(@PathVariable int menuCode) {
    return ResponseEntity.ok(menuService.findMenuByMenuCode(menuCode));
}

// 등록 — 201 Created
@PostMapping
public ResponseEntity<MenuResponseDTO> registMenu(@RequestBody MenuRequestDTO request) {
    MenuResponseDTO newMenu = menuService.registMenu(request);
    return ResponseEntity.status(HttpStatus.CREATED).body(newMenu);
}

// 수정 — 200 OK
@PutMapping("/{menuCode}")
public ResponseEntity<MenuResponseDTO> modifyMenu(@PathVariable int menuCode,
                                                  @RequestBody MenuRequestDTO requestDTO) {
    return ResponseEntity.ok(menuService.modifyMenu(menuCode, requestDTO));
}

// 삭제 — 204 No Content (body 없음)
@DeleteMapping("/{menuCode}")
public ResponseEntity<Void> deleteMenu(@PathVariable int menuCode) {
    menuService.deleteMenu(menuCode);
    return ResponseEntity.noContent().build();
}
```

| HTTP 메소드 | 상태코드 | 의미 |
| --- | --- | --- |
| GET | 200 OK | 조회 성공 |
| POST | 201 Created | 리소스 생성 성공 |
| PUT | 200 OK | 수정 성공 (수정된 리소스 반환) |
| DELETE | 204 No Content | 삭제 성공 (반환 body 없음) |

## 6. Stream 변환 — `collect()` vs `toList()`

```java
// Java 8+ 방식 — 가변 리스트 반환
return menuList.stream()
        .map(menu -> modelMapper.map(menu, MenuResponseDTO.class))
        .collect(Collectors.toList());

// Java 16+ 방식 — 불변 리스트 반환 (더 간결)
return menuList.stream()
        .map(menu -> modelMapper.map(menu, MenuResponseDTO.class))
        .toList();
```

| 방식 | Java 버전 | 반환 리스트 특성 |
| --- | --- | --- |
| `.collect(Collectors.toList())` | 8+ | 가변(mutable), add/remove 가능 |
| `.toList()` | 16+ | 불변(immutable), UnsupportedOperationException 발생 |

## 7. application.yml — SQL 포매팅 설정

```yaml
spring:
  jpa:
    show-sql: true          # 실행 SQL 콘솔 출력
    properties:
      hibernate:
        format_sql: true    # 출력 SQL 들여쓰기/줄바꿈으로 가독성 향상
```

- `show-sql: true` 만으로는 SQL이 한 줄로 출력됨
- `format_sql: true` 추가 시 JOIN, WHERE 등 절마다 줄바꿈되어 가독성 향상

## 핵심 요약 (추가분)

| 항목 | 내용 |
| --- | --- |
| `@Query` | JPQL로 직접 쿼리 작성, 엔티티/필드명 기준 |
| `@Query(nativeQuery=true)` / `@NativeQuery` | 실제 SQL 작성, 테이블/컬럼명 기준 |
| Dirty Checking | `@Transactional` 내 영속 엔티티 변경 → `save()` 없이 자동 UPDATE |
| `save()` 분기 | PK 없으면 INSERT(`persist`), 있으면 UPDATE(`merge`) |
| `deleteById()` | 엔티티 조회 없이 PK만으로 삭제, `existsById()`로 사전 검증 |
| HTTP 상태코드 | POST→201, DELETE→204 No Content, GET/PUT→200 |
| `.toList()` | Java 16+ 불변 리스트, `.collect(Collectors.toList())`는 가변 리스트 |
| `format_sql: true` | SQL 출력 시 들여쓰기/줄바꿈 적용, 디버깅 편의성 향상 |
