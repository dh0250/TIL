# chap03 — 연관 관계 매핑 (Association Mapping)

## 개념

객체의 참조와 테이블의 외래 키(FK)를 매핑하여 객체 간의 관계를 DB에 반영하는 과정

- 연관 관계의 주인: 외래 키를 관리하는 엔티티 (보통 N쪽이 주인)
- 로딩 전략: 연관된 객체를 가져오는 시점 결정 (EAGER/LAZY)
- 영속성 전이(Cascade): 부모의 상태 변화를 자식에게 전파하는 설정

## 주요 매핑 어노테이션

| 어노테이션 | 설명 | 주요 속성 |
| --- | --- | --- |
| `@ManyToOne` | 다대일(N:1) 관계 매핑 | `fetch`, `cascade`, `optional` |
| `@OneToMany` | 일대다(1:N) 관계 매핑 | `mappedBy`, `fetch`, `cascade` |
| `@JoinColumn` | 외래 키(FK) 컬럼 지정 | `name`: DB 컬럼명 (기본값: 필드명_PK명) |

---

## Fetch 전략 (로딩 방식)

| 전략 | 설명 | 기본값 |
| --- | --- | --- |
| **EAGER (즉시)** | 엔티티 조회 시 연관된 객체도 JOIN으로 함께 조회 | `@ManyToOne`, `@OneToOne` |
| **LAZY (지연)** | 연관된 객체를 실제 사용할 때(Getter 호출 등) 조회 | `@OneToMany`, `@ManyToMany` |

---

# section01 — 다대일 (Many-to-One)

## 엔티티 매핑 예시 (Menu.java)

```java
@ManyToOne(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
@JoinColumn(name = "categoryCode")
private Category category;
```

## 특징

- 권장 방식: DB의 FK 구조와 객체의 참조 구조가 일치하여 가장 많이 사용됨
- 성능 최적화: FetchType.LAZY를 통해 불필요한 조인을 방지할 수 있음
- 영속성 전이: CascadeType.PERSIST 설정 시 메뉴 저장 시점에 새로운 카테고리도 함께 DB에 저장됨

---

# section02 — 일대다 (One-to-Many)

## 엔티티 매핑 예시 (Category.java)

```java
@OneToMany(fetch = FetchType.EAGER, cascade = CascadeType.PERSIST)
@JoinColumn(name = "categoryCode") // FK가 상대 테이블(Menu)에 있음을 명시
private List<Menu> menuList;
```

## 특징

- 단방향 구조: 일(1) 측에서 다(N) 측을 참조하고 관리함
- 단점: 매핑 주체가 아닌 반대편 테이블(Menu)에 외래 키가 있어, 데이터 수정 시 추가적인 UPDATE 쿼리가 발생하여 성능상 불리할 수 있음

---

# section03 — 양방향 매핑 (Bidirectional)

## 엔티티 매핑 예시

```java
// Menu.java (연관 관계의 주인)
@ManyToOne(cascade = CascadeType.PERSIST)
@JoinColumn(name = "categoryCode")
private Category category;

// Category.java (주인이 아님)
@OneToMany(mappedBy = "category") // 주인 엔티티(Menu)의 필드명을 지정
private List<Menu> menuList;
```

## 특징

- `mappedBy`: 연관 관계의 주인이 아님을 명시하며, 외래 키를 가진 주인 엔티티의 필드명을 값으로 가짐
- 주인(Owner): 외래 키(FK)를 가진 쪽이 주인이 되어 등록/수정을 관리함
- 주의사항: 양방향 참조 시 `toString()`이나 Lombok에 의해 무한 루프가 발생할 수 있으므로 주의 필요

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 연관 관계 주인 | 외래 키(FK)가 있는 곳이 주인이 되어야 객체와 DB 상태를 일관되게 관리하기 쉬움 |
| LAZY vs EAGER | 실무에서는 N+1 문제 방지를 위해 가급적 LAZY로 설정하고, 조인이 필요할 때만 Fetch Join 사용 |
| CascadeType | 부모 엔티티 조작 시 자식까지 한꺼번에 처리할 때 유용 (예: 게시글 삭제 시 댓글도 삭제) |
| 단방향 vs 양방향 | 기본은 단방향으로 설계하고, 비즈니스 로직상 역방향 조회가 반드시 필요한 경우에만 양방향 추가 |

# chap04 — JPQL (Java Persistence Query Language)

## 개념

엔티티 객체를 대상으로 쿼리하는 객체지향 쿼리 언어

- SQL을 추상화해서 특정 데이터베이스 SQL에 의존하지 않음
- SQL과 문법이 유사하지만, 테이블이 아닌 엔티티 객체와 필드를 대상으로 조회함
- 최종적으로는 데이터베이스 방언(Dialect)에 맞는 SQL로 변환되어 실행됨

## 주요 특징 및 문법

| 구분 | 설명 | 예시 |
| --- | --- | --- |
| 대상 | 테이블이 아닌 엔티티 객체 대상 | `SELECT m FROM Menu m` |

| Query | 반환 타입을 명확히 지정할 수 없을 때 사용 | `em.createQuery(jpql)` |

# section01 — 기본 조회 (Simple JPQL)

## 1. 결과 조회 API

- `getResultList()`: 결과가 하나 이상일 때 리스트 반환 (결과 없으면 빈 리스트)
- `getSingleResult()`: 결과가 정확히 하나일 때 단일 객체 반환 (결과 없거나 둘 이상이면 예외 발생)

## 2. JPQL 연산자

- `DISTINCT`: 중복된 결과를 제거하고 조회
- `IN`: 지정된 목록에 포함된 값만 조회
- `LIKE`: 문자열 패턴 매칭 (`%`: 0개 이상의 문자, `_`: 1개의 문자)

---

# section02 — 파라미터 바인딩 (Parameter Binding)

## 1. 이름 기준 바인딩 (Named Parameters)

- 콜론(`:`) 뒤에 파라미터 이름을 지정하여 바인딩
- 위치에 의존하지 않아 유지보수가 쉽고 실무에서 권장됨

```java
String jpql = "SELECT m FROM Menu m WHERE m.menuName = :menuName";

List<Menu> result = em.createQuery(jpql, Menu.class)
                      .setParameter("menuName", menuName)
                      .getResultList();
```

## 2. 위치 기준 바인딩 (Positional Parameters)

- 물음표(`?`) 뒤에 숫자를 지정하여 바인딩 (1부터 시작)
- 파라미터 순서가 변경되면 오류가 발생할 위험이 있어 사용을 지양함

```java
String jpql = "SELECT m FROM Menu m WHERE m.menuName = ?1";

List<Menu> result = em.createQuery(jpql, Menu.class)
                      .setParameter(1, menuName)
                      .getResultList();
```

| 항목 | 내용 |

| 객체 중심 | JPQL은 테이블이 아닌 객체를 대상으로 하므로 `SELECT *` 대신 별칭을 사용함 (`SELECT m`) |
| 타입 안정성 | `TypedQuery`를 사용하여 런타임 캐스팅 오류를 방지하고 타입 안정성을 확보함 |

| 유연성 | 특정 DB SQL에 종속되지 않아 데이터베이스를 변경해도 쿼리 수정을 최소화할 수 있음 |
