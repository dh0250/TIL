# chap06 — Interceptor

## 개념

> Spring MVC에서 컨트롤러 실행 전후에 공통 로직을 끼워 넣는 구조
>
- Filter(서블릿 레벨)와 달리 **HandlerInterceptor**는 Spring MVC 컨텍스트 안에서 동작
- 따라서 **Spring Bean을 주입받을 수 있다**

---

## HandlerInterceptor 인터페이스

```
요청
 └─► preHandle
        └─► Controller
               └─► postHandle
                      └─► View
                             └─► afterCompletion
```

| 메서드 | 호출 시점 | 반환값 |
| --- | --- | --- |
| preHandle | 컨트롤러 실행 전 | true → 계속 진행 / false → 요청 중단 |
| postHandle | 컨트롤러 실행 후 (View 렌더링 전) | - |
| afterCompletion | 요청 완전 종료 후 (예외 발생해도 호출) | - |

---

## StopwatchInterceptor — 실행 시간 측정

```java
@Component
public class StopwatchInterceptor implements HandlerInterceptor {

    private static final String START_TIME = "startTime";

    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        long startTime = System.currentTimeMillis();
        request.setAttribute(START_TIME, startTime);  // 요청 객체에 임시 보관
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response, ...) {

        if (request.getAttribute(START_TIME) instanceof Long startTime) {  // 패턴 매칭
            long elapsed = System.currentTimeMillis() - startTime;
            response.setHeader("X-Elapsed-time", elapsed + "ms");
        }
    }
}
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| request.setAttribute | preHandle → afterCompletion 간 데이터 전달 통로 |
| instanceof 패턴 매칭 | Java 16+. 타입 확인 + 캐스팅을 한 줄로 처리 |
| X-Elapsed-time 헤더 | 응답 헤더에 소요 시간을 ms 단위로 기록 |
| afterCompletion 선택 이유 | 예외 발생 시에도 반드시 호출 → 측정 누락 없음 |

## ApiKeyInterceptor — 관리자 API 인증

```java
@Component
public class ApiKeyInterceptor implements HandlerInterceptor {

    private static final String API_KEY_HEADER = "X-API-KEY";
    private static final String VALID_API_KEY  = "secret-key";

    private final ObjectMapper objectMapper;

    public ApiKeyInterceptor(ObjectMapper objectMapper) {
        this.objectMapper = objectMapper;
    }

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response, ...) {

        String apiKey = request.getHeader(API_KEY_HEADER);

        if (VALID_API_KEY.equals(apiKey)) {
            return true;  // 인증 성공 → Controller 진입
        }

        // 인증 실패 → 401 직접 응답 후 요청 중단
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.setCharacterEncoding("UTF-8");
        response.getWriter().write(objectMapper.writeValueAsString(errorBody));

        return false;  // Controller 실행 차단
    }
}
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| ObjectMapper 주입 | @Component이므로 Spring Bean 주입 가능 (Filter와의 차이) |
| return false | 컨트롤러 실행을 막고, 인터셉터 안에서 직접 응답을 완성 |
| response.getWriter().write(...) | Spring의 Jackson 자동 변환 없이 직접 JSON 문자열을 바디에 씀 |

## 인터셉터 등록 — WebConfiguration

```java
@Configuration
public class WebConfiguration implements WebMvcConfigurer {

    @Override
    public void addInterceptors(InterceptorRegistry registry) {

        registry.addInterceptor(stopwatchInterceptor)
                .addPathPatterns("/api/v1/**")   // 모든 API에 적용
                .excludePathPatterns("/error")   // /error 제외
                .order(1);                       // 실행 순서 (숫자 작을수록 먼저)

        registry.addInterceptor(apiKeyInterceptor)
                .addPathPatterns("/api/v1/admin/**")  // 관리자 경로만 적용
                .excludePathPatterns("/error")
                .order(2);
    }
}
```

> ⚠️ `@Component`만으로는 인터셉터가 동작하지 않는다.
>
>
> `WebMvcConfigurer`를 통해 어느 경로에 적용할지 명시적으로 등록해야 한다.
>

## 요청 처리 흐름 비교

```
HTTP 요청
  └─> Filter (Servlet 스펙, DispatcherServlet 밖)
        └─> DispatcherServlet
              └─> Interceptor - preHandle (Controller 실행 전)
                    └─> AOP - @Before / @Around 진입 (프록시 메서드 실행 전)
                          └─> Controller / Service 메서드 실행
                    └─> AOP - @After / @AfterReturning / @Around 복귀 (메서드 실행 후)
                    └─> Interceptor - postHandle (Controller 실행 후, View 렌더링 전)
              └─> View 렌더링
              └─> Interceptor - afterCompletion (View 렌더링 후)
        └─> Filter
HTTP 응답
```

---

## Filter vs Interceptor vs AOP

| 구분 | Filter | Interceptor | AOP |
| --- | --- | --- | --- |
| 동작 레벨 | 서블릿 컨테이너 | Spring MVC | Spring Bean (프록시) |
| Spring Bean 주입 | 불가 (기본) | 가능 | 가능 |
| 적용 단위 | 모든 요청 (URL 패턴) | URL 패턴 | 메서드 단위 (포인트컷) |
| 적용 대상 | 모든 서블릿 요청 | Controller | Controller, Service, Repository 등 모든 Bean |
| 주요 용도 | 인코딩, CORS, 보안 헤더 | 인증/인가, 로깅, 성능 측정 | 트랜잭션, 공통 로깅 |

---

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| HandlerInterceptor | preHandle / postHandle / afterCompletion 세 시점 제공 |
| preHandle 반환값 | false면 요청 즉시 중단, Controller 미실행 |
| request.setAttribute | 인터셉터 메서드 간 데이터 공유 수단 |
| WebMvcConfigurer | 인터셉터를 경로와 함께 Spring MVC에 등록하는 설정 클래스 |
| @Component | Bean 등록, 생성자 주입 가능하게 함 |
| order() | 인터셉터 실행 순서 지정, 숫자 작을수록 먼저 실행 |
| excludePathPatterns | 특정 경로를 인터셉터 적용에서 제외 |

---

## 한 줄 정리

> **Interceptor는 Spring MVC 레벨에서 Controller 전후 로직을 처리하는 구조이며, `preHandle`, `postHandle`, `afterCompletion`을 통해 인증, 로깅, 성능 측정 등의 공통 기능을 구현한다.**
>

# Section 01 — ResponseEntity

## ResponseMessage — 공통 응답 바디 DTO

```java
@AllArgsConstructor
@Getter @Setter
public class ResponseMessage {
    private int httpStatus;               // 200, 201 등
    private String message;               // "조회 성공"
    private Map<String, Object> results;  // 실제 데이터 ("users", "user" 키로 담음)
}
```

> 응답 body 구조를 일정하게 맞추기 위한 포장 클래스.
>
>
> 실제 데이터는 results Map 안에 넣는다.
>

## ResponseEntity 사용 패턴

### 패턴 1 — 생성자 방식 (헤더 직접 세팅)

```java
HttpHeaders headers = new HttpHeaders();
headers.setContentType(
    new MediaType("application", "json", StandardCharsets.UTF_8)
);

return new ResponseEntity<>(responseMessage, headers, HttpStatus.OK);
```

### 패턴 2 — 빌더 방식 (가독성 우선)

```java
return ResponseEntity
        .ok()               // 200 OK
        .headers(headers)
        .body(responseMessage);
```

### 패턴 3 — 201 Created + Location 헤더

```java
return ResponseEntity
        .created(
            URI.create("/api/v1/response/users/" + newUser.getNo())
        )
        .build();  // body 없음
```

### 패턴 4 — 204 No Content (수정/삭제)

```java
return ResponseEntity
        .noContent()    // 204 No Content
        .build();
```

# Section 02 — Bean Validation + 전역 예외 처리

## Bean Validation 개념

> 객체의 필드에 규칙을 붙여두고, 그 객체가 규칙에 맞는지 검사하는 방식
>

```
요청 JSON
  └─► @RequestBody 역직렬화
        └─► @Valid 검증 실행
              ├─ 성공 → Controller 진입
              └─ 실패 → MethodArgumentNotValidException 발생
                        → ExceptionController가 처리
```

---

## UserDTO — 검증 어노테이션

```java
@NotNull(message = "아이디는 반드시 입력되어야 합니다.")
@NotBlank(message = "아이디는 공백일 수 없습니다.")
private String id;

@NotNull(message = "비밀번호는 반드시 입력 되어야 합니다.")
@Length(max = 10, message = "비밀번호 길이는 10을 초과할 수 없습니다.")
private String pwd;

@NotBlank(message = "이름은 공백일 수 없습니다.")
@Size(max = 10, message = "이름의 길이 10을 초과할 수 없습니다.")
private String name;

@Past(message = "가입일은 현재보다 과거 날짜가 입력 되어야 합니다.")
private Date enrollDate;
```

### 주요 검증 어노테이션

| 어노테이션 | 설명 | 출처 |
| --- | --- | --- |
| @NotNull | null 불허 | Jakarta Validation |
| @NotBlank | null + 공백 불허 | Jakarta Validation |
| @Size(max=N) | 문자열 길이 제한 | Jakarta Validation |
| @Past | 현재 이전 날짜만 허용 | Jakarta Validation |
| @Length(max=N) | 문자열 길이 제한 | Hibernate Validator |

> @NotNull vs @NotBlank
>
>
> NotNull은 null만 막는다.
>
> NotBlank는 null + "" + " " 모두 막는다.
>

## @Valid — 검증 트리거

```java
@PostMapping("/users")
public ResponseEntity<Void> registUser(
        @Valid @RequestBody UserDTO user) {

    // 여기 도달했다면 검증 통과

```

> @Valid가 없으면 어노테이션을 붙여도 검증이 실행되지 않는다.
>

## 전역 예외 처리 — @RestControllerAdvice

```java
@RestControllerAdvice
public class ExceptionController {

    // 커스텀 예외 처리
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> userException(
            UserNotFoundException e) {

        return new ResponseEntity<>(
            new ErrorResponse(
                "ERROR_CODE_0000",
                "회원 정보 조회 실패",
                e.getMessage()
            ),
            HttpStatus.NOT_FOUND
        );
    }

    // Bean Validation 실패 처리
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> methodValidException(
            MethodArgumentNotValidException e) {

        String bindResultCode =
                e.getBindingResult()
                 .getFieldError()
                 .getCode();

        // bindResultCode로 switch → 에러코드/설명 매핑

        return new ResponseEntity<>(
                errorResponse,
                HttpStatus.BAD_REQUEST
        );
    }
}
```

### @RestControllerAdvice vs @ControllerAdvice

| 항목 | @ControllerAdvice | @RestControllerAdvice |
| --- | --- | --- |
| 응답 방식 | View 이름 반환 가능 | JSON 자동 직렬화 |
| 사용 맥락 | MVC (서버 렌더링) | REST API |
