# chap04 — Exception Handler

## 개념

Spring Boot에서 예외가 발생했을 때 클라이언트에게 일관된 JSON 에러 응답을 돌려주는 전역 예외 처리 패턴.

`@RestControllerAdvice` + `@ExceptionHandler`를 사용해 모든 컨트롤러의 예외를 한 곳에서 처리한다.

---

# 프로젝트 구조

```
Application.java                    (Spring Boot 진입점)
MemberDTO.java                      (도메인 객체)
MemberController.java               (REST 컨트롤러 — 예외 발생 지점)
MemberNotFoundException.java        (커스텀 예외 — 회원 없음)
InvalidMemberRequestException.java  (커스텀 예외 — 잘못된 요청)
ErrorResponse.java                  (에러 응답 공통 DTO)
GlobalExceptionHandler.java         (전역 예외 처리기)
api-test.http                       (API 테스트 파일)
```

---

# 도메인 — MemberDTO

```java
public class MemberDTO {
    private int no;
    private String name;
    private int age;
}
```

---

# 커스텀 예외 클래스

## 공통 패턴

```java
public class MemberNotFoundException extends RuntimeException {
    public MemberNotFoundException(String message) {
        super(message);
    }
}
```

| 클래스 | 상위 클래스 | 사용 상황 |
| --- | --- | --- |
| MemberNotFoundException | RuntimeException | 해당 번호의 회원이 존재하지 않을 때 |
| InvalidMemberRequestException | RuntimeException | 요청 값이 유효하지 않을 때 (이름 공백, 나이 음수 등) |

> RuntimeException 상속 → Unchecked Exception → 메서드 시그니처에 throws 선언 불필요.
>

---

# 에러 응답 DTO — ErrorResponse

```java
public class ErrorResponse {
    private final LocalDateTime timestamp;  // 에러 발생 시각 (자동 주입)
    private final int status;               // HTTP 상태 코드 (예: 404)
    private final String error;             // 상태 코드명 (예: "NOT_FOUND")
    private final String message;           // 에러 상세 메시지
    private final String path;              // 요청 URI
}
```

## 클라이언트(프론트엔드)가 받는 JSON 예시

```json
{
  "timestamp": "2024-01-01T12:00:00",
  "status": 404,
  "error": "NOT_FOUND",
  "message": "4번 회원을 찾을 수 없습니다.",
  "path": "/api/v1/members/4"
}
```

> Jackson이 직렬화하려면 getter가 반드시 있어야 한다.
>

---

# 전역 예외 처리기 — GlobalExceptionHandler

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MemberNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleMemberNotFound(
            MemberNotFoundException exception, HttpServletRequest request) {
        // 404 NOT_FOUND 응답
    }

    @ExceptionHandler(InvalidMemberRequestException.class)
    public ResponseEntity<ErrorResponse> handleInvalidMemberRequest(
            InvalidMemberRequestException exception, HttpServletRequest request) {
        // 400 BAD_REQUEST 응답
    }

    @ExceptionHandler(Exception.class)  // 마지막 방어선
    public ResponseEntity<ErrorResponse> handleException(
            Exception exception, HttpServletRequest request) {
        // 500 INTERNAL_SERVER_ERROR 응답
    }
}
```

## 어노테이션 비교

| 어노테이션 | 설명 |
| --- | --- |
| @RestControllerAdvice | 여러 컨트롤러의 예외를 한 곳에서 처리 + JSON 응답 |
| @ExceptionHandler | 특정 예외 타입을 어떤 메서드가 처리할지 지정 |

> @RestControllerAdvice = @ControllerAdvice + @ResponseBody
>

---

# 예외 처리 흐름

```
컨트롤러에서 예외 발생
        ↓
Spring이 GlobalExceptionHandler 탐색
        ↓
@ExceptionHandler 중 일치하는 예외 타입 메서드 실행
        ↓
ErrorResponse 객체 생성 → JSON 직렬화 → 클라이언트 응답
```

# API 엔드포인트 & 예외 발생 조건

| HTTP | URL | 정상 응답 | 예외 |
| --- | --- | --- | --- |
| GET | /api/v1/members | 200 — 전체 목록 | - |
| GET | /api/v1/members/{memberNo} | 200 — 단건 조회 | 없는 번호 → MemberNotFoundException → 404 |
| POST | /api/v1/members | 201 — 등록 성공 | 이름 공백 / 나이 음수 → InvalidMemberRequestException → 400 |
| GET | /api/v1/members/server-error | - | NPE 의도적 발생 → Exception → 500 |

# 코드 포인트

## 단건 조회 — orElseThrow

```java
@GetMapping("/{memberNo}")
public ResponseEntity<MemberDTO> findMember(@PathVariable int memberNo) {
    MemberDTO foundMember = members.stream()
            .filter(member -> member.getNo() == memberNo)
            .findFirst()
            .orElseThrow(() -> new MemberNotFoundException(
                    memberNo + "번 회원을 찾을 수 없습니다."));

    return ResponseEntity.ok(foundMember);
}
```

> orElseThrow() — Optional이 비어 있을 때 예외를 던지는 메서드. 스트림 조회 결과가 없으면 커스텀 예외 발생.
>

---

## 등록 — 입력값 검증 후 예외 투척

```java
@PostMapping
public ResponseEntity<Map<String, Object>> registMember(@RequestBody MemberDTO member) {
    if (member.getName() == null || member.getName().isBlank())
        throw new InvalidMemberRequestException("회원 이름은 비어있을 수 없습니다.");

    if (member.getAge() < 0)
        throw new InvalidMemberRequestException("회원 나이는 0 이상이어야 합니다.");

    members.add(member);
    ...
}
```

> 컨트롤러는 예외를 던지기만 하고, 처리는 GlobalExceptionHandler에 위임한다.
>

---

## 의도적 서버 오류 (NPE)

```java
@GetMapping("/server-error")
public ResponseEntity<Void> serverError() {
    String text = null;
    text.length();  // NullPointerException 발생
    return ResponseEntity.ok().build();
}
```

> @ExceptionHandler(Exception.class) — 특정 예외 핸들러가 없는 모든 예외의 마지막 방어선. 클라이언트에 내부 구현이 노출되지 않도록 메시지를 고정 문자열로 감춘다.
>

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| @RestControllerAdvice | 전역 예외 처리 클래스 선언, JSON 응답 자동 적용 |
| @ExceptionHandler | 처리할 예외 타입을 메서드에 매핑 |
| ErrorResponse | 에러 응답 공통 구조 (timestamp / status / error / message / path) |
| 커스텀 예외 | RuntimeException 상속 → Unchecked, throws 불필요 |
| orElseThrow() | Optional 비어 있을 때 예외 투척 |
| 마지막 방어선 | @ExceptionHandler(Exception.class) — 미처리 예외 일괄 500 처리 |

# chap05 — File Upload

## 개념

▎ HTTP multipart/form-data 요청으로 파일을 받아 서버 디스크에 저장하는 패턴.

▎ Spring은 MultipartFile 인터페이스로 업로드 파일을 추상화하고, application.properties로 크기 제한을 설정한다.

---

# 프로젝트 구조

```
Application.java          (Spring Boot 진입점)
FileDTO.java              (업로드 결과 메타 정보 DTO)
FileUploadController.java (파일 업로드 REST 컨트롤러)
application.properties    (파일 크기 제한 · 저장 경로 설정)
```

---

# 도메인 — FileDTO

```java
public class FileDTO {
    private String originalFileName; // 원본 파일명
    private String savedFileName;    // UUID로 치환된 저장 파일명
    private String filePath;         // 서버에 실제 저장된 절대 경로
    private String fileDescription;  // 요청과 함께 온 파일 설명

    private String contentType;      // MIME 타입 (image/png, text/plain …)
```

## 다중 파일 업로드 + 롤백

```java
@PostMapping("/multiple")
public ResponseEntity<...> uploadMultipleFiles(
        @RequestParam("files") List<MultipartFile> files, ...) {

    List<FileDTO> uploadFiles = new ArrayList<>();

    try {
        for (MultipartFile file : files) {
            uploadFiles.add(saveFile(file, "multiple", description));
        }
    } catch (IOException | RuntimeException e) {

        // 중간에 실패하면 이미 저장된 파일을 모두 삭제 (부분 저장 방지)
        for (FileDTO uploaded : uploadFiles) {
            Files.deleteIfExists(Paths.get(uploaded.getFilePath()));
        }

        throw e;
    }

    ...
}
```

> 여러 파일 중 하나라도 실패하면 앞서 저장된 파일을 전부 삭제해 부분 저장 상태를 방지한다.
>

## 파일 저장 핵심 로직 — saveFile()

```java
// 1. 유효성 검사 (빈 파일·파일명 없음)
validateFile(file);

// 2. 저장 디렉터리 생성 (없으면 자동 생성)
Path uploadPath = uploadRoot.resolve(directoryName).normalize();
Files.createDirectories(uploadPath);

// 3. UUID로 저장 파일명 생성 (원본명 충돌·보안 문제 방지)
String savedFileName =
        UUID.randomUUID().toString().replace("-", "") + extension;

// 4. 디스크에 실제 저장
file.transferTo(targetPath);
```

| 단계 | 이유 |
| --- | --- |
| UUID 저장명 | 동명 파일 덮어쓰기 방지, 원본명 노출 방지 |
| normalize() | ../ 같은 경로 조작 공격 차단 |
| createDirectories | 경로가 없어도 자동 생성 |

# chap06 — Interceptor

## 개념

Spring MVC에서 컨트롤러 실행 전후에 공통 로직을 끼워 넣는 구조.

Filter(서블릿 레벨)와 달리 HandlerInterceptor는 Spring MVC 컨텍스트 안에서 동작하므로 Spring Bean을 주입받을 수 있다.

---

# 프로젝트 구조

```
Application.java          (Spring Boot 진입점)
MenuDTO.java              (도메인 객체)
MenuController.java       (REST 컨트롤러)
StopwatchInterceptor.java (실행 시간 측정 인터셉터)
ApiKeyInterceptor.java    (관리자 API 인증 인터셉터)
```

---

# 도메인 — MenuDTO

```java
public class MenuDTO {
    private int menuCode;
    private String menuName;
    private int menuPrice;
}
```

---

# HandlerInterceptor 인터페이스

```
요청 ──► preHandle ──► Controller ──► postHandle ──► View ──► afterCompletion
```

| 메서드 | 호출 시점 | 반환값 |
| --- | --- | --- |
| preHandle | 컨트롤러 실행 전 | true → 계속 진행 / false → 요청 중단 |
| postHandle | 컨트롤러 실행 후 (View 렌더링 전) | — |
| afterCompletion | 요청 완전 종료 후 (예외 발생해도 호출) | — |

---

# StopwatchInterceptor — 실행 시간 측정

```java
@Component
public class StopwatchInterceptor implements HandlerInterceptor {

    private static final String START_TIME = "startTime";

    @Override
    public boolean preHandle(HttpServletRequest request, ...) {
        long startTime = System.currentTimeMillis();
        request.setAttribute(START_TIME, startTime);  // 요청 객체에 임시 보관
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response, ...) {

        if (request.getAttribute(START_TIME) instanceof Long startTime) {  // 패턴 매칭
            long elapsed = System.currentTimeMillis() - startTime;
            response.setHeader("X-Elapsed-time", elapsed + "ms");
        }
    }
}
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| request.setAttribute | preHandle → afterCompletion 간 데이터 전달 통로 |
| instanceof 패턴 매칭 | Java 16+. 타입 확인 + 캐스팅을 한 줄로 처리 |
| X-Elapsed-time 헤더 | 응답 헤더에 소요 시간을 ms 단위로 기록 |
| afterCompletion 선택 이유 | 예외 발생 시에도 반드시 호출 → 측정 누락 없음 |

# ApiKeyInterceptor — 관리자 API 인증

```java
@Component
public class ApiKeyInterceptor implements HandlerInterceptor {

    private static final String API_KEY_HEADER = "X-API-KEY";
    private static final String VALID_API_KEY  = "secret-key";

    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response, ...) {

        String apiKey = request.getHeader(API_KEY_HEADER);

        if (VALID_API_KEY.equals(apiKey)) {
            return true;  // 인증 성공 → Controller 진입
        }

        // 인증 실패 → 401 직접 응답 후 요청 중단
        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.getWriter().write(objectMapper.writeValueAsString(errorBody));

        return false;  // Controller 실행 차단
    }
}
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| ObjectMapper 주입 | @Component이므로 Spring Bean 주입 가능 (Filter와의 차이) |
| return false | 컨트롤러 실행을 막고, 인터셉터 안에서 직접 응답을 완성 |
| response.getWriter().write(...) | Spring의 Jackson 자동 변환 없이 직접 JSON 문자열을 바디에 씀 |

# 인터셉터 등록 — WebMvcConfigurer

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addInterceptors(InterceptorRegistry registry) {

        registry.addInterceptor(stopwatchInterceptor)
                .addPathPatterns("/api/v1/**");       // 모든 API에 적용

        registry.addInterceptor(apiKeyInterceptor)
                .addPathPatterns("/api/v1/admin/**"); // 관리자 경로만 적용
    }
}
```

> @Component만으로는 인터셉터가 동작하지 않는다. WebMvcConfigurer를 통해 어느 경로에 적용할지 명시적으로 등록해야 한다.
>

# Filter vs Interceptor

| 구분 | Filter | HandlerInterceptor |
| --- | --- | --- |
| 동작 레벨 | 서블릿 컨테이너 | Spring MVC |
| Spring Bean 주입 | 불가 (기본) | 가능 |
| 적용 범위 | 모든 요청 | addPathPatterns로 제어 |
| 주요 용도 | 인코딩, CORS, 보안 | 인증/인가, 로깅, 성능 측정 |

---

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| HandlerInterceptor | preHandle / postHandle / afterCompletion 세 시점 제공 |
| preHandle 반환값 | false면 요청 즉시 중단, Controller 미실행 |
| request.setAttribute | 인터셉터 메서드 간 데이터 공유 수단 |
| WebMvcConfigurer | 인터셉터를 경로와 함께 Spring MVC에 등록하는 설정 클래스 |
| @Component | Bean 등록, 생성자 주입 가능하게 함 |
