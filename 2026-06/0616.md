# section06 — 조인 (Join)

JPQL의 조인은 SQL과 달리 **엔티티 간의 연관관계(필드)**를 기준으로 조인

## 엔티티 연관관계 설정

```java
// Menu.java — @ManyToOne (다대일, 연관관계의 주인)
@ManyToOne
@JoinColumn(name = "categoryCode")
private Category category;

// Category.java — @OneToMany (일대다, 양방향)
@OneToMany(mappedBy = "category")
private List<Menu> menuList;
```

---

## 조인 종류

### 1. 내부 조인 (INNER JOIN)

양쪽 모두 연관된 데이터만 조회 → **카테고리가 없는 메뉴는 제외**

```java
String jpql = "SELECT m FROM Section06Menu m JOIN m.category c";
List<Menu> result = entityManager.createQuery(jpql, Menu.class).getResultList();
```

- SQL의 `INNER JOIN`과 동일
- `JOIN` 키워드만 써도 내부 조인
- 테이블명이 아닌 **엔티티 필드명** (`m.category`)으로 조인

---

### 2. 외부 조인 (OUTER JOIN)

한쪽에 데이터가 없어도 포함하여 조회

```java
// RIGHT JOIN — 카테고리를 기준으로 메뉴가 없는 카테고리도 포함
String jpql = "SELECT m.menuName, c.categoryName FROM Section06Menu m RIGHT JOIN m.category c"
            + " ORDER BY m.category.categoryCode";
List<Object[]> result = entityManager.createQuery(jpql).getResultList();
```

- `LEFT JOIN` / `RIGHT JOIN` 모두 사용 가능
- 여러 필드를 선택하면 반환 타입은 `List<Object[]>`

---

### 3. 컬렉션 조인 (Collection Join)

`@OneToMany` 관계의 **컬렉션 필드**를 기준으로 조인

```java
// Category의 menuList 컬렉션을 통해 조인
String jpql = "SELECT c.categoryName, m.menuName FROM Section06Category c LEFT JOIN c.menuList m";
List<Object[]> result = entityManager.createQuery(jpql).getResultList();
```

- 다대일 관계의 반대 방향(일대다)에서 조인하는 방식
- `LEFT JOIN`을 사용해 메뉴가 없는 카테고리도 포함

---

### 4. 페치 조인 (FETCH JOIN) — 핵심

연관된 엔티티를 **한 번의 쿼리로 함께 즉시 로딩**하는 JPQL 전용 기능

```java
String jpql = "SELECT m FROM Section06Menu m JOIN FETCH m.category c";
List<Menu> result = entityManager.createQuery(jpql, Menu.class).getResultList();
```

- `JOIN FETCH` 키워드 사용
- 지연 로딩(LAZY) 설정이 있어도 **연관 엔티티를 즉시 로딩**하여 N+1 문제 해결
- 반환 타입은 엔티티 자체 (`List<Menu>`)

---

## 조인 비교 정리

| 조인 종류 | JPQL 문법 | 반환 타입 | 특징 |
|-----------|-----------|-----------|------|
| 내부 조인 | `JOIN m.category c` | `List<Menu>` | 양쪽 모두 존재하는 데이터만 조회 |
| 외부 조인 | `LEFT/RIGHT JOIN m.category c` | `List<Object[]>` | 한쪽 데이터 없어도 포함 |
| 컬렉션 조인 | `LEFT JOIN c.menuList m` | `List<Object[]>` | 일대다 컬렉션 필드 기준 조인 |
| 페치 조인 | `JOIN FETCH m.category c` | `List<Menu>` | 연관 엔티티 즉시 로딩, N+1 해결 |

---

## 핵심 포인트

| 항목 | 내용 |
|------|------|
| 조인 기준 | 테이블 컬럼이 아닌 **엔티티의 연관관계 필드** (`m.category`, `c.menuList`) |
| SQL 변환 | JPA가 방언에 맞는 SQL JOIN으로 자동 변환 |
| 페치 조인 | 지연 로딩으로 인한 **N+1 문제**를 해결하는 가장 일반적인 방법 |
| 다중 필드 조회 | 여러 필드 선택 시 `List<Object[]>` 로 받아 수동 처리 |

# chap05 — Native Query (네이티브 쿼리)

## 개념

JPA에서 **순수 SQL(Native SQL)** 을 직접 사용하는 방식

- JPQL이 지원하지 않는 DB 특화 문법(힌트, 함수, 복잡한 JOIN 등)을 사용해야 할 때 활용
- `createNativeQuery()`로 실행하며, SQL은 **테이블명·컬럼명** 기준으로 작성
- 결과 매핑 방식에 따라 세 가지로 나뉨

---

## 프로젝트 구조

```
chap05_native-query/
├── src/main/java/com/ohgiraffers/nativequery/
│   ├── section01/simple/
│   │   ├── Menu.java                   ← 엔티티 매핑 (@Entity(name="Section01Menu"))
│   │   ├── Category.java               ← @SqlResultSetMappings 정의 (자동 결과 매핑)
│   │   └── NativeQueryRepository.java  ← 결과 타입 지정 / 미지정 / 자동 매핑 쿼리
│   └── section02/namedquery/
│       ├── Category.java               ← @NamedNativeQueries + @SqlResultSetMapping 정의
│       └── NamedQueryRepository.java   ← createNamedQuery()로 이름 기반 네이티브 쿼리 실행
└── src/main/resources/
    └── application.yml                 ← ddl-auto: none (기존 데이터 기반 조회 테스트)
```

---

## section01 — 기본 Native Query (Simple)

### 1. 결과 타입 지정 — 엔티티 매핑

결과를 엔티티 클래스로 직접 받을 때. **모든 컬럼을 SELECT해야** 매핑 가능

```java
String query = "SELECT menu_code, menu_name, menu_price, category_code, orderable_status"
             + " FROM tbl_menu WHERE menu_code = ?";

Query nativeQuery = entityManager.createNativeQuery(query, Menu.class)
                                 .setParameter(1, menuCode);   // 위치 기준 바인딩 (1부터)

return (Menu) nativeQuery.getSingleResult();
```

> JPQL과 달리 파라미터 바인딩에 `?` (숫자 없이) 사용 가능 (positional)

### 2. 결과 타입 미지정 — Object[] 수동 처리

반환 타입이 불명확하거나 일부 컬럼만 조회할 때

```java
String query = "SELECT menu_name, menu_price FROM tbl_menu";
Query nativeQuery = entityManager.createNativeQuery(query);

List<Object[]> result = nativeQuery.getResultList();
```

### 3. 자동 결과 매핑 — @SqlResultSetMapping

엔티티 + 추가 컬럼이 혼합된 복잡한 결과를 한 번에 매핑할 때. 매핑 정의는 **엔티티 클래스에 선언**

```java
// Category.java 에 매핑 정의
@SqlResultSetMapping(
    name = "categoryCountAutoMapping",
    entities = {@EntityResult(entityClass = Category.class)},  // 엔티티 결과
    columns  = {@ColumnResult(name = "menu_count")}            // 추가 스칼라 컬럼
)
@Entity(name = "Section01Category")
public class Category { ... }
```

```java
// NativeQueryRepository.java — 매핑 이름 참조
String query = "SELECT a.category_code, a.category_name, a.ref_category_code,"
             + " COALESCE(v.menu_count, 0) menu_count"
             + " FROM tbl_category a"
             + " LEFT JOIN (...) v ON (a.category_code = v.category_code)"
             + " ORDER BY 1";

Query nativeQuery = entityManager.createNativeQuery(query, "categoryCountAutoMapping");
List<Object[]> result = nativeQuery.getResultList();
// row[0] → Category 엔티티, row[1] → menu_count(Long)
```

### section01 결과 매핑 방식 비교

| 방식 | createNativeQuery 두 번째 인자 | 반환 타입 | 특징 |
|------|-------------------------------|-----------|------|
| 엔티티 타입 지정 | `Menu.class` | `Menu` | 모든 컬럼 SELECT 필수 |
| 타입 미지정 | (없음) | `List<Object[]>` | 부분 컬럼 조회 가능 |
| 자동 결과 매핑 | `"매핑이름"` (String) | `List<Object[]>` | 엔티티+스칼라 혼합 매핑 |

## section02 — Named Native Query (이름 기반 네이티브 쿼리)

SQL을 **엔티티 클래스에 미리 정의**해두고 이름으로 호출하는 방식. 쿼리를 한 곳에서 관리할 수 있어 재사용에 유리

### 선언 — @NamedNativeQuery

```java
@SqlResultSetMapping(
    name = "categoryCountAutoMapping2",
    entities = {@EntityResult(entityClass = Category.class)},
    columns  = {@ColumnResult(name = "menu_count")}
)
@NamedNativeQueries(
    value = {
        @NamedNativeQuery(
            name             = "Category.menuCountOfCategory",   // 쿼리 이름
            query            = "SELECT a.category_code, ...",    // 실제 SQL
            resultSetMapping = "categoryCountAutoMapping2"       // 매핑 이름 참조
        )
    }
)
@Entity(name = "Section02Category")
public class Category { ... }
```

### 실행 — createNamedQuery()

```java
Query nativeQuery = entityManager.createNamedQuery("Category.menuCountOfCategory");
List<Object[]> result = nativeQuery.getResultList();
```

> `createNativeQuery()` 가 아닌 `createNamedQuery()` 로 호출

---

## Native Query vs JPQL 비교

| 항목 | JPQL | Native Query |
|------|------|--------------|
| 쿼리 대상 | 엔티티 객체 / 필드 | 테이블 / 컬럼 |
| DB 독립성 | O (방언 자동 변환) | X (DB 특화 SQL 직접 작성) |
| 기능 | 표준 JPA 문법 | DB 고유 함수, 힌트, 복잡한 JOIN 가능 |
| 타입 안정성 | TypedQuery로 확보 | 결과 매핑 직접 지정 필요 |

---

## 핵심 요약

| 항목 | 내용 |
|------|------|
| 사용 시점 | JPQL로 표현 불가한 복잡한 SQL, DB 전용 함수·힌트 사용 시 |
| 엔티티 매핑 조건 | 결과를 엔티티로 받으려면 해당 테이블의 모든 컬럼 SELECT 필수 |
| @SqlResultSetMapping | 엔티티 + 추가 스칼라 컬럼 혼합 결과를 Object[]로 매핑 |
| @NamedNativeQuery | 쿼리를 엔티티에 미리 선언 → createNamedQuery()로 재사용 |
| 파라미터 바인딩 | `?` (위치 기준, 1부터) 또는 `:name` (이름 기준) 모두 사용 가능 |
| 결과 없는 경우 | JPQL과 동일하게 getResultList()는 빈 리스트, getSingleResult()는 예외 |
