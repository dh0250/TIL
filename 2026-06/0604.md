# chap01 — MyBatis 연결 설정

## 개념

> SQL을 Java 코드와 분리하여 관리하는 SQL Mapper 프레임워크
>
- JDBC의 반복 코드(Connection 생성, PreparedStatement, ResultSet 닫기 등)를 MyBatis가 대신 처리
- SQL은 개발자가 직접 작성 → JPA와 달리 SQL 제어권이 개발자에게 있다
- 설정 방식은 두 가지: Java Config (코드로 설정) / XML Config (xml 파일로 설정)

---

# MyBatis 핵심 객체 흐름

```
설정 (Environment + Configuration)
         │
         ▼
SqlSessionFactory    ← 앱 실행 중 딱 하나만 생성 (싱글턴)
         │
         ▼
SqlSession           ← 요청마다 열고 닫는 DB 세션
         │
         ▼
Mapper (Proxy)       ← MyBatis가 동적 프록시로 자동 구현
         │
         ▼
SQL 실행 → 결과 반환
```

# DataSource / TransactionManager 선택지

| 구분 | 옵션 | 설명 |
| --- | --- | --- |
| TransactionManager | JDBC | 개발자가 직접 commit / rollback 호출 |
| TransactionManager | MANAGED | WAS / Spring 외부 환경이 트랜잭션 관리 |
| DataSource | POOLED | 커넥션 풀 사용 — 실무 표준 |
| DataSource | UNPOOLED | 매번 새 연결 생성 — 테스트용 |

# Section 01 — Java Config

## Mapper.java — 어노테이션 기반 SQL 정의

```java
public interface Mapper {

    // @Select: SQL을 어노테이션으로 직접 메서드에 붙인다
    // 메서드 이름이 이 쿼리의 식별자(ID) 역할을 한다
    @Select("SELECT CURDATE()")
    Date selectSysdate();
}
```

## Application.java — 단계별 설정

### 1. 환경(Environment) 설정

```java
Environment environment = new Environment(
    "dev",                          // 환경 ID
    new JdbcTransactionFactory(),   // 트랜잭션 매니저
    new PooledDataSource(DRIVER, URL, USER, PASSWORD)  // 커넥션 풀
);
```

### 2. Configuration 생성 + Mapper 등록

```java
Configuration configuration = new Configuration(environment);
configuration.addMapper(Mapper.class);
```

### 3. SqlSessionFactory 생성

```java
SqlSessionFactory sqlSessionFactory =
    new SqlSessionFactoryBuilder().build(configuration);
```

### 4. SqlSession 열기

```java
SqlSession sqlSession = sqlSessionFactory.openSession(false);
```

### 5. Mapper 프록시 가져와서 쿼리 실행

```java
Mapper mapper = sqlSession.getMapper(Mapper.class);
Date date = mapper.selectSysdate();

sqlSession.close();
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| addMapper(Mapper.class) | 인터페이스를 MyBatis에 등록해야 프록시 생성 가능 |
| getMapper() | 등록된 인터페이스의 동적 프록시 구현체 반환 |
| openSession(false) | auto-commit 끄기 → 명시적 commit 필요 |
| 동적 프록시 | 인터페이스만 있으면 MyBatis가 실제 구현 객체를 런타임에 생성 |

# Section 02 — XML Config

## mybatis-config.xml — 환경 설정

```xml
<configuration>
    <environments default="dev">
        <environment id="dev">
            <transactionManager type="JDBC"/>
            <dataSource type="POOLED">
                <property name="driver" value="com.mysql.cj.jdbc.Driver"/>
                <property name="url" value="jdbc:mysql://localhost/menudb"/>
                <property name="username" value="ohgiraffers"/>
                <property name="password" value="ohgiraffers"/>
            </dataSource>
        </environment>
    </environments>
    <mappers>
        <mapper resource="mapper.xml"/>
    </mappers>
</configuration>
```

## mapper.xml — SQL 정의

```xml
<mapper namespace="mapper">
    <select id="selectSysdate" resultType="java.util.Date">
        SELECT CURDATE()
    </select>
</mapper>
```

> namespace.id 조합이 쿼리의 고유 식별자가 된다 → "mapper.selectSysdate"
>

## Application.java — XML 기반 실행

### 1. classpath에서 XML 파일을 InputStream으로 읽기

```java
InputStream inputStream =
    Resources.getResourceAsStream("mybatis-config.xml");
```

### 2. XML 설정으로 SqlSessionFactory 생성

```java
SqlSessionFactory sqlSessionFactory =
    new SqlSessionFactoryBuilder().build(inputStream);
```

### 3. SqlSession 열기

```java
SqlSession session = sqlSessionFactory.openSession(false);
```

### 4. Mapper 인터페이스 없이 쿼리 문자열로 직접 실행

```java
Date date = session.selectOne("mapper.selectSysdate");

session.close();
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| Resources.getResourceAsStream() | classpath 기준으로 XML 파일을 읽는 MyBatis 유틸 |
| session.selectOne("namespace.id") | Mapper 인터페이스 없이 문자열 ID로 직접 쿼리 실행 |
| `<mappers>` 태그 | XML 설정에서 SQL 파일을 등록하는 방법 |
| namespace | 같은 ID를 가진 쿼리가 여러 파일에 있을 때 구분하는 식별자 |

# Java Config vs XML Config 비교

| 구분 | Java Config (section01) | XML Config (section02) |
| --- | --- | --- |
| 설정 방법 | 코드로 직접 객체 생성 | XML 파일 파싱 |
| SQL 위치 | @Select 어노테이션 | mapper.xml 파일 |
| 쿼리 실행 | getMapper() → 프록시 메서드 호출 | session.selectOne("namespace.id") |
| 타입 안전성 | 인터페이스 → 컴파일 타임 검증 | 문자열 ID → 런타임에 오류 발견 |
| 실무 선호 | Spring 연동 시 주로 사용 | 순수 MyBatis 또는 레거시 |

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| SqlSessionFactory | 앱 실행 중 하나만 생성하는 세션 공장, SqlSessionFactoryBuilder로 만든다 |
| SqlSession | 실제 DB 작업을 수행하는 객체, 요청마다 열고 반드시 닫아야 한다 |
| openSession(false) | auto-commit 비활성화, 직접 commit / rollback 필요 |
| POOLED DataSource | 커넥션을 미리 만들어 재사용 → 매번 연결 비용 절감 |
| JDBC TransactionManager | 개발자가 트랜잭션을 직접 제어, Spring 연동 시 MANAGED로 전환 |
| 동적 프록시 | MyBatis가 Mapper 인터페이스를 런타임에 자동 구현 |
| namespace.id | XML 방식에서 쿼리를 고유하게 식별하는 주소 |

# chap02 — MyBatis 라이프사이클

## 개념

> SqlSessionFactory와 SqlSession의 생명주기(Lifecycle)를 올바르게 관리하는 것이 MyBatis의 핵심
>
- SqlSessionFactory → 앱 실행 중 딱 하나만 생성 (싱글톤), 생성 비용이 크기 때문에 재사용
- SqlSession → 요청마다 새로 열고, 사용 후 반드시 닫아야 한다
- Template 클래스로 Factory를 싱글톤으로 캡슐화하고, Session 발급만 외부에 노출

---

```
    ├── Application.java    (SqlSession 라이프사이클 확인용 메인)
    └── Template.java       (SqlSessionFactory 싱글톤 + SqlSession 발급)

resources/
└── mybatis-config.xml      (DB 연결 설정)
```

# MyBatis 라이프사이클 흐름

```
앱 최초 실행
     │
     ▼
Template.getSqlSession() 최초 호출
     │  sqlSessionFactory == null 이므로
     ▼
mybatis-config.xml 읽기 (Resources.getResourceAsStream)
     │
     ▼
SqlSessionFactory 생성 → static 필드에 저장 (이후 재사용)
     │
     ▼
sqlSessionFactory.openSession(false) → SqlSession 발급
     │
     ▼
작업 수행 후 sqlSession.close() ← 반드시 닫아야 한다

두 번째 호출부터:
sqlSessionFactory == null 이 아니므로 생성 건너뜀
     │
     ▼
openSession(false) → 새 SqlSession 발급 (hashCode 다름)
```

---

# Template.java — 싱글톤 구현

```java
public class Template {

    // static 필드: 클래스 로딩 시 메모리에 올라가 앱 종료까지 유지
    private static SqlSessionFactory sqlSessionFactory;

    public static SqlSession getSqlSession() {

        if (sqlSessionFactory == null) {               // 최초 1회만 생성
            String resource = "mybatis-config.xml";
            try {
                InputStream inputStream = Resources.getResourceAsStream(resource);
                sqlSessionFactory = new SqlSessionFactoryBuilder().build(inputStream);
            } catch (IOException e) {
                e.printStackTrace();
            }
        }

        // Factory는 재사용, Session은 매번 새로 발급
        SqlSession sqlSession = sqlSessionFactory.openSession(false);

        System.out.println("sqlSessionFactory hashCode: " + sqlSessionFactory.hashCode());
        System.out.println("sqlSession       hashCode: " + sqlSession.hashCode());

        return sqlSession;
    }
}
```

## 핵심 포인트

| 항목 | 설명 |
| --- | --- |
| private static SqlSessionFactory | JVM 내에서 단 하나, 앱 종료까지 살아있음 |
| if (sqlSessionFactory == null) | Lazy Initialization — 처음 필요할 때 한 번만 생성 |
| openSession(false) | auto-commit OFF → 명시적 commit / rollback 필요 |
| hashCode 출력 | Factory는 항상 같은 값, Session은 호출마다 다른 값 → 라이프사이클 차이 시각화 |

---

# Application.java — 라이프사이클 확인

```java
public class Application {
    public static void main(String[] args) {

        // 동일 메서드 5회 반복 → Factory 해시코드 동일, Session 해시코드 상이 확인
        printAndCloseSession();
        printAndCloseSession();
        printAndCloseSession();
        printAndCloseSession();
        printAndCloseSession();
    }

    private static void printAndCloseSession() {
        SqlSession sqlSession = getSqlSession();
        try {
            System.out.println(sqlSession);
        } finally {
            sqlSession.close();    // finally로 반드시 닫기
        }
    }
}
```

## 실행 결과 (예시)

```
sqlSessionFactory hashCode: 123456789   ← 5번 모두 동일
sqlSession       hashCode: 111111111   ← 매번 다름

sqlSessionFactory hashCode: 123456789
sqlSession       hashCode: 222222222

...
```

> Factory는 단 하나, Session은 매번 새 객체 → 라이프사이클 차이 직접 확인
>

# SqlSessionFactory vs SqlSession 비교

| 항목 | SqlSessionFactory | SqlSession |
| --- | --- | --- |
| 생성 시점 | 앱 최초 실행 시 1회 | 요청마다 |
| 소멸 시점 | 앱 종료 시 | 작업 완료 후 즉시 close() |
| 생성 비용 | 크다 (XML 파싱, 커넥션 풀 초기화) | 작다 |
| 재사용 | 반드시 재사용 (싱글톤) | 재사용 금지 — 스레드 안전하지 않음 |
| 패턴 | Singleton | Per-request (요청 단위) |

---

# 싱글톤 패턴 — 구현 방식

## Eager Initialization (즉시 초기화)

```java
static SqlSessionFactory factory = new SqlSessionFactory(...);
```

→ 클래스 로딩 시 바로 생성 — 사용 안 해도 메모리 차지

## Lazy Initialization (지연 초기화) ← 이 프로젝트 방식

```java
if (factory == null) {
    factory = new SqlSessionFactory(...);
}
```

→ 처음 사용할 때 생성 — 자원 절약

> 멀티스레드 환경에서는 synchronized 또는 DCL(Double-Checked Locking) 필요 (이 예제는 단일 스레드 학습용)
>

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| SqlSessionFactory | 앱 실행 중 하나만 존재, static 필드로 싱글톤 구현 |
| SqlSession | 요청마다 새로 생성, finally 블록에서 반드시 close() |
| openSession(false) | auto-commit 비활성화, 직접 commit / rollback 필요 |
| Lazy Initialization | null 체크 후 최초 1회만 Factory 생성 |
| hashCode 비교 | Factory 동일 / Session 상이 → 라이프사이클 차이 시각적 증명 |
| Template 클래스 | Factory 싱글톤 캡슐화 → 이후 chap에서 재사용 가능한 공통 유틸 역할 |

# 한 줄 정리

# chap03 — MyBatis CRUD

## 개념

> MyBatis로 실제 데이터를 조회·등록·수정·삭제하는 CRUD 흐름 전체를 계층 분리(Layer Architecture)로 구현
>
- SQL은 XML Mapper에 분리, Java 코드에는 SQL이 없다
- DAO → Service → Controller → Application 4계층으로 책임을 나눈다
- Service 계층이 트랜잭션(commit / rollback)을 직접 관리한다
- ResultMap으로 DB 컬럼명(snake_case)과 Java 필드명(camelCase) 차이를 해결

# menu-mapper.xml — SQL + ResultMap 정의

## ResultMap — 컬럼-필드 매핑

```xml
<resultMap id="menuResultMap" type="com.ohgiraffers.section01.xmlconfig.MenuDTO">
    <id     property="code"            column="menu_code"/>
    <result property="name"            column="menu_name"/>
    <result property="price"           column="menu_price"/>
    <result property="categoryCode"    column="category_code"/>
    <result property="orderableStatus" column="orderable_status"/>
</resultMap>
```

> `<id>` : PK 컬럼 선언 — MyBatis 내부 캐시·동일성 판별에 사용
>
>
> `<result>` : 일반 컬럼 매핑
>

---

## SELECT 전체 조회

```xml
<select id="selectAllMenu" resultMap="menuResultMap">
    SELECT menu_code, menu_name, menu_price, category_code, orderable_status
    FROM tbl_menu
    WHERE orderable_status = 'Y'
    ORDER BY menu_code
</select>
```

---

## SELECT 단건 조회

```xml
<select id="selectMenuByCode" parameterType="_int" resultMap="menuResultMap">
    SELECT menu_code, menu_name, menu_price, category_code, orderable_status
    FROM tbl_menu
    WHERE orderable_status = 'Y'
      AND menu_code = #{code}
</select>
```

> parameterType="*int" : 언더스코어(*) 접두사 = primitive 타입 (int, long 등)
>
>
> #{code} : PreparedStatement 바인딩 — SQL Injection 방지
>

---

## INSERT 등록

```xml
<insert id="insertMenu" parameterType="com.ohgiraffers.section01.xmlconfig.MenuDTO">
    INSERT INTO tbl_menu (menu_name, menu_price, category_code, orderable_status)
    VALUES (#{name}, #{price}, #{categoryCode}, 'Y')
</insert>
```

> #{fieldName} : DTO의 Getter 이름 기준으로 값을 꺼낸다
>
>
> orderable_status는 항상 'Y'로 고정 — 삽입 시 자동 활성화
>

# MenuDAO.java — MyBatis 쿼리 실행

```java
public class MenuDAO {

    public List<MenuDTO> selectAllMenu(SqlSession sqlSession) {
        return sqlSession.selectList("MenuMapper.selectAllMenu");
    }

    public MenuDTO selectMenuByCode(SqlSession sqlSession, int code) {
        return sqlSession.selectOne("MenuMapper.selectMenuByCode", code);
    }

    public int insertMenu(SqlSession sqlSession, MenuDTO menu) {
        return sqlSession.insert("MenuMapper.insertMenu", menu);
    }
}
```

| 메서드 | MyBatis API | 반환값 |
| --- | --- | --- |
| selectAllMenu | selectList("namespace.id") | List |
| selectMenuByCode | selectOne("namespace.id", param) | MenuDTO or null |
| insertMenu | insert("namespace.id", param) | 영향받은 행 수 (int) |

> DAO는 SqlSession을 직접 생성하지 않는다 — Service에서 받아서 사용

---

## 등록 (트랜잭션 필요)

```java
public boolean registMenu(MenuDTO menu) {
    SqlSession sqlSession = getSqlSession();
    try {
        int result = new MenuDAO().insertMenu(sqlSession, menu);

        if (result > 0) {
            sqlSession.commit();     // DB 영구 반영
        } else {
            sqlSession.rollback();   // 변경 취소
        }

        return result > 0;

    } catch (RuntimeException e) {
        sqlSession.rollback();       // 예외 발생 시 취소
        throw e;                     // 상위 계층으로 예외 전파
    } finally {
        sqlSession.close();          // 성공/실패 무관하게 반드시 닫기
    }
}
```

---

## 트랜잭션 흐름 요약

```
SqlSession 획득 (autoCommit = false)
     │
     ├─── 작업 성공 (result > 0) ──→ commit()  → DB 영구 반영
     │
     ├─── 작업 실패 (result == 0) ─→ rollback() → 변경 취소
     │
     └─── 예외 발생 ─────────────→ rollback() + throw → 취소 후 전파
                │
                └─ finally: close()  ← 항상 실행
```

# #{} vs ${} — 파라미터 바인딩 비교

| 구분 | #{} | ${} |
| --- | --- | --- |
| 처리 방식 | PreparedStatement 바인딩 | 문자열 직접 치환 |
| SQL Injection | 안전 | 위험 |
| 사용 대상 | 값(Value) 바인딩 | 컬럼명·테이블명 동적 지정 시만 사용 |
| 실무 기본 | ✅ 항상 사용 | ⚠️ 불가피한 경우만 |

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| ResultMap | DB snake_case 컬럼 → Java camelCase 필드 자동 매핑 |
| `<id>` 태그 | PK 선언 — 캐시·동일성 판별에 영향 |
| _int (parameterType) | 언더스코어 접두사 = Java primitive 타입 |
| #{} | PreparedStatement 바인딩 — SQL Injection 방지 |
| selectOne | 결과 없으면 null, 두 건 이상이면 예외 발생 |
| insert 반환값 | 영향받은 행 수 → > 0 이면 성공 |
| Service commit/rollback | autoCommit=false이므로 Service가 직접 commit 호출 |
| finally close() | SELECT든 INSERT든 항상 SqlSession을 닫아야 한다 |
| 계층 분리 | SQL은 XML, 트랜잭션은 Service, 출력은 PrintResult — 각 계층이 한 가지 책임만 |
