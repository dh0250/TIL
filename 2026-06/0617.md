# chap06 — section06 조인 (Join)

JPQL의 조인은 SQL과 달리 **엔티티 간의 연관관계(필드)**를 기준으로 조인

## 엔티티 연관관계 설정

```java
// Menu.java — @ManyToOne (다대일, 연관관계의 주인)
@ManyToOne
@JoinColumn(name = "categoryCode")
private Category category;

// Category.java — @OneToMany (일대다, 양방향)
@OneToMany(mappedBy = "category")
private List<Menu> menuList;
```

---

## 조인 종류

### 1. 내부 조인 (INNER JOIN)

양쪽 모두 연관된 데이터만 조회 → **카테고리가 없는 메뉴는 제외**

```java
String jpql = "SELECT m FROM Section06Menu m JOIN m.category c";
List<Menu> result = entityManager.createQuery(jpql, Menu.class).getResultList();
```

- SQL의 `INNER JOIN`과 동일
- `JOIN` 키워드만 써도 내부 조인
- 테이블명이 아닌 **엔티티 필드명** (`m.category`)으로 조인

---

### 2. 외부 조인 (OUTER JOIN)

한쪽에 데이터가 없어도 포함하여 조회

```java
// RIGHT JOIN — 카테고리를 기준으로 메뉴가 없는 카테고리도 포함
String jpql = "SELECT m.menuName, c.categoryName FROM Section06Menu m RIGHT JOIN m.category c"
            + " ORDER BY m.category.categoryCode";
List<Object[]> result = entityManager.createQuery(jpql).getResultList();
```

- `LEFT JOIN` / `RIGHT JOIN` 모두 사용 가능
- 여러 필드를 선택하면 반환 타입은 `List<Object[]>`

---

### 3. 컬렉션 조인 (Collection Join)

`@OneToMany` 관계의 **컬렉션 필드**를 기준으로 조인

```java
// Category의 menuList 컬렉션을 통해 조인
String jpql = "SELECT c.categoryName, m.menuName FROM Section06Category c LEFT JOIN c.menuList m";
List<Object[]> result = entityManager.createQuery(jpql).getResultList();
```

- 다대일 관계의 반대 방향(일대다)에서 조인하는 방식
- `LEFT JOIN`을 사용해 메뉴가 없는 카테고리도 포함

---

### 4. 페치 조인 (FETCH JOIN) — 핵심

연관된 엔티티를 **한 번의 쿼리로 함께 즉시 로딩**하는 JPQL 전용 기능

```java
String jpql = "SELECT m FROM Section06Menu m JOIN FETCH m.category c";
List<Menu> result = entityManager.createQuery(jpql, Menu.class).getResultList();
```

- `JOIN FETCH` 키워드 사용
- 지연 로딩(LAZY) 설정이 있어도 **연관 엔티티를 즉시 로딩**하여 N+1 문제 해결
- 반환 타입은 엔티티 자체 (`List<Menu>`)

---

## 조인 비교 정리

| 조인 종류 | JPQL 문법 | 반환 타입 | 특징 |
| --- | --- | --- | --- |
| 내부 조인 | `JOIN m.category c` | `List<Menu>` | 양쪽 모두 존재하는 데이터만 조회 |
| 외부 조인 | `LEFT/RIGHT JOIN m.category c` | `List<Object[]>` | 한쪽 데이터 없어도 포함 |
| 컬렉션 조인 | `LEFT JOIN c.menuList m` | `List<Object[]>` | 일대다 컬렉션 필드 기준 조인 |
| 페치 조인 | `JOIN FETCH m.category c` | `List<Menu>` | 연관 엔티티 즉시 로딩, N+1 해결 |

---

## 핵심 포인트

| 항목 | 내용 |
| --- | --- |
| 조인 기준 | 테이블 컬럼이 아닌 **엔티티의 연관관계 필드** (`m.category`, `c.menuList`) |
| SQL 변환 | JPA가 방언에 맞는 SQL JOIN으로 자동 변환 |
| 페치 조인 | 지연 로딩으로 인한 **N+1 문제**를 해결하는 가장 일반적인 방법 |
| 다중 필드 조회 | 여러 필드 선택 시 `List<Object[]>` 로 받아 수동 처리 |

# chap06 — Spring Data JPA

## 개념

**Spring Data JPA**는 JPA 기반의 Repository 추상화 레이어

- `EntityManager`를 직접 다루지 않고 **인터페이스 선언만으로** CRUD + 페이징 + 정렬 제공
- 쿼리 메소드(메소드명 규칙), `@Query`, Pageable 등 다양한 방식으로 쿼리 작성 가능
- `JpaRepository<엔티티, PK타입>` 를 상속하면 기본 CRUD 메소드 자동 제공

---

## 1. JpaRepository — 기본 제공 메소드

`extends JpaRepository<Menu, Integer>` 선언만으로 아래 메소드를 즉시 사용 가능

```java
public interface MenuRepository extends JpaRepository<Menu, Integer> { }
```

| 메소드 | 반환 타입 | 설명 |
| --- | --- | --- |
| `findById(id)` | `Optional<T>` | PK로 단건 조회 |
| `findAll()` | `List<T>` | 전체 조회 |
| `findAll(Pageable)` | `Page<T>` | 전체 조회 + 페이징 |
| `save(entity)` | `T` | INSERT / UPDATE (PK 존재 여부로 구분) |
| `delete(entity)` | `void` | DELETE |
| `existsById(id)` | `boolean` | 존재 여부 확인 |
| `count()` | `long` | 전체 개수 |

---

## 2. 쿼리 메소드 (Query Method)

**메소드 이름 규칙**에 따라 JPA가 자동으로 JPQL을 생성

```java
public interface MenuRepository extends JpaRepository<Menu, Integer> {

    // 전달받은 가격 초과 메뉴 목록
    List<Menu> findByMenuPriceGreaterThan(Integer menuPrice);

    // 가격 초과 + 가격 기준 내림차순 정렬
    List<Menu> findByMenuPriceGreaterThanOrderByMenuPriceDesc(Integer menuPrice);

    // 가격 초과 + 페이징 (Pageable로 동적 정렬도 지원)
    Page<Menu> findByMenuPriceGreaterThan(Integer menuPrice, Pageable pageable);
}
```

### 주요 키워드 패턴

| 키워드 | 예시 메소드명 | 의미 |
| --- | --- | --- |
| `findBy` | `findByMenuName` | WHERE 조건 시작 |
| `GreaterThan` | `findByMenuPriceGreaterThan` | `>` 조건 |
| `LessThan` | `findByMenuPriceLessThan` | `<` 조건 |
| `Between` | `findByMenuPriceBetween` | `BETWEEN` |
| `Like` | `findByMenuNameLike` | `LIKE` |
| `OrderBy...Desc/Asc` | `...OrderByMenuPriceDesc` | 정렬 조건 |
| `And` / `Or` | `findByNameAndPrice` | 복합 조건 |

---

## 3. 페이징 (Pagination)

### Repository — Pageable 파라미터

```java
// Page<T>로 반환 → 페이지 메타정보(총 개수, 총 페이지 수 등) 포함
Page<Menu> findAll(Pageable pageable);
Page<Menu> findByMenuPriceGreaterThan(Integer menuPrice, Pageable pageable);
```

### Controller — @PageableDefault

```java
@GetMapping
public ResponseEntity<Page<MenuResponseDTO>> findMenuList(
        @PageableDefault(size = 30, sort = "menuCode") Pageable pageable) {

    return ResponseEntity.ok(menuService.findMenuList(pageable));
}
```

- `@PageableDefault` : 클라이언트가 파라미터를 보내지 않을 때의 기본값 설정
- 클라이언트는 `?page=0&size=10&sort=menuPrice,desc` 형식으로 동적 제어 가능

### Service — Page 변환

```java
public Page<MenuResponseDTO> findMenuList(Pageable pageable) {
    Page<Menu> menuList = menuRepository.findAll(pageable);

    // Page.map()으로 엔티티 → DTO 변환 (페이징 메타정보 유지)
    return menuList.map(menu -> modelMapper.map(menu, MenuResponseDTO.class));
}
```

> `Page.map()` 사용 시 페이지 메타정보(totalElements, totalPages 등)가 그대로 유지된다.

## 4. DTO 분리와 ModelMapper

### DTO 구조

```java
// 요청 수신용 — 클라이언트 입력 필드만
public class MenuRequestDTO {
    private String menuName;
    private int menuPrice;
    private int categoryCode;       // Category 엔티티 대신 코드만 받음
    private String orderableStatus;
}

// 응답 반환용 — 중첩 DTO 포함
public class MenuResponseDTO {
    private int menuCode;
    private String menuName;
    private int menuPrice;
    private String orderableStatus;
    private CategoryDTO category;   // 엔티티 대신 DTO로 노출
}
```

### ModelMapper Bean 설정

```java
@Bean
public ModelMapper modelMapper() {
    ModelMapper modelMapper = new ModelMapper();
    modelMapper.getConfiguration()
            .setFieldAccessLevel(AccessLevel.PRIVATE)  // private 필드 직접 접근
            .setFieldMatchingEnabled(true);             // 필드명 기준 매칭
    return modelMapper;
}
```

> `setFieldAccessLevel(PRIVATE)` 없이는 Lombok `@Getter` 없는 `PROTECTED` 생성자 엔티티에서 매핑 실패 가능

## 5. 엔티티 설계 — Lombok + Builder 패턴

```java
@Entity
@Table(name = "tbl_menu")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)   // JPA 프록시용 기본 생성자 (외부 직접 생성 차단)
@AllArgsConstructor(access = AccessLevel.PRIVATE)    // @Builder가 내부적으로 사용
@Builder
public class Menu {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int menuCode;
    private String menuName;
    private int menuPrice;

    @ManyToOne
    @JoinColumn(name = "category_code")   // FK 컬럼명 명시
    private Category category;

    private String orderableStatus;

    // 수정 메소드 — setter 대신 의도가 명확한 메소드로 캡슐화
    public void modify(String menuName, int menuPrice, Category category, String orderableStatus) { ... }
    public void modifyMenuPrice(int menuPrice) { ... }
    public void modifyMenuName(String menuName) { ... }
}
```

| Lombok 어노테이션 | 접근 제한 | 목적 |
| --- | --- | --- |
| `@NoArgsConstructor(PROTECTED)` | `protected` | JPA 스펙 충족, 외부 직접 생성 방지 |
| `@AllArgsConstructor(PRIVATE)` | `private` | `@Builder` 내부 전용 생성자 |
| `@Builder` | `public` | 객체 생성 시 Builder 패턴 강제 |

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| Repository 선언 | `JpaRepository<엔티티, PK>` 상속만으로 기본 CRUD + 페이징 즉시 사용 |
| 쿼리 메소드 | 메소드명 규칙으로 자동 JPQL 생성, `Pageable` 파라미터로 페이징 추가 |
| 페이징 반환 | `Page<T>` — 데이터 + 메타정보(총 개수, 페이지 수) 포함 |
| DTO 분리 | 요청(`RequestDTO`) / 응답(`ResponseDTO`) 분리, 엔티티 직접 노출 금지 |
| ModelMapper | `PRIVATE` 접근 + 필드 매칭 활성화 필요, `Page.map()`으로 변환 시 메타정보 유지 |
| 엔티티 생성자 | `PROTECTED` 기본 생성자 + `@Builder` 조합으로 불변성 + JPA 호환성 확보 |
