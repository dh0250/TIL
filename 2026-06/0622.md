# chap07 — 세션 인증 vs 토큰(JWT) 인증

## 1. 인증 방식 개요

웹 애플리케이션에서 사용자의 신원을 확인하고 상태를 유지하기 위한 대표적인 두 가지 메커니즘. **"상태를 서버가 관리할 것인가(Stateful), 클라이언트가 관리할 것인가(Stateless)"** 가 핵심 차이점

`[세션 방식]      클라이언트  --(Cookie: 세션ID)-->  서버  -->  세션 저장소 (Stateful)
[토큰(JWT) 방식] 클라이언트  --(Header: JWT)----->  서버 (자체 검증)         (Stateless)`

## 2. 세션(Session) 인증 방식

### 인증 흐름

1. **로그인 요청**: 클라이언트가 서버에 ID와 비밀번호(ID / PW)를 전송함.
2. **세션 생성 후 세션 ID 발급**: 서버는 세션 저장소(서버 메모리, Redis, DB 등)에 유저 정보를 저장하고, 고유한 세션 ID를 생성하여 클라이언트에 응답 헤더(`Set-Cookie: JSESSIONID=abc123`)로 발급함.
3. **이후 요청 시, 쿠키 자동 전송**: 클라이언트는 브라우저의 쿠키 저장소에 이를 보관하며, 이후 요청마다 `Cookie: JSESSIONID=abc123`를 자동으로 포함하여 전송함.
4. **인증 성공**: 서버는 요청과 함께 넘어온 세션 ID를 세션 저장소에서 조회하여 신원이 확인되면 인증을 성공 처리함.

### 특징

- **Stateful (상태 유지)**: 서버가 클라이언트의 로그인 상태 정보를 직접 저장하고 유지함.
- **높은 보안성**: 실제 민감한 유저 정보는 서버에만 존재하고, 클라이언트는 임의의 문자열인 세션 ID만 가짐.
- **통제 용이성**: 서버에서 세션 저장소의 데이터를 삭제함으로써 원격으로 강제 로그아웃을 처리할 수 있음.
- **서버 확장성 제약**: 서버 인프라를 확장(Scale-out)할 때, 여러 서버가 동일한 세션 정보를 공유할 수 있도록 Redis나 세션 DB 같은 별도의 인프라 구축이 필요함.

## 3. 토큰(Token, JWT) 인증 방식

### 인증 흐름

1. **로그인 요청**: 클라이언트가 서버에 ID와 비밀번호를 전송함.
2. **토큰 생성 후 발급**: 서버는 유저 정보를 기반으로 암호화/서명된 문자열인 JWT(토큰)를 생성하여 클라이언트에 발급함. (페이로드에 id, name, role, exp 등이 포함됨)
3. **클라이언트에 토큰 저장**: 클라이언트는 발급받은 토큰을 Local Storage, Session Storage 또는 Cookie에 저장함.
4. **이후 요청 시, 헤더에 토큰 포함**: 클라이언트가 요청할 때 HTTP 요청 헤더에 `Authorization: Bearer [토큰]` 형태로 포함하여 전송함.
5. **토큰 검증**: 서버는 세션 저장소를 조회하지 않고, 토큰 자체의 서명(Signature) 검증, 만료 시간 검증, 위·변조 검증을 거쳐 정상적인 토큰이면 인증을 성공 처리함.

### 특징

- **Stateless (무상태)**: 서버가 클라이언트의 상태를 저장하지 않으므로 서버의 부담이 줄어듦.
- **뛰어난 확장성**: 토큰 자체에 인증에 필요한 정보가 포함되어 있어, 별도의 저장소 공유 없이도 어떤 서버에서나 자체 검증이 가능함.
- **현대적 환경 적합성**: 웹 브라우저뿐만 아니라 쿠키 관리가 까다로운 모바일 앱(iOS/Android), SPA(Single Page Application), MSA(Microservice Architecture) 환경에 매우 적합함.
- **강제 제어의 어려움**: 한 번 발급된 토큰은 서버가 제어할 수 없으므로, 토큰이 탈취되더라도 만료 전까지는 강제로 로그아웃(유효화 취소) 시키기가 어려움.

## 4. 세션 vs 토큰 비교

| **항목** | **세션 (Session)** | **토큰 (JWT)** |
| --- | --- | --- |
| **상태 관리** | 서버가 세션 정보를 관리 (Stateful) | 클라이언트가 토큰을 관리 (Stateless) |
| **서버 저장소** | 필요 (메모리, Redis, DB 등) | 불필요 |
| **보안 (정보 저장 위치)** | 민감 정보는 서버 저장소에 안전하게 보관 | 토큰 자체에 정보 포함 (서명 및 암호화 필요) |
| **강제 로그아웃** | 가능 (서버에서 세션 삭제 시 즉시 만료) | 어려움 (토큰이 만료되기 전까지 유효) |
| **확장성** | 세션 공유 인프라 필요 $\rightarrow$ 확장 어려움 | 각 서버에서 자체 검증 가능 $\rightarrow$ 확장 용이 |
| **모바일 앱 적합성** | 쿠키 메커니즘 의존적 $\rightarrow$ 구현이 불편함 | HTTP 헤더 기반 $\rightarrow$ 매우 적합함 |
| **구현 난이도** | 비교적 쉬움 (프레임워크 자체 지원 내장) | 보통 (인코딩/디코딩 및 검증 로직 구현 필요) |
| **토큰 크기 / 네트워크** | 작음 (단순한 세션 ID 문자열) | 큼 (Header, Payload, Signature 포함으로 1~2KB) |
| **주요 사용 예시** | 전통적인 웹 서비스, 관리자 시스템, 사내 시스템 등 | 모바일 앱, SPA(React/Vue), MSA, 오픈 API 서비스 등 |

## 5. JWT 구조

JWT는 점(`.`)을 구분자로 하여 세 가지 부분으로 구성되어 있으며, `Header.Payload.Signature` 형식을 가짐. (예시: `aaaaa.bbbbb.ccccc`)

`Header (헤더)      --> 토큰의 타입(JWT) 및 사용된 해싱 알고리즘(예: HS256) 정의
  .
Payload (페이로드)  --> 내용(Claims)을 담는 곳으로, 사용자 ID, 권한(role), 만료시간(exp) 등 포함
  .
Signature (서명)   --> 위·변조를 방지하기 위해 헤더와 페이로드를 서버의 비밀키로 해싱한 값`

> **핵심**: 서버는 요청으로 들어온 JWT의 `Signature`를 검증하여 토큰이 도중에 위·변조되었는지 판단하며, 서명이 일치하면 `Payload`에 담긴 사용자 정보를 신뢰하고 별도 조회 없이 인증을 수행함.
>

## 6. 서버 확장 환경에서의 차이

- **세션 방식**: 로드 밸런서(Load Balancer)를 통해 트래픽이 여러 서버(Server A, Server B, Server C)로 분산될 때, 유저가 로그인한 서버가 아닌 다른 서버로 요청이 가면 로그인이 풀림. 이를 해결하기 위해 모든 서버가 접근할 수 있는 **중앙 세션 저장소(Redis 등)나 데이터베이스 인프라가 필수적으로 요구**됨.
- **JWT 방식**: 토큰 검증에 필요한 비밀키(Secret Key)만 모든 서버가 공유하고 있다면, 로드 밸런싱에 의해 **어떤 서버로 요청이 유입되더라도 자체적으로 토큰을 검증**할 수 있어 별도의 저장소 인프라가 불필요함.

## 7. 실무에서의 선택 기준

### 세션을 선택하는 경우

- 순수 브라우저 기반의 웹 서비스일 때
- 서버 수가 적고 아키텍처 구조가 단순할 때
- 금융, 결제, 회원 관리 등 보안 통제 및 강제 로그아웃 기능이 매우 중요할 때
- 관리자 전용 시스템 또는 사내 인트라넷 시스템을 구축할 때
- **기술 스택 예시**: Spring MVC + Thymeleaf, JSP, Django, Laravel 등

### JWT를 선택하는 경우

- iOS, Android 등 모바일 네이티브 앱과 연동해야 할 때
- React, Vue, Next.js 등 프론트엔드와 백엔드가 완전히 분리된 SPA 구조일 때
- 다수의 마이크로서비스가 통신하는 MSA(Microservice Architecture) 환경일 때
- 외부 서드파티에 제공하는 RESTful API 서비스를 설계할 때
- **기술 스택 예시**: Spring Boot + React, Node.js + Express, FastAPI 등

## 8. 현대적인 인증 구조 (권장 패턴)

JWT의 탈취 위험성과 강제 로그아웃이 불가능한 단점을 보완하기 위해, 실무에서는 **Access Token**과 **Refresh Token**을 혼합하여 사용하는 방식을 주로 권장함.

`[로그인 성공]  -->  Access Token (짧은 유효기간: 예 30분)  &  Refresh Token (긴 유효기간: 예 2주) 동시 발급
[일반 요청]    -->  Access Token을 HTTP 헤더에 담아서 API 요청 수행
[토큰 만료]    -->  Access Token 만료 시, 안전하게 보관해 둔 Refresh Token을 보내어 새로운 Access Token 재발급`

- **Access Token**: 유효 기간을 매우 짧게 가져가서(예: 30분~2시간) 탈취당했을 때의 리스크를 최소화함. 실제 리소스(API) 접근에 사용됨.
- **Refresh Token**: 유효 기간을 길게 설정하며(예: 1주~2주), 오직 Access Token을 재발급받는 용도로만 사용됨. 이 토큰은 데이터베이스나 Redis 등의 저장소에 기록하여 탈취 시 서버에서 강제로 무효화할 수 있는 안전장치를 마련해 둠.

## 핵심 요약 (추가분)

| **항목** | **내용** |
| --- | --- |
| **세션 인증** | 서버 저장소 기반 관리, 보안 제어 용이(Stateful), 서버 확장 시 추가 인프라(Redis) 필요 |
| **토큰(JWT) 인증** | 클라이언트 보관 및 서버 자체 검증, 확장 용이(Stateless), 모바일/SPA/MSA에 최적화 |
| **JWT 구조** | `Header.Payload.Signature`로 구성, 서버는 Signature를 통해 위·변조 여부 검증 |
| **서버 확장성** | 세션은 세션 동기화(공유)가 수반되나, JWT는 비밀키만 있으면 각 서버 독립 검증 가능 |
| **기술 스택 궁합** | 세션은 서버 사이드 렌더링(Thymeleaf 등) 중심, JWT는 분리형 아키텍처(React, 모바일 앱) 중심 |
| **현대적 권장 패턴** | Access Token(단기)과 Refresh Token(장기)을 조합하여 보안성과 편의성을 동시에 확보 |
