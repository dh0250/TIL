# chap02 — 엔티티 매핑 (Entity Mapping)

## 개념

객체(Java)와 관계형 데이터베이스(DB)의 테이블을 어떻게 연결할 것인지 정의하는 과정

- JPA의 핵심은 객체와 테이블을 정확히 매핑하는 것
- 매핑 어노테이션을 통해 DB 스키마 자동 생성 및 SQL 생성을 제어함
- 데이터 타입, 제약 조건, 기본키 생성 전략 등을 설정

## 주요 매핑 어노테이션

| 어노테이션 | 설명 | 주요 속성 |
| --- | --- | --- |
| `@Entity` | 클래스를 테이블과 매핑할 엔티티로 지정 | `name`: JPQL에서 사용할 이름 |
| `@Table` | 엔티티와 매핑할 실제 DB 테이블 명시 | `name`: 테이블명 (ex: tbl_member) |
| `@Id` | 테이블의 기본키(PK) 매핑 | 필드에 선언하여 PK 임을 명시 |
| `@GeneratedValue` | 기본키 생성 전략 설정 | `IDENTITY(MySQL 등)`, `SEQUENCE(Oracle 등)` |
| `@Column` | 필드를 컬럼에 매핑 | `unique`, `nullable`, `length`, `columnDefinition` |
| `@Enumerated` | 자바의 enum 타입을 매핑 | `EnumType.STRING (이름 저장, 권장)` |
| `@Access` | 엔티티 데이터 접근 방식 | `FIELD(필드 직접 접근)`, `PROPERTY(Getter 접근)` |
| `@Transient` | 매핑에서 제외 | DB에 저장/조회하지 않는 필드 |

---

## 상세 매핑 설정 (Member.java 예시)

### 1. 기본키 및 컬럼 제약 조건

- IDENTITY 전략: DB의 auto_increment를 사용하여 번호를 생성함
- columnDefinition: `varchar(10)` 또는 `default 'Y'` 처럼 특정 SQL 구문을 직접 지정 가능
- length: 문자열 길이를 제한 (기본값 255, 주소 등은 900으로 확장 가능)

### 2. 접근 타입 (@Access)

- `AccessType.FIELD`: 필드에 직접 접근 (private이어도 접근 가능)
- `AccessType.PROPERTY`: Getter 메소드를 통해 접근 (Getter 내 로직 추가 가능)

# section02 — 임베디드 타입 (Embedded Type)

## 개념

새로운 값 타입을 직접 정의해서 사용하는 것으로, JPA에서는 이를 임베디드 타입이라 함

| 어노테이션 | 설명 |
| --- | --- |
| `@Embeddable` | 값 타입을 정의하는 곳에 표시 (ex: Price 클래스) |
| `@Embedded` | 값 타입을 사용하는 곳에 표시 (ex: Book 클래스의 price 필드) |

---

## 특징 및 장점

- 응집도 향상: 가격(정가, 할인율, 판매가)처럼 관련된 필드를 하나의 클래스로 묶어 관리
- 재사용성: 여러 엔티티에서 동일한 값 타입을 공유 가능
- 객체지향적: Price 클래스 내부에 할인율 계산(calcSellPrice)이나 검증 로직을 포함할 수 있음

---

## 핵심 로직 동작 요약

```java
// 1. 임베디드 타입 활용 (Price.java)
public class Price {
    private int regularPrice;
    private double discountRate;
    private int sellPrice; // 내부 로직으로 자동 계산 가능

    public Price(int regularPrice, double discountRate) {
        // 검증 및 계산 로직 포함
        this.sellPrice = (int) (regularPrice - (regularPrice * discountRate));
    }
}

// 2. 서비스 로직 (BookService.java)
@Transactional
public void registBook(BookRegistDTO newBook) {
    Price price = new Price(newBook.getRegularPrice(), newBook.getDiscountRate());
    Book book = new Book(
        newBook.getBookTitle(),
        newBook.getAuthor(),
        newBook.getPublisher(),
        newBook.getPublishedDate(),
        price // 임베디드 객체 주입
    );
    bookRepository.save(book);
}
```

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| `@Enumerated` | 반드시 `EnumType.STRING`을 사용해야 enum 순서 변경 시 발생하는 장애를 방지함 |
| `@Access` | 명시하지 않으면 `@Id`의 위치(필드/Getter)에 따라 접근 방식이 결정됨 |
| 임베디드 타입 | DB 테이블상에서는 평범한 컬럼들로 풀려서 저장되지만, 객체로 관리하여 코드 가독성 증대 |
| `ddl-auto: create` | 애플리케이션 실행 시 매핑 어노테이션을 기반으로 기존 테이블을 드랍하고 새로 생성함 |
| 기본 생성자 | 엔티티 및 임베디드 클래스 모두 JPA가 객체를 생성할 수 있도록 기본 생성자(protected 이상) 필수 |

# chap03 — 연관 관계 매핑 (Association Mapping)

## 개념

객체의 참조와 테이블의 외래 키(FK)를 매핑하여 객체 간의 관계를 DB에 반영하는 과정

- 연관 관계의 주인: 외래 키를 관리하는 엔티티 (보통 N쪽이 주인)
- 로딩 전략: 연관된 객체를 가져오는 시점 결정 (EAGER/LAZY)
- 영속성 전이(Cascade): 부모의 상태 변화를 자식에게 전파하는 설정

---

## 주요 매핑 어노테이션

| 어노테이션 | 설명 | 주요 속성 |
| --- | --- | --- |
| `@ManyToOne` | 다대일(N:1) 관계 매핑 | `fetch`, `cascade` |
| `@OneToMany` | 일대다(1:N) 관계 매핑 | `mappedBy`, `fetch`, `cascade` |
| `@JoinColumn` | 외래 키(FK) 컬럼 지정 | `name`: DB 컬럼명 (기본값: 필드명_PK명) |

---

## Fetch 전략 (로딩 방식)

| 전략 | 설명 | 기본값 |
| --- | --- | --- |
| `EAGER (즉시)` | 엔티티 조회 시 연관된 객체도 JOIN으로 함께 조회 | `@ManyToOne`, `@OneToOne` |
| `LAZY (지연)` | 연관된 객체를 실제 사용할 때(Getter 호출 등) 조회 | `@OneToMany`, `@ManyToMany` |

---

# section01 — 다대일 (Many-to-One)

## 엔티티 매핑 예시

```java
@Entity
public class Menu {
    @Id
    private int menuCode;

    // N:1 관계, 카테고리가 없어도 메뉴 저장 가능하도록 설정 가능
    // CascadeType.PERSIST: 메뉴 저장 시 새로운 카테고리 객체도 함께 저장
    @ManyToOne(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
    @JoinColumn(name = "categoryCode")
    private Category category;
}
```

## 특징

- 가장 권장되는 방식 (DB의 FK 구조와 객체의 참조 구조가 일치함)
- `FetchType.LAZY` 사용을 통해 불필요한 조인을 방지하고 성능 최적화 가능

---

# section02 — 일대다 (One-to-Many)

## 엔티티 매핑 예시

```java
@Entity
public class Category {
    @Id
    private int categoryCode;

    // 1:N 관계, 기본값은 LAZY이나 테스트 목적으로 EAGER 설정 가능
    @OneToMany(fetch = FetchType.EAGER)
    @JoinColumn(name = "categoryCode") // FK가 Menu 테이블에 있음을 명시
    private List<Menu> menuList;
}
```

## 특징

- 일(1)이 다(N)를 관리하는 형태
- 매핑 주체가 아닌 테이블(Menu)에 외래 키가 있어 관리가 복잡할 수 있음
- 엔티티 수정 시 다른 테이블에 대한 UPDATE 쿼리가 추가로 발생함

---

# 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 연관 관계 주인 | 외래 키가 있는 곳(보통 Many 쪽)이 주인이 되어야 관리가 효율적임 |
| LAZY vs EAGER | 실무에서는 N+1 문제 방지를 위해 전부 LAZY로 설정하고 필요시 fetch join 사용 |
| CascadeType | 부모 엔티티 조작 시 자식까지 한꺼번에 처리할 때 사용 (영속성 전이) |
| 단방향 vs 양방향 | 처음엔 단방향으로 설계하고, 반대 방향 조회가 필요할 때만 양방향 추가 |
