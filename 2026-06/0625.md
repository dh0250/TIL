# GitHub 이슈 관리 & 협업 규칙 정리

## 1. 브랜치 전략

```
main (락 걸림 - 직접 푸시 불가)
 └── dev
       └── feature/BE-[기능명]
```

- 모든 작업은 `feature/*` 브랜치에서 진행
- 완성 후 `dev`로 PR
- `dev` → `main`은 팀 전체 병합

**브랜치 네이밍 규칙**

```
feature/BE-admin-init
feature/BE-admin-member
feature/BE-admin-inquiry
```

---

## 2. 이슈 관리

**이슈 등록 시 필수 항목**

- Labels: `feat`, `BE`
- Assignees: 본인
- Milestone: 핵심 기능 개발 (6/25 ~ 7/3)
- Priority / Start date / Target date 설정

**이슈 템플릿**

```
## 작업 목표
- 구현할 내용을 bullet point로 작성

## 관련 문서
- API 명세서, ERD 링크 등

## 완료 조건 (Definition of Done)
- [ ] 체크리스트 형식으로 완료 기준 명시

## 참고 사항
- 선택 기재
```

**이슈 상태 흐름**

```
Todo → In Progress → Done
```

- 작업 시작 시 `In Progress`로 변경
- PR 머지 후 자동 `Close` (Closes # 사용 시)
- 이슈는 삭제 불가, Close만 가능
- 중복 이슈는 `duplicate`로 Close

---

## 3. 커밋 메시지 컨벤션

**형식**

```
[타입]: 작업 내용 #이슈번호
```

**타입 종류**

| 타입 | 설명 |
| --- | --- |
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `refactor` | 코드 개선 |
| `docs` | 문서 수정 |
| `test` | 테스트 코드 |
| `chore` | 빌드/의존성 변경 |
| `design` | UI 변경 |
| `style` | 포맷 변경 |

**예시**

```
feat: 관리자 회원 목록 조회 API 구현 #58
fix: 유통기한 만료 필터 조건 오류 수정 #23
chore: PR 템플릿 추가
```

**주의사항**

- `Closes #번호`는 커밋 메시지에 쓰지 않음
- `#번호`는 참조용으로만 사용

---

## 4. PR 규칙

**PR 제목 형식**

```
[feat/BE] 관리자 회원 목록 조회 API 구현
[fix/FE] 레시피 이미지 깨짐 수정
```

**PR 본문 템플릿** (`.github/PULL_REQUEST_TEMPLATE.md`)

```markdown
## 작업 내용
- 구현한 내용 bullet point로 작성

## 관련 이슈
- closes #이슈번호

## 테스트 방법
- 로컬 테스트 방법 기재

## 참고 사항
- 리뷰어가 알아야 할 정보
```

**PR 규칙**

- 반드시 `dev` 브랜치로 PR
- 최소 1명 팀원 리뷰 후 머지
- 본인 PR 본인 머지 금지
- `closes #번호` 쓰면 머지 시 이슈 자동 Close
- `#번호`만 쓰면 연결만 되고 Close 안 됨

---

## 5. 이슈 - 커밋 - PR 연결 흐름

```
이슈 등록 (#58)
    ↓
브랜치 생성 (feature/BE-admin-member)
    ↓
커밋 (feat: 관리자 회원 목록 조회 API 구현 #58)
    ↓
PR 생성 (관련 이슈: closes #58)
    ↓
코드 리뷰 & 승인
    ↓
dev 머지 → 이슈 자동 Close
```

---

## 6. 확인한 내용

- `dev` 브랜치도 protected → 직접 푸시 불가, PR로만 가능
- PR 템플릿은 `.github/PULL_REQUEST_TEMPLATE.md` 별도 파일 필요
- 이슈는 삭제 불가, 중복이면 `duplicate`로 Close
- `Closes #번호`는 **PR 본문**에만, 커밋 메시지엔 `#번호`만
