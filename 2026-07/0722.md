# 챗봇 Memory 실습 (06_chatbot_memory 01~07)

LLM 없이 규칙 기반으로 대화 기록(`messages`), 사용자 정보(`profile`), 요약(`summary`)을 관리하는 `ChatMemory` 클래스를 처음부터 만들어보고, Memory Window·민감정보 차단·요약·세션별 영속화·실제 LangChain 프롬프트 결합까지 단계적으로 확장하는 실습.

# Rule-based Chat Memory 기본 구조 (01_chatbot_memory.py, 02_memory_practice.py)

`"내 이름은 민수야"`처럼 정해진 표현이 들어오면 문자열 `split`으로 값을 뽑아 `profile` dict에 저장하고, `"내 이름이 뭐였지?"` 같은 질문에는 저장된 profile을 그대로 되돌려주는 최소 Memory 챗봇.

```python
def remember_profile(self, user_message):
    if "내 이름은" in user_message:
        name = clean_korean_suffix(user_message.split("내 이름은", 1)[1])
        if name:
            self.profile["name"] = name

def generate_reply(user_message, memory):
    memory.remember_profile(user_message)
    context = memory.context()
    if "내 이름" in user_message and "뭐" in user_message:
        name = context["profile"].get("name")
        return f"이름은 {name}라고 하셨어요." if name else "아직 이름을 듣지 못했어요."
```

- `clean_korean_suffix`로 "민수야", "민수입니다" 같은 어미를 잘라내 값만 깔끔하게 저장한다. LLM 없이도 간단한 규칙만으로 "정보 저장 → 이후 질문에 반영"이라는 Memory의 핵심 동작을 구현할 수 있음을 보여주는 예제.
- `--interactive` 옵션으로 직접 입력하며 `/memory`(현재 profile·summary·최근 대화 확인), `/help` 명령을 테스트해볼 수 있다.

# 여러 시나리오로 Memory 동작 확인 (02_memory_scenarios.py)

`importlib.util`로 `02_memory_practice.py`를 동적으로 불러와, "기본 기억", "정보 변경", "없는 정보 질문", "전체 profile 요약" 네 가지 시나리오를 돌려보며 Memory가 예상대로 동작하는지 검증하는 테스트 스크립트.

```python
SCENARIOS = {
    "정보 변경": [
        "내 이름은 민수야",
        "내 이름은 지훈이야",
        "내 이름이 뭐였지?"
    ],
    ...
}
```

- "정보 변경" 시나리오에서 이름을 두 번 다르게 말하면 `profile["name"]`이 마지막 값(지훈)으로 덮어써지는 것을 확인 — Memory는 누적이 아니라 최신 값으로 갱신된다는 점을 보여준다.
- "없는 정보 질문" 시나리오로 저장된 적 없는 목표를 물으면 "아직 모릅니다" 류의 안전한 fallback 응답이 나오는지 확인.

# Memory Window 실험 (03_window_memory_lab.py)

같은 10턴 대화를 `max_recent_messages`(Window 크기) 2/4/6/7로 각각 저장해보면서, Window가 좁을수록 `recent_messages()`에서 초반 대화(이름 소개 등)가 먼저 잘려나가는 것을 확인.

```python
for window_size in (2, 4, 6, 7):
    memory = ChatMemory(max_recent_messages=window_size)
    for role, content in CONVERSATION:
        memory.add_message(role, content)
        if role == "user":
            memory.remember_profile(content)
    print(memory.recent_messages())
    print("Profile:", memory.profile)  # 최근 대화에서 이름이 사라져도 profile에는 남아있다
```

- 핵심 포인트: `recent_messages()`(대화창에 보여줄 최근 N개)와 `profile`(장기 기억)은 분리되어 있어서, Window가 좁아 "내 이름은 민수야"라는 원문이 최근 대화에서 사라져도 `profile["name"]`은 그대로 남아있다. 이게 Memory Window와 Profile(장기 기억)을 굳이 나누는 이유.

# Profile 삭제 & 민감정보 차단 (04_profile_privacy_lab.py, memory_core.py)

이름 삭제(`"이름을 잊어"`)·전체 삭제(`"모든 기억을 삭제"`) 명령을 처리하는 `forget_profile()`과, `memory_core.py`의 `contains_sensitive_info()`로 비밀번호·주민등록번호·API 키 패턴이 들어오면 원문 자체를 저장하지 않고 걸러내는 로직.

```python
SENSITIVE_KEYWORDS = ("비밀번호", "패스워드", "주민등록번호", "계좌번호", "api key", ...)

def contains_sensitive_info(text: str) -> bool:
    if any(keyword in text.lower() for keyword in SENSITIVE_KEYWORDS):
        return True
    if re.search(r"\bsk-[A-Za-z0-9_-]{8,}\b", text):      # OpenAI API 키 패턴
        return True
    if re.search(r"\b\d{6}-[1-4]\d{6}\b", text):           # 주민등록번호 패턴
        return True
    return False

def add_message(self, role, content):
    safe_content = "[민감정보가 제거된 메시지]" if contains_sensitive_info(content) else content.strip()
    self.messages.append({"role": role, "content": safe_content, ...})
```

- 민감정보 검사를 `remember_profile`뿐 아니라 `add_message` 단계에서도 걸어서, profile에 안 남기는 것은 물론 대화 기록(`messages`) 원문에도 아예 남기지 않도록 이중으로 막았다.
- `forget_profile(key=None)`은 특정 key 하나만 지우거나 인자 없이 호출하면 profile 전체를 초기화 — 실제 서비스의 "기억 삭제해줘" 요청에 대응하는 최소 구현.

# 전체 대화 vs Summary Memory 길이 비교 (05_summary_memory.py)

profile 4개 + RAG 관련 대화 8턴을 쌓은 뒤, "전체 대화 그대로"와 "규칙 기반 요약(summary)"의 글자 수를 비교해 Summary Memory가 프롬프트 길이를 얼마나 줄여주는지 수치로 확인.

```python
full_history = format_recent_messages(memory.messages)
recent_history = format_recent_messages(memory.recent_messages())
summary = memory.update_summary()

print("전체 대화 글자 수:", len(full_history))
print("최근 대화 글자 수:", len(recent_history))
print("요약 글자 수: ", len(summary))
```

- 실제 서비스에서는 이 요약을 LLM이 생성하지만, 여기서는 profile 값들을 문장으로 이어붙이는 규칙 기반 요약으로 대체 — 그래도 전체 대화 대비 압축률을 확인하는 목적은 동일하게 달성된다.
- 대화가 길어질수록(오늘처럼 RAG 질문 8개 추가) "최근 N개만 유지"(Window)와 "요약으로 압축"(Summary) 두 전략을 같이 써야 토큰을 아끼면서도 맥락을 유지할 수 있다는 걸 보여준다.

# Session별 Memory 영속화 (06_session_persistence.py)

`session_id`(예: `student-a`, `student-b`)마다 독립된 `ChatMemory`를 관리하는 `MemoryStore`를 만들고, 전체 세션을 JSON 파일(`runtime/session_memories.json`)로 저장했다가 다시 읽어 복원하는 실습.

```python
def save(self, path: Path) -> None:
    data = {sid: memory.to_dict() for sid, memory in self.sessions.items()}
    path.write_text(json.dumps(data, ensure_ascii=False, indent=2), encoding="utf-8")

@classmethod
def load(cls, path: Path) -> "MemoryStore":
    store = cls()
    if not path.exists():
        return store
    data = json.loads(path.read_text(encoding="utf-8"))
    store.sessions = {sid: ChatMemory.from_dict(d) for sid, d in data.items()}
    return store
```

- `ChatMemory.to_dict()` / `from_dict()`가 직렬화·복원을 담당하기 때문에 `MemoryStore`는 파일 입출력에만 집중하면 된다 — 역할 분리 덕분에 저장 형식이 바뀌어도 `ChatMemory` 쪽만 수정하면 된다.
- 프로그램을 재시작한 상황을 가정해 `MemoryStore.load()`로 복원한 뒤에도 `student-a`와 `student-b`의 profile이 서로 섞이지 않고 각자 유지되는 것을 확인 — 실제 서비스에서 사용자별 세션을 격리해야 하는 이유를 보여주는 예제.

# LangChain 프롬프트에 Memory 결합 (07_openai_memory_chatbot.py)

`ChatMemory`의 profile·summary·최근 대화를 하나의 프롬프트 문자열로 합쳐 `ChatOpenAI(gpt-4o-mini)`에 넘기는, Memory와 실제 LLM 호출을 잇는 마지막 단계.

```python
def build_prompt(memory: ChatMemory, user_message: str) -> str:
    context = memory.context()
    return f"""
    너는 AI 수업 도우미다.
    사용자의 학습 수준과 선호를 반영하되, 기억에 없는 정보는 추측하지 마라.

    [Profile]
    {context['profile'] or '저장된 profile 없음'}

    [Summary]
    {context['summary'] or '저장된 summary 없음'}

    [Recent Messages]
    {format_recent_messages(context['recent_messages'])}

    [Current User Message]
    {user_message}
    """
```

- `--run` 플래그가 없으면 실제 API 호출 없이 완성된 프롬프트만 출력하는 dry-run 모드로 동작 — API 비용 없이 "Memory가 프롬프트에 어떻게 녹아드는지"만 먼저 확인할 수 있다(어제 RAG 실습의 dry-run 패턴과 동일).
- `memory.remember_profile(user_message)`를 프롬프트 생성 **전에** 먼저 호출해서, "내 수준은 Python 초보자야"라고 말한 바로 그 턴의 답변부터 이미 profile이 반영되도록 순서를 맞췄다.
- 프롬프트에 "기억에 없는 정보는 추측하지 마라"는 제약을 명시해, RAG에서 썼던 "근거 없으면 모른다고 답하라"는 환각 방지 패턴을 Memory 프롬프트에도 동일하게 적용했다.

# 공통 모듈 정리 (memory_core.py)

01~02번에서 각자 구현했던 `ChatMemory`를 하나의 모듈로 통합하고, `level`(학습 수준)·`preferred_style`(설명 방식)까지 profile 항목을 확장, 민감정보 차단·삭제·직렬화 기능까지 모두 포함한 최종 버전. 03~07 스크립트가 모두 이 모듈을 공유해서 쓴다.

- `PROFILE_MARKERS`(표현→key)와 `PROFILE_LABELS`(key→한글 라벨)를 딕셔너리로 분리해둬서, 새로운 profile 항목을 추가할 때 이 두 딕셔너리에만 항목을 더하면 되고 나머지 로직(`remember_profile`, `update_summary`)은 수정할 필요가 없다.
- `remember_profile`이 저장 성공/실패 메시지를 반환값으로 주기 때문에, 04번 실습처럼 "정보를 기억했습니다" 같은 사용자 피드백을 별도 로직 없이 그대로 응답에 쓸 수 있다.
