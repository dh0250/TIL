# Spring Security 핵심 정리

## 1. Spring Security란?

Spring 애플리케이션에서 **인증(Authentication)**, **인가(Authorization)**, 보안 공격 방어를 담당하는 프레임워크다.

- 인증: 사용자가 누구인지 확인한다. 예) 아이디·비밀번호 로그인
- 인가: 인증된 사용자가 특정 기능을 사용할 권한이 있는지 확인한다.
- 주요 보호 기능: CSRF, 세션 고정 공격, 보안 헤더 등
- 요청은 Controller에 도달하기 전에 `SecurityFilterChain`을 통과한다.

```
Client Request
  → SecurityFilterChain
  → 인증 정보 확인
  → 접근 권한 검사
  → Controller
```

### 주요 구성요소

- `SecurityFilterChain`: URL별 접근 정책과 보안 필터 설정
- `Authentication`: 현재 사용자의 신원과 권한
- `AuthenticationManager`: 인증 요청 처리
- `AuthenticationProvider`: 실제 인증 로직 수행
- `UserDetailsService`: DB에서 사용자 조회
- `PasswordEncoder`: 비밀번호 해시 및 비교
- `SecurityContextHolder`: 현재 요청의 인증정보 보관

> 비밀번호는 원문으로 저장하지 않고 `BCryptPasswordEncoder`로 해시하여 저장한다.
>

## 2. 세션 인증과 JWT 인증 비교

| 구분 | 세션 인증 | JWT 인증 |
| --- | --- | --- |
| 인증 상태 | 서버 세션에 저장 | 토큰에 담아 클라이언트가 보관 |
| 요청 방법 | 세션 쿠키 자동 전송 | `Authorization: Bearer {token}` |
| 서버 설정 | 기본적으로 Stateful | `SessionCreationPolicy.STATELESS` |
| 로그아웃 | 서버 세션 삭제 | Refresh Token 폐기 등의 별도 설계 필요 |
| 적합한 용도 | 서버 렌더링 웹, 관리자 페이지 | REST API, 모바일 앱, 분리된 프론트엔드 |

JWT가 세션보다 무조건 안전한 것은 아니다. 토큰 탈취, 만료, 갱신, 폐기 전략을 함께 설계해야 한다.

## 3. CSRF란?

CSRF(Cross-Site Request Forgery)는 로그인된 사용자의 브라우저를 이용해 사용자가 의도하지 않은 요청을 보내는 공격이다.

브라우저는 세션 쿠키를 자동으로 전송하므로, 사용자가 공격 사이트를 방문했을 때 공격자가 은행 송금 같은 요청을 실행하게 만들 수 있다.

```
사용자가 정상 사이트에 로그인
  → 브라우저에 세션 쿠키 저장
  → 공격 사이트 방문
  → 공격 사이트가 정상 사이트로 변경 요청 전송
  → 브라우저가 세션 쿠키를 자동 첨부
```

### 방어 방법

서버가 예측하기 어려운 CSRF 토큰을 발급하고 POST, PUT, PATCH, DELETE 요청에서 함께 검증한다. 공격 사이트는 동일 출처 정책 때문에 이 값을 알기 어렵다.

### JWT API에서 CSRF를 끄는 조건

JWT를 쿠키가 아니라 `Authorization` 헤더로만 전달하고 서버가 세션을 사용하지 않는 REST API라면 브라우저가 인증정보를 자동 첨부하지 않으므로 일반적으로 CSRF를 끈다.

```java
http.csrf(csrf -> csrf.disable());
```

다만 JWT를 쿠키에 저장하면 브라우저가 자동 전송하므로 CSRF 위험이 다시 생긴다.

- JWT를 HttpOnly 쿠키에 저장: XSS 토큰 탈취 방어에 유리하지만 CSRF 대책 필요
- JWT를 localStorage에 저장: 자동 전송되지 않지만 XSS가 발생하면 탈취 가능
- 쿠키 사용 시 `HttpOnly`, `Secure`, `SameSite`와 CSRF 방어를 함께 설계한다.

## 5. Lombok 사용 원칙

```java
@Entity
@Getter
@Builder
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@AllArgsConstructor
public class Member {
    // fields
}
```

- 의존성 주입에는 `@RequiredArgsConstructor`를 사용한다.
- DTO에는 `@Getter`, `@NoArgsConstructor`를 사용할 수 있다.
- 엔티티에는 무분별한 `@Data`, `@Setter` 사용을 피한다.
- `@Data`가 만드는 `equals`, `hashCode`, `toString`은 JPA 연관관계에서 문제를 만들 수 있다.

## 6. JWT 인증 처리 흐름

### 회원가입

```
POST /api/auth/signup
  → 입력값 검증
  → username 중복 확인
  → BCrypt로 비밀번호 암호화
  → Role.USER로 Member 저장
```

클라이언트가 회원가입 요청에서 관리자 권한을 지정하게 하면 안 된다. 일반 가입자는 서버에서 강제로 `USER` 권한을 부여한다.

### 로그인

```
POST /api/auth/login
  → AuthenticationManager.authenticate()
  → UserDetailsService로 회원 조회
  → PasswordEncoder로 비밀번호 비교
  → 인증 성공
  → Access Token 발급
```

### 인증 요청

```
Authorization: Bearer {JWT}
  → JwtAuthenticationFilter
  → JWT 서명과 만료시간 검증
  → username으로 사용자 조회
  → Authentication 생성
  → SecurityContextHolder에 저장
  → URL 접근 권한 검사
  → Controller 실행
```

## 7. 핵심 SecurityConfig

```java
@Configuration
@RequiredArgsConstructor
public class SecurityConfig {

    private final JwtAuthenticationFilter jwtAuthenticationFilter;

    @Bean
    SecurityFilterChain securityFilterChain(HttpSecurity http)
            throws Exception {

        http
            .csrf(csrf -> csrf.disable())
            .formLogin(form -> form.disable())
            .httpBasic(basic -> basic.disable())
            .sessionManagement(session -> session
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            )
            .authorizeHttpRequests(auth -> auth
                .requestMatchers(
                    "/api/auth/signup",
                    "/api/auth/login"
                ).permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/members/**")
                    .hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated()
            )
            .addFilterBefore(
                jwtAuthenticationFilter,
                UsernamePasswordAuthenticationFilter.class
            );

        return http.build();
    }

    @Bean
    PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

- `permitAll()`: 인증 없이 접근 가능
- `authenticated()`: 로그인 필요
- `hasRole("ADMIN")`: 내부적으로 `ROLE_ADMIN` 검사
- `STATELESS`: 인증정보를 HTTP 세션에 저장하거나 조회하지 않음
- JWT 필터는 Spring의 사용자명·비밀번호 필터보다 앞에 추가

## 8. JWT 필터 핵심

```java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter
        extends OncePerRequestFilter {

    private final JwtTokenProvider tokenProvider;
    private final CustomUserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain chain
    ) throws ServletException, IOException {

        String token = resolveToken(request);

        if (token != null
                && tokenProvider.validateToken(token)
                && SecurityContextHolder.getContext()
                    .getAuthentication() == null) {

            String username = tokenProvider.getUsername(token);
            UserDetails user =
                userDetailsService.loadUserByUsername(username);

            Authentication authentication =
                new UsernamePasswordAuthenticationToken(
                    user,
                    null,
                    user.getAuthorities()
                );

            SecurityContext context =
                SecurityContextHolder.createEmptyContext();
            context.setAuthentication(authentication);
            SecurityContextHolder.setContext(context);
        }

        chain.doFilter(request, response);
    }
}
```

JWT Payload는 암호화된 데이터가 아니다. 누구나 디코딩할 수 있으므로 비밀번호나 개인정보를 넣으면 안 된다. 서명은 데이터를 숨기는 기능이 아니라 변조 여부를 확인하는 기능이다.

## 9. HTTP 상태 코드

- `401 Unauthorized`: 인증정보가 없거나 토큰이 잘못됨
- `403 Forbidden`: 인증은 되었지만 필요한 권한이 없음

예시:

```json
{
  "status": 401,
  "message": "인증이 필요합니다."
}
```

```json
{
  "status": 403,
  "message": "접근 권한이 없습니다."
}
```

## 10. 운영 환경에서 추가할 것

현재 구조는 Access Token 중심의 학습용 기본 구조다. 운영 환경에서는 다음 항목을 추가한다.

- 짧은 수명의 Access Token
- Refresh Token 발급 및 Rotation
- Refresh Token을 DB 또는 Redis에서 관리
- 로그아웃 시 Refresh Token 폐기
- Refresh Token 재사용 감지
- CORS 허용 출처 제한
- HTTPS 적용
- 로그인 실패 횟수 제한
- JWT 키를 환경변수 또는 Secret Manager에서 관리
- JWT 키 교체 전략
- 토큰 원문을 로그에 남기지 않기
- 공통 예외 처리 및 입력값 검증

## 한 줄 정리
