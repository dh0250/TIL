# VectorStore 실습 (07_vectorstore 01~04)

Embedding Vector를 직접 비교하던 방식에서 출발해, LangChain `InMemoryVectorStore`가 Embedding 변환·원문 연결·유사도 검색을 어떻게 대신 처리해주는지, 그리고 metadata filter·삭제·검색 결과 평가까지 단계적으로 확장.

---

# Embedding Vector 내부 흐름 확인 (01_embedding_vector_microscope.py)

"수강 신청을 취소하면 결제 금액을 환불받을 수 있습니다"와 "강의를 철회하면 지불한 돈을 돌려드립니다"처럼 표현은 다르지만 의미가 비슷한 문장을, 관련 없는 문장("오늘 점심 메뉴는 김치찌개입니다")과 함께 Embedding으로 변환해 Cosine Similarity로 비교.

```python
document_vectors = embeddings.embed_documents(DOCUMENTS)
question_vector = embeddings.embed_query(QUESTION)

for text, vector in zip(DOCUMENTS, document_vectors):
    score = cosine_similarity(question_vector, vector)
    scored_coduments.append((score, text))
```

- 문서는 `embed_documents`, 질문은 `embed_query`로 변환하지만 결국 같은 Embedding 모델을 통과하기 때문에 서로 비교 가능한 벡터가 된다는 점이 핵심 — 모델이 다르면 벡터 공간 자체가 달라 비교가 무의미해진다.
- "환불"이라는 단어가 한 번도 안 겹치는데도 "취소 → 환불"과 "철회 → 돌려주다"의 유사도가 무관한 문장보다 훨씬 높게 나와, Embedding이 표면적 단어가 아니라 의미를 벡터화한다는 것을 수치로 확인.

# VectorStore를 왜 사용하는가 (02_vectorstore_basics.py)

같은 질문("강의를 취소하면 결제한 돈을 돌려받을 수 있나요?")을 "VectorStore 없이 직접 짠 검색 로직"과 "LangChain `InMemoryVectorStore`" 두 방식으로 나란히 돌려서, VectorStore가 실제로 대신 해주는 일이 무엇인지 비교.

```python
def manual_search(question: str) -> list[tuple[Document, float]]:
    document_vectors = embeddings.embed_documents([d.page_content for d in DOCUMENTS])
    question_vector = embeddings.embed_query(question)
    results = [
        (document, cosine_similarity(question_vector, vector))
        for document, vector in zip(DOCUMENTS, document_vectors)
    ]
    return sorted(results, key=lambda item: item[1], reverse=True)[:2]

def vectorstore_search(question: str) -> list[tuple[Document, float]]:
    store = InMemoryVectorStore(create_embeddings())
    store.add_documents(DOCUMENTS, ids=[str(d.id) for d in DOCUMENTS])
    return store.similarity_search_with_score(question, k=2)
```

- `manual_search`는 Document와 Vector를 같은 index로 직접 맞춰야 하고, 전체 비교·정렬·상위 선택까지 개발자가 다 책임져야 한다 — index가 하나라도 어긋나면 엉뚱한 문서에 엉뚱한 점수가 매겨진다.
- `vectorstore_search`는 `add_documents`로 저장만 해두면 `similarity_search_with_score`가 Embedding 호출부터 정렬까지 알아서 처리 — 두 결과가 동일하게 나오는 것을 확인해 VectorStore가 수동 로직을 대체할 뿐 다른 답을 내는 게 아니라는 점을 검증.

# VectorStore 작업 API: filter·delete (03_vectorstore_operations.py)

`vectorstore_common.load_documents()`로 강의 지식베이스 전체를 실제 `Document`로 불러온 뒤, `similarity_search_with_score`에 top_k·metadata filter·삭제 옵션을 CLI 인자로 걸어보는 실습.

```python
def make_filter(topic: str | None):
    if topic is None:
        return None
    return lambda document: document.metadata.get("topic") == topic

def search(store, question, top_k, topic):
    return store.similarity_search_with_score(question, k=top_k, filter=make_filter(topic))

if args.delete_first and results:
    deleted_id = str(results[0][0].id)
    store.delete(ids=[deleted_id])
    print(format_results(search(store, args.question, args.top_k, args.topic)))
```

- `filter`에는 boolean을 반환하는 함수를 넘겨야 해서, `topic == "memory"` 같은 단순 조건도 클로저(`make_filter`)로 감싸 InMemoryVectorStore가 요구하는 시그니처에 맞춰야 한다.
- `--delete-first`로 1등 검색 결과를 `store.delete(ids=[...])`한 뒤 같은 질문으로 재검색하면 2등이었던 문서가 1등으로 올라오는 것을 확인 — VectorStore가 삭제 후에도 남은 Document들 사이에서 다시 정상적으로 유사도를 재계산한다는 것을 보여준다.
- `vectorstore_common.load_documents()`가 텍스트 파일의 각 줄을 `topic_for_text()`로 자동 분류(`memory`/`retrieval`/`course`/`llm`)해서 metadata를 붙여주기 때문에, 03번 스크립트는 filter 옵션만 CLI로 받으면 된다.

# 검색 결과 평가 데이터 설계 (04_vectorstore_search_eval.py)

질문과 "정답 문서에 포함되어야 할 키워드"를 짝지은 `EVAL_CASES` 목록을 만들어, `similarity_search_with_score` 결과에 그 키워드가 실제로 들어있는지 자동 채점하기 위한 평가 데이터셋 정의.

```python
EVAL_CASES = [
    ("외부 자료를 찾아 답변에 사용하는 방법은?", "RAG는"),
    ("파일이나 웹 페이지를 읽어오는 구성 요소는?", "Loader는"),
    ("긴 문서를 검색 가능한 작은 조각으로 나누는 이유는?", "TextSplitter는"),
    ("벡터와 원문을 함께 보관하는 검색 저장소는?", "VectorStore는"),
    ("긴 대화를 짧게 압축해서 기억하는 방법은?", "Summary memory는"),
    ("모델의 출력 형식과 말투를 원하는 대로 맞추는 방법은?", "Fine-tuning는"),
]
```

- 각 항목을 (질문, 정답 키워드) 튜플로 맞춰둬서, 다음 단계에서 `build_inmemory_store`로 store를 만들고 `EVAL_CASES`를 순회하며 검색 결과 안에 키워드가 들어있는지 확인하는 채점 로직을 이어 붙일 수 있다.

# 공통 모듈 정리 (vectorstore_common.py)

01~04 스크립트가 공유하는 `.env` 로딩, Embedding 생성, 지식베이스 로딩, VectorStore 빌드, 결과 포맷팅 함수를 모아둔 모듈.

```python
def load_documents(path: Path = DATA_PATH) -> list[Document]:
    documents = []
    for line_number, line in enumerate(path.read_text(encoding="utf-8").splitlines(), start=1):
        text = line.strip()
        if not text:
            continue
        documents.append(Document(
            id=f"class-line-{line_number:03d}",
            page_content=text,
            metadata={"source": path.name, "line": line_number, "topic": topic_for_text(text)},
        ))
    return documents

def build_inmemory_store(documents=None):
    docs = documents or load_documents()
    store = InMemoryVectorStore(create_embeddings())
    ids = store.add_documents(docs, ids=[str(doc.id) for doc in docs])
    return store, ids
```

- `topic_for_text()`가 문장에 포함된 키워드로 `memory`/`retrieval`/`course`/`llm` 중 하나를 자동으로 매겨서, 03번의 `--topic` filter가 별도 라벨링 없이 바로 동작할 수 있다.
- `create_embeddings()`가 `.env`를 프로젝트 루트와 폴더 양쪽에서 찾고 `OPENAI_API_KEY`가 없으면 바로 `SystemExit`으로 죽여서, Embedding 호출 도중이 아니라 시작 시점에 설정 문제를 알 수 있게 했다.
- `format_results()`가 rank·similarity·id·topic·source(파일명:줄번호)를 한 줄로 묶어 출력하므로, 01번의 수동 출력과 02·03번의 VectorStore 출력을 같은 형식으로 눈으로 비교할 수 있다.
