# No-RAG vs RAG 비교 (01_no_rag_vs_rag.py)

같은 질문을 "그냥 LLM에게 묻기"와 "문서에서 검색한 근거(Context)를 프롬프트에 넣어서 묻기" 두 방식으로 보내 답변 품질을 비교. RAG 프롬프트는 "Context에 있는 내용만 사용", "근거 없으면 모른다고 답", "출처를 [파일명]으로 표시" 같은 제약을 명시해 환각(hallucination)을 줄인다.

```python
def build_prompts(question: str, context: str) -> tuple[str, str]:
    no_rag_prompt = f"다음 질문에 아는 내용을 사용해 답하세요.\n\n질문: {question}"

    rag_prompt = f"""
    아래 Context에 있는 내용만 사용해 질문에 답하세요.
    근거가 없으면 '문서에서 관련 근거를 찾을 수 없습니다.'라고 답하세요.
    답변 마지막에는 사용한 출처를 [파일명]형식으로 표시하세요.

    [Context]
    {context or "검색된 근거 없음"}

    [Question]
    {question}
    """
    return no_rag_prompt, rag_prompt

documents = load_documents(DATA_DIR)
index = build_index(documents, chunk_size=220, overlap_sentences=1)
results = retrieve(index, args.question, top_k=3)
relevant_results = [r for r in results if r["score"] > 0]
context = format_context(relevant_results)
```

- `--run` 플래그가 없으면 실제 OpenAI 호출 없이 완성된 RAG 프롬프트만 출력하는 dry-run 모드로 동작해, API 비용 없이 검색 결과만 먼저 확인할 수 있다.
- 유사도 `score`가 0인 결과는 우연히 상위 k에 낀 것일 뿐이므로 필터링해서 context에서 제외한다.

# 문서 로더 (02_loader_and_documents.py)

`rag_core.load_documents()`가 폴더 안의 모든 `.txt` 파일을 읽어 `{"source": 파일명, "text": 내용}` 형태의 문서 리스트로 변환하는지 확인하는 간단한 검증 스크립트. 빈 파일은 자동으로 건너뛴다.

```python
def load_documents(data_dir: Path) -> list[dict]:
    documents = []
    for path in sorted(data_dir.glob("*.txt")):
        text = path.read_text(encoding="utf-8").strip()
        if not text:
            continue
        documents.append({"source": path.name, "text": text})
    return documents
```

- 문서별로 글자수와 앞부분 미리보기(100자)를 출력해, chunk 분할 전에 원본 데이터가 의도대로 로드됐는지 눈으로 확인하는 용도.

# Text Splitter 실험 (03_text_splitter_lab.py)

같은 문서를 `chunk_size`/`overlap_sentences` 조합 5가지(120·0, 120·1, 260·1, 50·0, 50·1)로 나눠보며 chunk 크기와 문장 overlap이 분할 결과에 어떤 영향을 주는지 비교.

```python
def split_text(text, chunk_size=220, overlap_sentences=1) -> list[str]:
    sentences = split_sentences(text)
    chunks, current = [], []
    for sentence in sentences:
        candidate = " ".join(current + [sentence])
        if current and len(candidate) > chunk_size:
            chunks.append(" ".join(current))
            current = current[-overlap_sentences:] if overlap_sentences else []
        current.append(sentence)
    if current:
        chunks.append(" ".join(current))
    return chunks
```

- 문장 단위(`split_sentences`, 마침표/느낌표/물음표/개행 기준)로 먼저 자른 뒤 문장을 이어 붙여 `chunk_size`를 넘기 직전까지 채우는 방식이라, 단어나 문장이 중간에 잘리지 않는다.
- `overlap_sentences`만큼 이전 chunk의 마지막 문장을 다음 chunk 앞에 다시 붙여 문맥 단절을 줄인다. chunk가 너무 작으면(50자) overlap이 있어도 문맥이 거의 안 살고, 너무 크면(260자) 문서 하나가 chunk 1개로 안 나뉘는 경우도 생긴다.

# 단어 빈도 벡터 + 코사인 유사도 (04_similarity_lab.py)

임베딩 API 없이 `Counter` 기반 단어 빈도 벡터로 "질문과 문장이 얼마나 비슷한가"를 코사인 유사도로 계산해보는 최소 구현. 조사(은/는/이/가 등)와 의미 없는 의문형 표현은 `STOPWORDS`로 제거한 뒤 토큰화한다.

```python
def tokenize(text: str) -> list[str]:
    words = re.findall(r"[A-Za-z0-9]+|[가-힣]+", text.lower())
    return [word for word in words if word not in STOPWORDS]

def vectorize(text: str) -> Counter:
    return Counter(tokenize(text))

def consine_similarity(vector_a: Counter, vector_b: Counter) -> float:
    common_words = set(vector_a) & set(vector_b)
    dot_product = sum(vector_a[word] * vector_b[word] for word in common_words)
    norm_a = math.sqrt(sum(v * v for v in vector_a.values()))
    norm_b = math.sqrt(sum(v * v for v in vector_b.values()))
    return dot_product / (norm_a * norm_b) if norm_a and norm_b else 0.0
```

- "수강 취소 환불 규정을 알려주세요"라는 질문에 대해, 핵심 단어(수강·취소·환불)를 공유하는 문장은 유사도가 높게, 전혀 무관한 문장(실습실 운영시간)은 0에 가깝게 나와 이 단순한 방식으로도 방향성은 잡힌다는 걸 확인.
- 다만 단어가 겹치지 않으면(동의어·바꿔 쓴 표현) 의미가 같아도 유사도가 낮게 나오는 한계가 있고, 이는 이후 OpenAI Embeddings(06)로 넘어가는 이유가 된다.

# 검색 품질 평가: Hit@k (05_retrieval_quality_lab.py)

`chunk_size`/`overlap`/`top_k` 설정 3가지를 바꿔가며 8개의 질문-정답 문서 쌍(`TEST_CASES`)에 대해 실제로 정답 문서를 검색해내는 비율(Hit@k)을 자동으로 측정하는 평가 스크립트.

```python
SETTINGS = [
    {"chunk_size": 100, "overlap": 0, "top_k": 1},
    {"chunk_size": 180, "overlap": 1, "top_k": 1},
    {"chunk_size": 260, "overlap": 1, "top_k": 2},
]

def evaluate(documents, setting) -> None:
    index = build_index(documents, chunk_size=setting["chunk_size"], overlap_sentences=setting["overlap"])
    hit_count = 0
    for question, expected_source in TEST_CASES:
        results = retrieve(index, question, top_k=setting["top_k"])
        found_sources = {r["source"] for r in results if r["score"] > 0}
        hit_count += int(expected_source in found_sources)
    print(f"Hit@{setting['top_k']}: {hit_count}/{len(TEST_CASES)} = {hit_count/len(TEST_CASES):.0%}")
```

- "검색됐다"의 기준을 단순히 top_k 안에 들었는지가 아니라 `score > 0`(질문과 겹치는 단어가 하나라도 있는지)까지로 좁혀서, 우연히 순위에 낀 무관한 chunk를 성공으로 잘못 세지 않게 했다.
- chunk를 잘게 쪼갤수록(100자) 정답 문서만 담긴 chunk가 늘어 오히려 특정 질문엔 유리하지만, top_k=1로는 애매한 질문에서 실패가 늘어 `chunk_size`와 `top_k`를 함께 조정해야 한다는 걸 수치로 확인.

# OpenAI Embeddings API로 실제 RAG (06_openai_enbeddings_rag.py)

지금까지의 단어 빈도 기반 유사도 대신, `text-embedding-3-small` 모델로 문장을 실제 임베딩 벡터로 변환해 코사인 유사도를 계산하는 버전. 문단(`\n\n`) 단위로 모아 `max_chars`를 넘기지 않게 chunk를 만드는 별도의 간단한 `split_text`를 사용한다.

```python
def create_embeddings(texts: list[str]) -> list[list[float]]:
    from openai import OpenAI
    client = OpenAI()
    response = client.embeddings.create(model="text-embedding-3-small", input=texts)
    return [item.embedding for item in response.data]

def retrieve(question: str, chunks: list[str], top_k: int) -> list[tuple[float, str]]:
    embeddings = create_embeddings([question] + chunks)
    question_embedding, chunk_embeddings = embeddings[0], embeddings[1:]
    scored_chunks = [
        (cosine_similarity(question_embedding, chunk_embedding), chunk)
        for chunk, chunk_embedding in zip(chunks, chunk_embeddings)
    ]
    return sorted(scored_chunks, reverse=True)[:top_k]
```

- 질문과 모든 chunk를 한 번의 `embeddings.create()` 호출로 같이 임베딩해서 API 호출 횟수를 줄인다(질문 1개 + chunk N개를 하나의 배치로 전송).
- 04번의 단어 빈도 벡터와 달리 의미 기반 벡터라 동의어·바꿔 쓴 표현에도 유사도가 잘 잡힐 것으로 기대되며, 04~05에서 만든 Hit@k 평가 방식을 그대로 이 임베딩 버전에도 적용해볼 수 있다.

# 공통 검색 모듈 정리 (rag_core.py)

`load_documents`(문서 로딩) → `split_text`(문장 기반 chunk 분할) → `build_index`(chunk별 벡터 생성) → `retrieve`(질문과 유사도 비교 후 top_k 반환) → `format_context`(LLM 프롬프트용 문자열 변환)로 이어지는 RAG 파이프라인 전체를 한 파일에 재사용 가능한 함수들로 모아둠. 01, 03, 04, 05 스크립트가 모두 이 모듈을 공유해서 쓴다.

- chunk마다 원본 문서명(`source`)과 chunk 번호를 같이 저장해두기 때문에, 검색 결과를 답변에 인용할 때 "어느 파일의 몇 번째 조각"인지 바로 알 수 있다.
- 오늘 실습들은 이 모듈의 `vectorize`(단어 빈도)를 그대로 쓰지만, 06번처럼 임베딩 기반으로 바꾸면 `build_index`/`retrieve`의 구조는 그대로 두고 벡터 생성 부분만 교체하면 되는 구조라 확장이 쉽다.
