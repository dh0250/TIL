# Retriever → Prompt → LLM로 RAG 완성 (05_rag_with_vectorstore.py)

지금까지는 Retriever가 찾아준 Document를 눈으로 확인하는 데 그쳤다면, 이번엔 그 Document를 `[Context]`로 Prompt에 주입해 실제로 출처 있는 LLM 답변을 만들어내는 마지막 단계.

```python
PROMPT = ChatPromptTemplate.from_messages([
    ("system",
     "너는 AI 수업 도우미다. 아래 Context만 근거로 답하라."
     "근거가 없으면 문서에서 찾을 수 없다고 답하라"
     "답변 마지막에 사용한 출처를 표시하라.\n\n[Context]\n{context}"),
    ("human", "{question}")
])

retriever = store.as_retriever(search_kwargs={"k": args.top_k})
retrieved_docs = retriever.invoke(args.question)
context = format_context(retrieved_docs)

prompt_value = PROMPT.invoke({"context": context, "question": args.question})
model = ChatOpenAI(model="gpt-4o-mini", temperature=0, max_tokens=300)
answer = model.invoke(prompt_value).content
```

- System 프롬프트에 "Context만 근거로 답하라 / 근거 없으면 모른다고 답하라 / 출처를 표시하라"는 세 가지 규칙을 못박아, LLM이 학습 지식으로 답하거나(hallucination) 근거 없이 그럴듯하게 답하는 것을 최대한 차단 — RAG의 핵심은 검색 자체가 아니라 "검색된 것 밖으로 못 나가게 가두는 프롬프트 설계"라는 점을 보여준다.
- `format_context()`가 각 Document 앞에 `[출처: source:line]`을 붙여서 Context에 밀어 넣기 때문에, 모델이 "답변 마지막에 출처 표시" 지시를 따르려면 Context 안의 이 표기를 그대로 인용하면 된다 — 출처 추적이 별도 로직 없이 프롬프트 설계만으로 해결된다.
- `--rebuild-index` 플래그로 Chroma collection을 초기화할지 기존 색인을 재사용할지 선택할 수 있어, 매 실행마다 Embedding API를 다시 호출하지 않고도 반복 테스트가 가능 — `open_chroma_store()`가 반환하는 `indexed_now`로 어느 쪽이었는지 로그에 남긴다.
- Retriever 검색 결과(`[1. Retriever가 찾은 Document]`)와 최종 LLM 답변(`[실제 LLM 답변]`)을 나란히 출력해서, "검색이 잘못됐는지" vs "검색은 맞았는데 LLM이 엉뚱하게 답했는지"를 구분해서 디버깅할 수 있는 구조.

# FastAPI 객체와 기본 Decorator (01_fastapi_start/main.py)

FastAPI 앱을 처음 띄워보면서 라우팅 decorator, 자동 JSON 변환, 허용하지 않은 Method의 405 응답, 그리고 sync/async 엔드포인트가 실제로 어떻게 다르게 동작하는지 확인.

```python
@app.get("/wait/sync", tags=["async"])
def sync_wait(seconds: Annotated[float, Query(ge=0, le=2)] = 1) -> dict[str, float | str]:
    time.sleep(seconds)
    return {"type": "sync", "waited_seconds": seconds}

@app.get("/wait/async", tags=["async"])
async def async_wait(seconds: Annotated[float, Query(ge=0, le=2)] = 1) -> dict[str, float | str]:
    await asyncio.sleep(seconds)
    return {"type": "async", "waited_seconds": seconds}
```

- Python Dictionary를 그대로 반환하면 FastAPI가 알아서 JSON으로 직렬화해주기 때문에, `/hello/{name}`처럼 간단한 엔드포인트는 별도 변환 코드 없이 dict 리턴만으로 끝난다.
- `/echo`는 POST만 등록해뒀기 때문에 같은 URL로 GET 요청을 보내면 FastAPI가 자동으로 405(Method Not Allowed)를 응답 — 라우팅이 URL뿐 아니라 Method까지 함께 매칭한다는 걸 에러로 확인.
- `time.sleep`을 쓰는 `sync_wait`는 이벤트 루프를 블로킹해서 그 시간 동안 다른 요청 처리가 막히지만, `asyncio.sleep`을 쓰는 `async_wait`는 대기 중에도 다른 요청을 동시에 처리할 수 있다 — 같은 "N초 대기"라도 sync/async 구현에 따라 서버 전체의 동시성이 달라진다는 걸 보여주기 위한 대조 엔드포인트.
- `Query(ge=0, le=2)`로 쿼리 파라미터의 허용 범위를 0~2초로 제한해둬서, 테스트 중 실수로 큰 값을 넣어도 서버가 오래 멈추지 않도록 방어.

# 경로·쿼리 매개변수와 선언 순서 (02_request_parameters/main.py)

URL Path의 일부(`item_id`, `model_name`)를 받는 방식과 `?skip=&limit=` 같은 Query 매개변수를 받는 방식을 구분하고, 고정 경로와 동적 경로가 충돌할 때의 순서 문제를 실습.

```python
@app.get("/user/me", tags=["path-order"])
async def read_current_user() -> dict[str, str]:
    return {"user_id": "me", "role": "current user"}

@app.get("/user/{user_id}", tags=["path-order"])
async def read_user(user_id: str) -> dict[str, str]:
    return {"user_id": user_id, "role": "selected user"}
```

- `/user/me`를 `/user/{user_id}`보다 먼저 선언해야 하는 이유는 FastAPI가 등록된 순서대로 라우트를 매칭하기 때문 — 순서가 바뀌면 `/user/me` 요청도 `user_id="me"`인 동적 라우트로 먼저 잡혀버려 의도한 핸들러가 실행되지 않는다.
- `ModelName(StrEnum)`으로 경로 값을 `alexnet`/`resnet`/`lenet`만 허용하도록 제한해서, 정의되지 않은 모델명이 들어오면 FastAPI가 자동으로 422를 응답 — 문자열 Path보다 Enum Path가 훨씬 안전한 이유.
- `Path(ge=1, description=...)`처럼 Path/Query에도 Pydantic 스타일 제약을 그대로 걸 수 있어서, `item_id`가 1 미만이면 핸들러 코드에 도달하기도 전에 요청이 거절된다.
- `response_model=list[ItemResponse]`를 지정해두면 반환값이 리스트 슬라이싱(`filtered[skip:skip+limit]`) 결과여도 각 항목이 `ItemResponse` 스키마로 강제 검증·직렬화된다.

# 여러 개의 Body를 한 요청에 담기 (03_request_body/main.py)

Pydantic 모델 하나를 Body로 받는 기본 패턴에서, 모델 여러 개와 단일 값을 동시에 하나의 JSON Body에 섞어 받는 패턴까지 확장.

```python
@app.post("/users/{user_id}/items", tags=["multiple-body"])
async def create_user_item(
    user_id: int,
    item: ItemCreate,
    user: UserInfo,
    importance: Annotated[int, Body(gt=0, le=5)]
) -> dict:
    return {"user_id": user_id, "item": item, "user": user, "importance": importance}
```

- FastAPI는 매개변수의 타입 선언만 보고 Path(`user_id`)·Body(`item`, `user`, `importance`)를 자동으로 구분한다 — Pydantic `BaseModel` 타입이면 Body로, 그 외 단순 타입은 기본적으로 Query로 취급하되 `Body(...)`로 명시하면 같은 JSON Body 안의 한 필드로 묶인다.
- Body 모델이 두 개(`item`, `user`) 이상이면 FastAPI가 자동으로 `{"item": {...}, "user": {...}, "importance": 3}` 형태로 요청 Body를 감싸서 파싱 — 모델이 하나일 때(`create_item`)는 감싸지 않고 필드를 Body 최상위에 바로 매핑하는 것과 대비된다.
- `Field(gt=0)`, `Field(min_length=1, max_length=50)` 같은 제약을 필드 단위로 걸어두면, dict 하나만 받을 때와 달리 잘못된 값이 들어오는 즉시 422로 걸러진다는 걸 04_validation과 연결해서 체감.

# Pydantic 검증 심화: pattern·Literal·field_validator (04_validation/main.py)

raw dict로 받을 때와 Pydantic 모델로 받을 때를 나란히 비교하고, `Literal`·정규식 `pattern`·커스텀 `field_validator`로 검증을 더 세밀하게 조이는 방법을 실습.

```python
class ProductCreate(BaseModel):
    model_config = ConfigDict(extra="forbid")
    name: str = Field(min_length=2, max_length=50)
    price: float = Field(gt=0, le=1_000_000)
    category: Literal["book", "course", "tool"]
    tags: list[str] = Field(default_factory=list, max_length=5)

    @field_validator("name")
    @classmethod
    def normalized_name(cls, value: str) -> str:
        normalized = value.strip()
        if not normalized:
            raise ValueError("상품명은 공백만 입력할 수 없습니다.")
        if "무료" in normalized:
            raise ValueError("상품명에는 '무료'를 사용할 수 없습니다.")
```

- `/examples/raw`처럼 매개변수를 그냥 `dict`로 받으면 타입·범위·필수값을 전혀 검증하지 않아 아무 JSON이나 그대로 통과 — `ProductCreate`처럼 Pydantic 모델로 받을 때와 대조하기 위해 일부러 넣어둔 "나쁜 예".
- `Literal["book", "course", "tool"]`은 Enum을 따로 정의하지 않고도 허용 값을 제한할 수 있는 더 가벼운 방법이고, `ConfigDict(extra="forbid")`는 스키마에 없는 필드가 오면 조용히 무시하는 대신 422 에러로 응답하게 만든다.
- `Query(pattern=r"^[A-Za-z0-9가-힣]+$")`로 검색어에 특수문자가 섞이면 애초에 핸들러에 도달하지 못하게 막아, SQL Injection 같은 특수문자 기반 공격 표면을 Path/Query 단계에서 먼저 줄이는 역할도 겸한다.
- `normalized_name` validator는 공백/`"무료"` 체크를 통과하면 `strip()`한 `normalized` 값을 그대로 반환해서, `name` 필드가 앞뒤 공백이 제거된 값으로 저장된다 — `field_validator`는 예외를 던지지 않는 한 반환값이 그 필드의 최종 값이 된다는 걸 보여주는 예시.

# response_model로 민감 정보 가리기 + HTTPException (05_response_error/main.py)

내부 dict에는 `password`가 섞여 있어도 API 응답에서는 이를 제외하는 `response_model`의 역할과, 존재하지 않는 리소스를 404로 응답하는 표준적인 에러 처리 패턴을 실습.

```python
class UserResponse(BaseModel):
    id: int
    username: str
    display_name: str

def find_user_or_404(user_id: int) -> dict:
    user = users.get(user_id)
    if user is None:
        raise HTTPException(status_code=404, detail="사용자를 찾을 수 없습니다.")
    return user

@app.get("/demo/unsafe-user", response_model=UserResponse, tags=["comparison"])
async def unsafe_user() -> dict:
    return users[1]
```

- 핸들러는 `password`가 포함된 원본 dict(`users[1]`)를 그대로 반환하지만, `response_model=UserResponse`가 응답 직렬화 단계에서 스키마에 없는 `password` 필드를 걸러내기 때문에 실제 HTTP 응답에는 노출되지 않는다 — 주석에 "response_model을 넣지 않으면 password가 노출된다"고 명시해둔 대조 실험용 엔드포인트.
- `find_user_or_404()`처럼 조회+예외를 하나의 함수로 묶어두면, `/users/{user_id}` 핸들러 본문은 `return find_user_or_404(user_id)` 한 줄로 끝나고 404 처리 로직이 여러 엔드포인트에서 중복되지 않는다.
- `HTTPException(status_code=404, detail=...)`을 그냥 `raise`하면 FastAPI가 이를 잡아서 `{"detail": "..."}` 형태의 JSON과 함께 404 응답으로 변환 — 일반 Python 예외와 달리 500으로 떨어지지 않고 의도한 상태 코드로 클라이언트에 전달된다.
- `list_users`에서 `response: Response`를 매개변수로 받아 `response.headers["X-Total-Count"]`를 직접 설정하는 방식으로, `response_model`이 Body 스키마를 강제하는 것과 별개로 커스텀 헤더를 추가할 수 있다는 것도 함께 확인.
