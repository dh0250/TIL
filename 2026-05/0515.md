## chap07-group-by

### 01. GROUP BY (그룹화)

특정 컬럼을 기준으로 데이터를 그룹으로 묶어 집계 데이터를 산출하는 구문.

---

### ✔ 01. Aggregate Functions (집계 함수)

* `COUNT()`: 행의 개수 산출.
* `SUM()` / `AVG()`: 합계 및 평균 계산.
* `GROUP BY` 절에 명시된 컬럼이나 집계 함수만 `SELECT` 절에 사용 가능.

```sql id="g1a2bc"
-- 카테고리별 메뉴 개수 조회
SELECT category_code, COUNT(*)
FROM tbl_menu
GROUP BY category_code;
```

### ✔ 02. HAVING & ROLLUP (조건 필터링 및 총계)

* `HAVING`: 그룹화된 결과에 대한 조건 필터링 (`WHERE`는 그룹화 전 원본 데이터 필터링).
* `WITH ROLLUP`: 각 그룹별 요약 정보와 함께 전체 총계를 마지막 행에 추가.

```sql id="f8j2kd"
-- 메뉴가 3개 이상인 카테고리만 조회
SELECT category_code, COUNT(*)
FROM tbl_menu
GROUP BY category_code
HAVING COUNT(*) >= 3;

-- 그룹별 합계와 전체 총계 표시
SELECT category_code, SUM(menu_price)
FROM tbl_menu
GROUP BY category_code WITH ROLLUP;
```

---

### 👀 핵심 요약

* 실행 순서: `FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY`
* 집계 기준: 데이터의 분포나 통계치를 확인하기 위한 필수 단계.
* 필터링 차이: 그룹 생성 전 필터링은 `WHERE`, 생성 후 결과 필터링은 `HAVING` 담당.

---

# chap08-subquery

## 01. SUBQUERY (서브쿼리)

하나의 쿼리 문장 안에 포함된 또 다른 `SELECT` 문.

---

### ✔ 01. Positioning (서브쿼리 위치 및 역할)

* `WHERE`절 서브쿼리: 비교 대상의 값을 동적으로 결정할 때 사용.
* `FROM`절 서브쿼리: '파생 테이블(Derived Table)'이라 불리며, 반드시 별칭(Alias)을 부여해야 함.
* 상관 서브쿼리: 바깥쪽 쿼리의 행 값을 서브쿼리 내부에서 참조하여 실행.

```sql id="q7mx1z"
-- '민트미역국'과 동일한 카테고리의 메뉴 조회
SELECT menu_name, category_code
FROM tbl_menu
WHERE category_code = (
    SELECT category_code
    FROM tbl_menu
    WHERE menu_name = '민트미역국'
);

-- 파생 테이블 활용 예시
SELECT MAX(count)
FROM (
    SELECT COUNT(*) AS count
    FROM tbl_menu
    GROUP BY category_code
) AS count_table;
```

---

### 👀 핵심 요약

* 동적 조회: 고정된 값이 아닌 쿼리 결과를 조건으로 활용하여 유연성 확보.
* 계층 구조: 메인쿼리와 서브쿼리의 관계를 통해 복잡한 비즈니스 로직 해결.
* 성능 고려: 대량 데이터 처리 시 상관 서브쿼리는 성능 저하의 원인이 될 수 있음.

---

# chap09-set-operators

## 01. SET OPERATORS (집합 연산자)

두 개 이상의 `SELECT` 문 결과 집합을 결합하는 연산자.

---

### ✔ 01. Union & Union All (합집합)

* `UNION`: 두 결과 집합을 합치되 중복된 행은 제거.
* `UNION ALL`: 중복을 포함하여 모든 행을 단순히 합침. (속도가 더 빠름)

```sql id="y9r4tm"
-- 카테고리가 10번이거나 가격이 9,000원 미만인 메뉴 합치기
SELECT menu_name, menu_price
FROM tbl_menu
WHERE category_code = 10

UNION

SELECT menu_name, menu_price
FROM tbl_menu
WHERE menu_price < 9000;
```

### ✔ 02. Intersection & Minus (교집합 및 차집합)

* MySQL은 `INTERSECT`와 `MINUS` 연산자를 직접 지원하지 않음.
* `INNER JOIN` 또는 `LEFT JOIN + IS NULL` 등을 활용하여 논리적으로 구현.

---

### 👀 핵심 요약

* 컬럼 일치: 결합되는 모든 `SELECT` 문의 컬럼 개수와 데이터 타입이 일치해야 함.
* 데이터 통합: 분산된 데이터를 하나의 가상 테이블 형태로 병합할 때 유용.

---

# chap10-dml

## 01. DML (Data Manipulation Language)

데이터를 삽입(Insert), 수정(Update), 삭제(Delete)하는 조작 언어.

---

### ✔ 01. Modification (입력/수정/삭제)

* `INSERT`: 테이블에 새로운 행을 추가.
* `UPDATE`: 기존 데이터의 값을 수정 (`WHERE` 절 생략 시 모든 행 수정 주의).
* `DELETE`: 테이블의 행을 삭제.
* `REPLACE`: 중복 키 발생 시 기존 데이터를 삭제하고 새 데이터로 덮어씀.

```sql id="h2n8vb"
-- 데이터 삽입
INSERT INTO tbl_menu(menu_name, menu_price, category_code, orderable_status)
VALUES ('초콜릿죽', 6500, 7, 'N');

-- 데이터 수정
UPDATE tbl_menu
SET category_code = 7
WHERE menu_code = 22;

-- 데이터 삭제
DELETE FROM tbl_menu
WHERE menu_code = 22;
```

---

### 👀 핵심 요약

* 데이터 생명주기: DB 내 실제 데이터를 관리하는 핵심 명령군.
* 안전장치: `UPDATE`/`DELETE` 수행 전 반드시 `WHERE` 조건을 검토하여 의도치 않은 손실 방지.

---

# chap11-transaction

## 01. TRANSACTION (트랜잭션)

데이터베이스의 상태를 변화시키는 작업의 최소 논리적 단위.

---

### ✔ 01. Control (제어 명령)

* `COMMIT`: 모든 작업이 성공적으로 완료되었음을 확정하고 DB에 영구 저장.
* `ROLLBACK`: 마지막 `COMMIT` 시점 이후의 작업을 취소하고 이전 상태로 복구.
* `autocommit off`: 기본 자동 저장 모드를 수동 저장 모드로 전환.

```sql id="m4k9pz"
SET autocommit = off;

START TRANSACTION;

-- 수정 및 삭제 수행
UPDATE tbl_menu
SET menu_name = '수정된 메뉴'
WHERE menu_code = 1;

DELETE FROM tbl_menu
WHERE menu_code = 2;

-- 작업 확정 또는 취소
COMMIT;

-- 또는
ROLLBACK;
```

---

### 👀 핵심 요약

* ACID 원칙: 데이터의 무결성과 일관성을 보장하기 위한 메커니즘.
* All or Nothing: 여러 단계의 작업이 하나처럼 움직여야 하는 상황(예: 계좌이체)에 필수.

---

# chap12-ddl

## 01. DDL (Data Definition Language)

데이터베이스 객체(테이블 등)의 구조를 정의하고 관리하는 언어.

---

### ✔ 01. Structure Management (생성/변경/삭제)

* `CREATE`: 새로운 테이블이나 인덱스 생성.
* `ALTER`: 기존 테이블의 컬럼 추가, 삭제, 속성 변경.
* `DROP`: 객체를 완전히 영구 삭제.
* `TRUNCATE`: 테이블의 구조는 남기고 모든 데이터만 초기화(`AUTO_INCREMENT` 리셋).

```sql id="d8sw2l"
-- 테이블 생성 및 컬럼 추가/삭제
CREATE TABLE tb1 (
    pk INT PRIMARY KEY,
    col1 VARCHAR(255)
);

ALTER TABLE tb1
ADD col2 INT NOT NULL;

DROP TABLE IF EXISTS tb1;
```

---

### 👀 핵심 요약

* 객체 관리: 데이터 자체가 아닌 '그릇'에 해당하는 스키마 구조 정의.
* 신중성: `DROP`과 `TRUNCATE`는 복구가 불가능하므로 사용 시 극도의 주의 필요.

---

# chap13-constraints

## 01. CONSTRAINTS (제약조건)

데이터의 정확성과 무결성을 보장하기 위해 테이블 컬럼에 적용하는 규칙.

---

### ✔ 01. Common Constraints (주요 제약조건)

* `PRIMARY KEY`: 행을 식별하는 고유 키 (`NOT NULL + UNIQUE`)
* `FOREIGN KEY`: 다른 테이블과의 관계를 설정하여 데이터 참조 무결성 유지.
* `UNIQUE`: 중복값 허용 안 함 (`NULL`은 허용 가능).
* `NOT NULL`: 빈 값(`NULL`) 입력을 금지.
* `CHECK`: 특정 값의 범위나 조건을 직접 지정.
* `DEFAULT`: 값을 주지 않았을 때 자동으로 입력될 기본값 설정.

```sql id="w1u7qe"
CREATE TABLE user (
    user_no INT PRIMARY KEY AUTO_INCREMENT,
    user_id VARCHAR(50) NOT NULL UNIQUE,
    age INT CHECK(age >= 19),
    status CHAR(1) DEFAULT 'Y'
);
```

---

### 👀 핵심 요약

* 무결성 방어선: 잘못된 데이터가 유입되는 것을 원천적으로 차단.
* 비즈니스 규칙 반영: 시스템에서 허용하는 데이터의 규칙을 DB 레벨에서 강제.

---

# chap14-data-type

## 01. DATA TYPE (데이터 형변환)

데이터 타입을 명시적으로 변환하여 일관된 연산을 수행하도록 돕는 함수.

---

### ✔ 01. Conversion (형변환 함수)

* `CAST(값 AS 타입)`: 값을 지정된 타입으로 변환.
* `CONVERT(값, 타입)`: `CAST`와 유사한 기능의 변환 함수.

```sql id="r5vb8n"
-- 실수를 정수로 변환하여 조회
SELECT CAST(AVG(menu_price) AS SIGNED INTEGER)
FROM tbl_menu;

SELECT CONVERT('2026-05-15', DATE);
```

---

### 👀 핵심 요약

* 연산 호환성: 서로 다른 데이터 타입 간의 연산 시 발생하는 예외 방지.
* 데이터 정제: 문자열 날짜를 실제 날짜 형식으로 바꾸는 등 데이터 활용성 증대.

---

# chap15-index

## 01. INDEX (인덱스)

데이터 조회 성능 향상을 위해 테이블 컬럼에 생성하는 색인 객체.

---

### ✔ 01. Performance Tuning (색인 관리)

* `CREATE INDEX`: 특정 컬럼에 인덱스 생성 (`WHERE` 절의 성능 비약적 상승).
* `SHOW INDEX`: 생성된 인덱스 목록 및 정보 확인.
* `EXPLAIN`: 쿼리 실행 계획을 분석하여 인덱스 사용 여부 확인.

```sql id="k2v7xp"
-- 이름 컬럼에 인덱스 생성
CREATE INDEX idx_phone_name
ON phone(phone_name);

-- 실행 계획 확인
EXPLAIN
SELECT *
FROM phone
WHERE phone_name = 'iPhone';
```

---

### 👀 핵심 요약

* 조회 최적화: 대량의 데이터에서 원하는 행을 빠르게 찾는 '목차' 역할.
* 트레이드 오프: 조회 속도는 빨라지지만, 데이터 변경(`INSERT`/`UPDATE`/`DELETE`) 시 성능 부담 발생.
