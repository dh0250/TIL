# chap05 - Spring AOP (Aspect-Oriented Programming)

---

## 개념 — AOP란?

> 공통 관심사(Cross-Cutting Concern)를 핵심 비즈니스 로직에서 분리해,
>
>
> 필요한 실행 지점에 자동으로 끼워 넣는 프로그래밍 방식.
>

---

## 문제 상황 (AOP 적용 전)

```java
public Map<Long, MemberDTO> selectMembers() {
    System.out.println("selectMembers 메소드 실행");  // 공통 관심사 (로그)
    return memberDAO.selectMembers();                  // 핵심 관심사 (업무 로직)
}

public MemberDTO selectMember(Long id) {
    System.out.println("selectMember 메소드 실행");   // 동일한 로그 코드 반복
    return memberDAO.selectMember(id);
}
```

→ 로그·트랜잭션·보안 같은 공통 기능이 모든 메서드에 중복 작성됨

---

## 해결 (AOP 적용 후)

공통 기능을 Aspect 클래스 하나에 모아두고, 지정한 실행 지점에 자동 삽입

---

## 핵심 용어

| 용어 | 설명 |
| --- | --- |
| Aspect | 공통 관심사를 모아둔 클래스 (`@Aspect`) |
| Advice | 실제 실행될 공통 기능 메서드 |
| Pointcut | Advice를 어디에 적용할지 정의하는 조건 (`@Pointcut`) |
| JoinPoint | Advice가 끼어들 수 있는 실행 지점 (메서드 호출 등) |
| Target | 공통 기능이 적용될 실제 대상 객체 (`MemberService`) |
| Proxy | Target 앞에서 요청을 가로채는 대리 객체 |
| Weaving | Target에 Advice가 적용되는 과정 |

---

## 도메인 구조

```
MemberDTO   (id, name)
MemberDAO   (@Repository — 인메모리 Map으로 회원 데이터 보관)
MemberService (@Service — Target, Proxy가 감싸는 대상)
LoggingAspect (@Aspect @Component — 공통 기능 모음)
```

## 설정 — ContextConfiguration.java

```java
@Configuration

public class ContextConfiguration {
}
```

> `@EnableAspectJAutoProxy`
>
>
> Spring이 Target 앞에 Proxy 객체를 자동 생성해
>
> 메서드 실행 전후에 공통 기능을 끼워 넣도록 활성화
>

## Pointcut 표현식

```java
@Pointcut("execution(* com.ohgiraffers.section01.*Service.*(..))")
public void logPointcut() {}
```

```
execution( * com.ohgiraffers.section01.*Service.*(..) )
           ↑ 리턴타입(모두)
                                         ↑ *Service : 이름이 Service로 끝나는 모든 클래스
                                                   ↑ .* : 모든 메서드
                                                      ↑ (..) : 파라미터 개수·타입 무관
```

## Advice 종류

### @Before — 메서드 실행 전

```java
@Before("LoggingAspect.logPointcut()")
public void logBefore(JoinPoint joinPoint) {
    System.out.println("Before : " + joinPoint.getTarget());     // 대상 객체
    System.out.println("Before : " + joinPoint.getSignature());  // 메서드 시그니처

    if (joinPoint.getArgs().length > 0) {
        System.out.println("Args : " + joinPoint.getArgs()[0]);  // 파라미터
    }
}
```

---

### @After — 메서드 실행 후 (성공/예외 무관)

```java
@After("logPointcut()")  // 동일 클래스 내에서는 클래스명 생략 가능
public void logAfter(JoinPoint joinPoint) {
    System.out.println("After : " + joinPoint.getSignature());
}
```

---

### @AfterReturning — 정상 반환 후

```java
@AfterReturning(pointcut = "logPointcut()", returning = "result")
public void logAfterReturning(JoinPoint joinPoint, Object result) {
    System.out.println("After Returning result : " + result);

    // 반환값 가공도 가능
    if (result instanceof Map) {
        ((Map<Long, MemberDTO>) result).put(100L, new MemberDTO(100L, "반환 값 가공"));
    }
}
```

> `returning` 속성명 = 매개변수명 일치 필수
>
>
> `JoinPoint`는 반드시 첫 번째 매개변수
>

---

### @AfterThrowing — 예외 발생 후

```java
@AfterThrowing(pointcut = "logPointcut()", throwing = "exception")
public void logAfterThrowing(Throwable exception) {
    System.out.println("After Throwing : " + exception);
}
```

> `throwing` 속성명 = 매개변수명 일치 필수
>

---

### @Around — 메서드 실행 전후 전체 제어

```java
@Around("logPointcut()")
public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
    System.out.println("Around Before : " +
joinPoint.getSignature().getName());

    Object result = joinPoint.proceed();  // 실제 Target 메서드 실행

    System.out.println("Around After : " +
joinPoint.getSignature().getName());

    return result;
}
```

> `ProceedingJoinPoint` 사용 (일반 `JoinPoint`와 구분)
>
>
> `proceed()` 를 호출해야 실제 메서드가 실행됨 — 호출 안 하면 실행 차단 가능
>

---

## 실행 흐름 (selectMember 호출 시)

```
memberService.selectMember(2L) 호출
        ↓
Spring Proxy가 요청 가로챔
        ↓
Pointcut 조건 확인 → *Service.* 에 해당 → 통과
        ↓
Around Before
Before
        ↓
실제 MemberService.selectMember(2L) 실행
        ↓
Around After
After
AfterReturning (정상) 또는 AfterThrowing (예외)
        ↓
결과 반환
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext("com.ohgiraffers.section01");

MemberService memberService = context.getBean("memberService",
MemberService.class);

memberService.selectMembers();     // 전체 조회
memberService.selectMember(2L);    // id=2 조회 (정상)
memberService.selectMember(3L);    // id=3 조회 → 예외 발생 → @AfterThrowing 실행
```

### 핵심 요약

| Advice | 어노테이션 | 실행 시점 |
| --- | --- | --- |
| Before | `@Before` | 메서드 실행 전 |
| After | `@After` | 메서드 실행 후 (항상) |
| AfterReturning | `@AfterReturning` | 정상 반환 후 |
| AfterThrowing | `@AfterThrowing` | 예외 발생 후 |
| Around | `@Around` | 실행 전후 전체 감쌈 (가장 강력) |

## Advice 적용 우선순위 (실행 순서)

```
Around Before
  └── Before
        └── 실제 메서드
  └── After
  └── AfterReturning / AfterThrowing
Around After
```

## OOP vs AOP

|  | OOP | AOP |
| --- | --- | --- |
| 관심사 | 클래스 단위 모듈화 | 횡단 관심사 모듈화 |
| 주 목적 | 핵심 비즈니스 로직 캡슐화 | 로그·트랜잭션·보안 등 공통 기능 분리 |
| 적용 방식 | 상속·인터페이스 | Proxy + Weaving |

# chap01 - Spring Boot Request Mapping

---

## Request Mapping

▎ 클라이언트의 HTTP 요청(URL + HTTP 메서드)을

▎ 어떤 컨트롤러의 어떤 메서드가 처리할지 연결하는 매핑 설정.

---

## 프로젝트 구조

```
Application.java                  (Spring Boot 진입점)
ClassMappingTestController.java   (@RequestMapping 클래스 레벨 적용)
MethodMappingTestController.java  (HTTP 메서드별 매핑 적용)
```

---

## 핵심 어노테이션

| 어노테이션 | 설명 |
| --- | --- |
| `@RestController` | `@Controller + @ResponseBody` — JSON 응답 자동 직렬화 |
| `@RequestMapping` | URL 경로를 클래스 또는 메서드에 매핑 |
| `@GetMapping` | HTTP GET 요청 처리 |
| `@PostMapping` | HTTP POST 요청 처리 |
| `@PutMapping` | HTTP PUT 요청 처리 |
| `@DeleteMapping` | HTTP DELETE 요청 처리 |
| `@PathVariable` | URL 경로의 동적 값 추출 |
| `ResponseEntity<T>` | HTTP 상태코드 + 응답 바디를 함께 반환 |

---

## 클래스 레벨 `@RequestMapping` — `ClassMappingTestController`

클래스에 `@RequestMapping`을 붙이면 하위 모든 메서드의 URL prefix가 됨

```java
@RestController
@RequestMapping("/api/v1/orders")   // 모든 메서드에 공통으로 붙는 URL
public class ClassMappingTestController {

    @GetMapping                              // GET /api/v1/orders
    public ResponseEntity<Map<String, Object>> findOrders() { ... }

    @GetMapping("/{orderNumber}")            // GET /api/v1/orders/{orderNumber}
    public ResponseEntity<Map<String, Object>> selectOrderDetail(
            @PathVariable int orderNumber) { ... }
}
```

▎ `@PathVariable` — URL 경로의 `{orderNumber}` 자리를 메서드 파라미터로 바인딩

---

## 메서드 레벨 HTTP 동사 매핑 — `MethodMappingTestController`

CRUD 작업을 HTTP 메서드로 구분

```java
@RestController
@RequestMapping("/api/v1/menus")
public class MethodMappingTestController {

    @GetMapping                              // GET    /api/v1/menus
    public ResponseEntity<...> findMenus() { ... }

    @PostMapping                             // POST   /api/v1/menus
    public ResponseEntity<...> registMenu() {
        return ResponseEntity.status(201).body(response);   // 201 Created
    }

    @PutMapping("/{menuCode}")               // PUT    /api/v1/menus/{menuCode}
    public ResponseEntity<...> modifyMenu(@PathVariable int menuCode) { ... }

    @DeleteMapping("{menuCode}")             // DELETE /api/v1/menus/{menuCode}
    public ResponseEntity<...> deleteMenu(@PathVariable int menuCode) { ... }
}
```

---

## `ResponseEntity`

HTTP 응답 전체(상태코드 + 헤더 + 바디)를 직접 제어하는 객체

```java
ResponseEntity.ok(response)               // 200 OK + 바디
ResponseEntity.status(201).body(response) // 201 Created + 바디
```

---

## API 엔드포인트 전체 목록

| HTTP 메서드 | URL | 동작 |
| --- | --- | --- |
| GET | `/api/v1/menus` | 메뉴 목록 조회 |
| POST | `/api/v1/menus` | 메뉴 등록 |
| PUT | `/api/v1/menus/{menuCode}` | 메뉴 수정 |
| DELETE | `/api/v1/menus/{menuCode}` | 메뉴 삭제 |
| GET | `/api/v1/orders` | 주문 목록 조회 |
| GET | `/api/v1/orders/{orderNumber}` | 주문 상세 조회 |

---

## 요청 흐름

```
클라이언트 → HTTP 요청 (GET /api/v1/menus)
        ↓
DispatcherServlet이 요청 수신
        ↓
HandlerMapping이 URL + HTTP 메서드 → 컨트롤러 메서드 매칭
        ↓
MethodMappingTestController.findMenus() 실행
        ↓
ResponseEntity → JSON 직렬화 → HTTP 응답 반환
```

---

## RESTful URL 설계 패턴

```
/api/v1/menus          ← 컬렉션 (복수형 명사)
/api/v1/menus/{id}     ← 단일 리소스 (경로 변수로 식별)
/api/v2/menus          ← 버전 관리 (v1, v2...)
```

▎ URL에 동사를 쓰지 않고 HTTP 메서드(GET/POST/PUT/DELETE)로 행위를 표현하는 것이 REST 원칙

---

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| 클래스 `@RequestMapping` | 하위 메서드 전체에 URL prefix 적용 |
| `@GetMapping` 등 | HTTP 동사별 메서드 매핑 단축 어노테이션 |
| `@PathVariable` | `{변수명}` → 메서드 파라미터 자동 바인딩 |
| `ResponseEntity` | 상태코드를 직접 지정할 때 사용 (특히 201, 404 등) |
| `@RestController` | 모든 메서드의 반환값을 JSON으로 자동 변환 |

# chap02 - Spring Boot Handler Method

---

## Handler Method

▎ 클라이언트의 HTTP 요청을 받아 처리하는 컨트롤러 메서드.

▎ 요청 파라미터·헤더·쿠키·바디·세션 등 다양한 데이터를 어노테이션으로 간편하게 메서드 파라미터에 바인딩할 수 있다.

---

## 프로젝트 구조

```
Application.java             (Spring Boot 진입점)
MenuDTO.java                 (요청/응답 데이터 객체)
MenuRequestController.java   (핸들러 메서드 예제 컨트롤러)
```

---

## 도메인 — `MenuDTO`

```java
public class MenuDTO {
    private String name;
    private int price;
    private int categoryCode;
    private String orderableStatus;
}
```

---

## 핵심 어노테이션 정리

| 어노테이션 | 바인딩 대상 | 설명 |
| --- | --- | --- |
| `@RequestParam` | 쿼리스트링 파라미터 | URL `?key=value` 형식의 값 추출 |
| `@PathVariable` | URL 경로 변수 | `/{변수명}` 자리의 값 추출 |
| `@RequestBody` | 요청 본문 (Body) | JSON → Java 객체 자동 역직렬화 |
| `@RequestHeader` | 요청 헤더 | 특정 헤더 또는 전체 헤더 Map 추출 |
| `@CookieValue` | 쿠키 | 특정 쿠키 값 추출 |
| `HttpSession` | 세션 객체 | 현재 요청의 세션 직접 주입 |

---

## `@RequestParam` — 쿼리스트링 파라미터

```java
// GET /api/v1/menus?keyword=coffee&categoryCode=2&page=1
@GetMapping
public ResponseEntity<Map<String, Object>> findMenus(
        @RequestParam(required = false) String keyword,       // 없어도 null 허용
        @RequestParam(defaultValue = "0") int categoryCode,  // 없으면 기본값 0
        @RequestParam(defaultValue = "1") int page) {        // 없으면 기본값 1
    ...
}
```

## `@RequestParam` 주요 속성

```
required      = true(기본값) → 없으면 400 Bad Request
                false        → 없으면 null

defaultValue  = "값"         → 없을 때 사용할 기본값
```

---

## `@RequestParam Map` — 파라미터 전체 수집

```java
// GET /api/v1/menus/search?name=라떼&minPrice=3000
@GetMapping("/search")
public ResponseEntity<Map<String, Object>> searchMenus(
        @RequestParam Map<String, String> parameters) {  // 모든 쿼리 파라미터를 Map으로 수집
    ...
}
```

▎ 파라미터 이름·개수가 고정되지 않을 때 Map으로 한꺼번에 받는 방식

---

## `@PathVariable` — URL 경로 변수

```java
// GET /api/v1/menus/{menuCode}
@GetMapping("/{menuCode}")
public ResponseEntity<Map<String, Object>> findMenu(
        @PathVariable int menuCode) {    // {menuCode} → int 자동 변환
    ...
}
```

---

## `@RequestBody` — JSON 요청 본문 역직렬화

```java
// POST /api/v1/menus
// Body: {"name":"아메리카노","price":2000,"categoryCode":1,"orderableStatus":"Y"}
@PostMapping
public ResponseEntity<Map<String, Object>> createMenu(
        @RequestBody MenuDTO menu) {   // JSON → MenuDTO 객체로 자동 변환
    ...
    return ResponseEntity.status(HttpStatus.CREATED).body(response);  // 201 Created
}
```

▎ `Content-Type: application/json` 헤더 필수

▎ `@RequestBody String body` 로 받으면 JSON 문자열 그대로 수신 가능

---

## `@RequestHeader` — 요청 헤더

```java
// GET /api/v1/menus/request-info
@GetMapping("request-info")
public ResponseEntity<Map<String, Object>> requestInfo(
        @RequestHeader(value = "user-agent", required = false) String userAgent,  // 특정 헤더
        @RequestHeader Map<String, String> headers,                               // 전체 헤더 Map
        @CookieValue(value = "JSESSIONID", required = false) String sessionId     // 쿠키
) { ... }
```

## `@RequestHeader`

```
value     : 헤더 이름 (대소문자 무관)
required  : false 지정 시 헤더가 없어도 null 허용
```

## `@CookieValue`

```
value     : 쿠키 이름
required  : false 지정 시 쿠키가 없어도 null 허용
```

```java
    session.setAttribute("id", id);  // 세션에 키-값 저장
    ...
}
```

```java
// GET /api/v1/menus/session
@GetMapping("/session")
public ResponseEntity<Map<String, Object>> readSession(HttpSession session) {
    session.getAttribute("id");   // 저장된 값 조회
    ...
}
```

## 세션 만료

```java
// DELETE /api/v1/menus/session
@DeleteMapping("/session")
public ResponseEntity<Map<String, Object>> removeSession(HttpSession session) {
    session.invalidate();   // 세션 전체 만료 (로그아웃 등에 사용)
    ...
}
```

## 요청 흐름

```
클라이언트 → HTTP 요청 (POST /api/v1/menus, Body: JSON)
        ↓
DispatcherServlet이 요청 수신
        ↓
HandlerMapping → MenuRequestController.createMenu() 매칭
        ↓
HttpMessageConverter가 JSON Body → MenuDTO 역직렬화 (@RequestBody)
        ↓
createMenu(MenuDTO menu) 실행
        ↓
ResponseEntity (201 Created + JSON Body) 반환
        ↓
HttpMessageConverter가 Map → JSON 직렬화
        ↓
클라이언트에 HTTP 응답 전달
```

## `@RequestParam` vs `@PathVariable` vs `@RequestBody`

|  | `@RequestParam` | `@PathVariable` | `@RequestBody` |
| --- | --- | --- | --- |
| 위치 | URL 쿼리스트링 (`?key=val`) | URL 경로 (`/{id}`) | 요청 본문 (Body) |
| HTTP 메서드 | 주로 GET | GET / DELETE | POST / PUT |
| 복수 수신 | `Map<String, String>` 가능 | 불가 | DTO / String |
| 필수 여부 제어 | `required = false` | 경로에 있으면 항상 필수 | 항상 필수 |

---

## 핵심 요약

| 항목 | 내용 |
| --- | --- |
| `@RequestParam` | 쿼리스트링 → 파라미터 바인딩. `required`·`defaultValue` 속성으로 유연하게 제어 |
| `@PathVariable` | URL 경로의 `{변수}` → 파라미터 자동 바인딩 |
| `@RequestBody` | 요청 JSON 바디 → DTO 자동 역직렬화. `Content-Type: application/json` 필수 |
| `@RequestHeader` | 특정 헤더 또는 전체 헤더 Map 추출 |
| `@CookieValue` | 쿠키 이름으로 값 추출 |
| `HttpSession` | 파라미터로 선언하면 Spring이 자동 주입. `setAttribute` / `getAttribute` / `invalidate` 사용 |
