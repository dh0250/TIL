### 2. UPDATE (데이터 수정)

데이터베이스에 저장된 기존 레코드를 특정 조건에 맞춰 변경하는 작업.

---

✔ 01. PreparedStatement를 통한 동적 값 설정

- 수정할 대상의 식별자(PK)와 변경할 데이터를 위치 홀더(?)에 안전하게 매핑.
- executeUpdate(): 쿼리 실행 후 성공적으로 수정된 행의 개수(int)를 반환받아 결과 판별.

✔ 02. 효율적인 자원 해제

- JDBCTemplate.close(): 작업 완료 후 Connection과 Statement를 안전하게 닫아 메모리 누수 방지.

```java
// 메뉴 코드(PK)를 기준으로 이름과 가격 수정
String query = prop.getProperty("updateMenu");
pstmt = con.prepareStatement(query);

pstmt.setString(1, changedMenu.getMenuName());
        pstmt.setInt(2, changedMenu.getMenuPrice());
        pstmt.setInt(3, changedMenu.getMenuCode());

result = pstmt.executeUpdate();
```

---

👀 핵심 요약

- 정확한 타겟팅: WHERE 절에 고유 키(PK)를 사용하여 의도치 않은 데이터 변경 방지.
- 보안 유지: 사용자 입력값이 포함되는 수정 작업에서도 PreparedStatement 사용을 원칙으로 함.
- 결과 확인: executeUpdate() 반환값을 통해 수정 성공 여부를 판단 가능.

### 3. DELETE (데이터 삭제)

불필요하거나 잘못된 데이터를 데이터베이스에서 영구적으로 제거하는 작업.

---

✔ 01. 식별자를 이용한 단일 행 삭제

- 메뉴 코드와 같은 고유 번호를 입력받아 특정 데이터만 정확하게 삭제 처리.
- 영향받은 행이 1이면 성공, 0이면 대상 없음으로 로직 처리.

✔ 02. XML 쿼리 연동

- 삭제 쿼리 역시 외부 XML 파일에서 관리하여 프로젝트의 일관성 유지.

```java
// 특정 메뉴 코드를 이용한 삭제
String query = prop.getProperty("deleteMenu");

pstmt = con.prepareStatement(query);
pstmt.setInt(1, menuCode);

result = pstmt.executeUpdate();
```

---

#### 👀 핵심 요약

- 신중한 처리: 삭제 작업은 복구가 어려우므로 WHERE 조건의 정확성이 매우 중요함.
- 일관성: CRUD 전체 단계에서 XML 매퍼와 JDBCTemplate을 활용한 동일한 패턴의 코드 구현.
- 안정성: PreparedStatement를 사용하여 SQL Injection 방지 가능.

### 4. DAO & INTEGRATION (객체화 및 통합)

DB 접근 로직을 전담하는 DAO 클래스를 도입하고, 실제 비즈니스 흐름에 맞게 기능을 통합하는 단계.

---

✔ 01. DAO (Database Access Object)

- CRUD 연산을 담당하는 메소드들을 하나의 클래스(MenuDAO)로 캡슐화.
- 데이터베이스 접근 로직과 비즈니스 로직을 분리하여 코드의 재사용성 극대화.

✔ 02. 데이터 조회와 등록의 결합

- 신규 등록 전 유효한 카테고리 목록을 먼저 조회하여 사용자에게 제공하는 등 실제 서비스 흐름 구현.
- List 형태의 반환값을 활용하여 다중 행 데이터를 처리.

```java
// DAO를 활용한 통합 로직 예시
MenuDAO menuDAO = new MenuDAO();

menuDAO.insertNewMenu(con, newMenu);
```

# [데이터 모델링]

## 1. Data Modeling 개요 & ERD (기본 개념)

복잡한 현실 세계의 비즈니스 프로세스를 단순화, 추상화, 명확화하여 데이터베이스 구조로 표현하는 정형화된 방법론.

### ✔ 01. 데이터 모델의 표기법 및 관계 정의

- IE 표기법(범용적)과 Barker 표기법(구체적)을 사용해 엔터티 간의 연관성을 시각화.
- 관계의 3대 요소: **관계명**(행위), **차수**(Cardinality: 1:1, 1:M, M:N), **선택성**(Optionality: 필수 필수`Mandatory`, 선택`Optional`).
- 다대다(M:M) 관계는 데이터 구조적 모순을 방지하기 위해 반드시 교차 엔터티(Association Entity)를 도입하여 1:M 관계로 해소해야 함.

```
[카테고리] -------◀ [메뉴]  (IE 표기법 예시: 1 대 다 관계)
- 하나의 카테고리는 여러 개의 메뉴를 가질 수 있음
```

### 👀 핵심 요약

- **독립성 유지**: 데이터의 정의(Schema)와 실제 물리적 저장 구조를 분리하여 시스템 유지보수성 극대화.
- **의사소통 도구**: 현업 사용자, 개발자, 모델러 간의 모호함을 제거하고 동일한 비즈니스 규칙을 공유.

---

## 2. 개념 모델 (Conceptual Model)

요구사항 분석 단계 직후, 핵심 엔터티와 주요 속성을 도출하여 시스템의 전체적인 뼈대를 잡는 상위 수준의 설계 단계.

### ✔ 01. 핵심 엔터티(Entity) 중심의 도출

- 업무 기술서, 전표, 인터뷰 자료 등의 '명사'를 분석하여 핵심 데이터 집합을 정의.
- 복잡한 세부 속성이나 정규화에 집중하기보다, 시스템 전체를 관통하는 굵직한 개념적 관계를 정립하는 데 목적이 있음.

### ✔ 02. 이해관계자 간의 합의

- 논리 모델링 기간의 약 1/3을 투자할 만큼 충분한 소통이 필요하며, 데이터 모델에 행위(로직)가 아닌 순수 데이터만 포함되도록 제어.

```
[고객] ---- (주문한다) ---- [상품]  (개념적 ERD 흐름)
```

### 👀 핵심 요약

- **정확한 방향성**: 대규모 프로젝트에서 개발자 및 의사결정자가 업무의 큰 틀을 직관적으로 파악할 수 있도록 도움.
- **해석의 일치**: 이해관계자 간 명확한 소통을 통해 이후 단계에서 발생할 수 있는 구조적 재작업 리스크를 방지.

## 3. 논리 모델 (Logical Model)

특정 DBMS의 특성을 고려하지 않고, 비즈니스 규칙을 완벽하게 반영하여 데이터의 일관성과 무결성을 확보하는 상세 설계 단계.

### ✔ 01. 이상 현상(Anomaly) 제거와 정규화(Normalization)

- 중복 데이터로 인해 발생하는 **삽입, 갱신, 삭제 이상**을 방지하기 위해 정규화 수행.
- **1정규형(1NF)**: 모든 속성은 원자 값(Atomic value)을 가져야 함 (다중값/복합속성 제거).
- **2정규형(2NF)**: 부분 함수 종속성 제거 (식별자 전체에 종속적이어야 함).
- **3정규형(3NF)**: 이행적 함수 종속성 제거 (기본키가 아닌 일반 속성 간의 종속 제거).

---

### 1. 1정규형 (1NF) 예시

- **개념**: 관계형 데이터베이스의 테이블에서 **모든 속성은 반드시 하나의 원자 값(Atomic Value)만 가져야 한다**는 규칙입니다. 즉, 하나의 칸(Cell)에 다중 값이나 복합 속성이 들어간 경우를 제거하는 단계입니다.

#### ❌ 1정규형 위반 (다중 값 발생)

[회원] 테이블의 기본키는 `회원번호`입니다.

- 홍길동 회원의 취미 칸을 보면 '등산, 낚시, 테니스'라는 3개의 값이 쉼표(,)로 구분되어 한 칸에 동시에 들어가 있어 위반입니다.

```
[회원] 회원번호(PK), 회원명, 취미(등산, 낚시, 테니스 -> 위반)
```

> **문제점**: 낚시가 취미인 회원을 검색하려면 `LIKE '%낚시%'` 같은 복잡하고 성능이 느린 쿼리를 써야 합니다. 또한, 취미 하나를 삭제하거나 추가할 때도 글자 전처리 과정이 필요해 데이터 관리가 매우 까다로워집니다.
>

### ▼ 1정규화 완료 (엔터티 분리 또는 인조식별자 도입)

다중 값을 유발하던 취미 속성을 별도의 **[회원별취미]** 엔터티로 분리하고, 부모 테이블의 `회원번호`를 자식 테이블의 외래키(FK)이자 기본키(PK)의 일부로 묶어 다대다 관계를 해소합니다.

```
[회원]     회원번호(PK), 회원명
[회원별취미] 회원번호(FK, PK), 취미명(PK)
```

*(※ 만약 복합키 구조가 불편하다면, `회원별취미번호` 같은 새로운 인조식별자(PK)를 만들고 `회원번호(FK)`와 `취미명`은 일반 속성으로 두는 방식으로도 해결할 수 있습니다.)*

---

### 2. 2정규형 (2NF) 예시

- **개념**: 1정규형을 만족하고, 기본키가 2개 이상인 복합키일 때 **부분 함수 종속성을 제거**하는 단계입니다. 기본키의 일부분에만 종속되는 일반 속성을 별도 엔터티로 분리합니다.

#### ❌ 2정규형 위반 (부분 함수 종속 발생)

[주문상세] 테이블의 기본키가 `주문번호` + `상품번호` 복합키인 상황입니다.

- `주문수량`은 주문번호와 상품번호 둘 다 알아야 결정되므로 정상입니다.
- 하지만 `상품명`과 `단가`는 복합키 전체가 아니라, 일부분인 `상품번호`에만 종속되므로 위반입니다.

```
[주문상세] 주문번호(PK), 상품번호(PK), 주문수량, 상품명(위반), 단가(위반)
```

### ▼ 2정규화 완료 (엔터티 분리)

복합 식별자의 일부에만 종속되던 속성들을 떼어내어 별도의 **[상품]** 엔터티로 분리합니다.

```
[주문상세] 주문번호(PK), 상품번호(FK, PK), 주문수량
[상품]     상품번호(PK), 상품명, 단가
```

---

### 3. 3정규형 (3NF) 예시

- **개념**: 2정규형을 만족하고, 기본키가 아닌 **일반 속성 간의 종속 관계(이행적 함수 종속성)를 제거**하는 단계입니다. `A ➔ B`이고 `B ➔ C`일 때 `A ➔ C`가 성립하는 주 종속 관계를 분리합니다.

#### ❌ 3정규형 위반 (이행적 함수 종속 발생)

[회원] 테이블의 기본키는 `회원번호`입니다.

- `등급`은 `회원번호`에 종속됩니다. (`회원번호` ➔ `등급`)
- 하지만 `할인율`은 기본키가 아닌 일반 속성인 `등급`에 의해 결정됩니다. (`등급` ➔ `할인율`)
- 결국 `회원번호` ➔ `등급` ➔ `할인율`이라는 징검다리 구조가 생겨 위반입니다.

```
[회원] 회원번호(PK), 회원명, 등급, 할인율(위반)
```

### ▼ 3정규화 완료 (엔터티 분리)

일반 속성 간의 종속을 유발하던 `등급`과 `할인율`을 별도의 **[등급정책]** 엔터티로 분리합니다.

```
[회원]     회원번호(PK), 회원명, 등급(FK)
[등급정책] 등급(PK), 할인율
```

---

### 👀 정규화 핵심 요약 (한눈에 보기)

- **1정규형(1NF)**: 하나의 칸에는 오직 하나의 값만 들어간다. (**원자성** 확보)
- **2정규형(2NF)**: 복합키(`A+B`) 구조에서 일반 속성들이 키 전체(`A+B`)에만 매달리게 한다. 일부분(`A`)에만 매달리면 분리한다. (**부분 함수 종속** 제거)
- **3정규형(3NF)**: 주인이 아닌 일반 속성들끼리 징검다리 서열을 나누지 않게 독립시킨다. (**이행적 함수 종속** 제거)

```
// 1정규형 위반 (하나의 속성에 여러 값 존재) -> 엔터티 분리 또는 인조식별자(PK) 도입으로 해결
[회원] 회원번호(PK), 회원명, 취미(등산, 낚시, 테니스 -> 위반)
  ▼
[회원별취미] 회원번호(FK), 취미명(PK) 형태로 다정규화 처리
```

### 👀 핵심 요약

- **데이터 무결성**: 정규화를 통해 데이터의 중복을 원천 차단하고 구조적 안정성 확보.
- **DBMS 독립성**: 오라클, MySQL 등 특정 제품군에 종속되지 않는 가장 이상적인 논리 구조 완성.

---

## 4. 물리 모델 (Physical Model)

논리 모델을 바탕으로, 실제 선택한 토겟 DBMS의 특성과 하드웨어 성능을 고려하여 테이블, 인덱스, 뷰 등을 구체적으로 설계하는 단계.

### ✔ 01. 성능 최적화를 위한 반정규화(Denormalization)

- 시스템 성능(조회 속도) 향상을 위해 정규화된 모델을 의도적으로 통합, 분할, 추가하는 작업.
- **테이블 수직 분할**: 자주 검색되는 속성 고성능 그룹과 나머지 속성 그룹을 분리(1:1 관계 형성).
- **테이블 수평 분할**: 데이터 건수가 너무 많을 때, 특정 기준(예: 날짜, 지역)으로 행을 쪼갬(파티셔닝).

### ✔ 02. 무결성 및 오브젝트 정의

- 데이터 타입(VARCHAR2, NUMBER 등)과 정확한 크기를 지정.
- 성능 중심의 **인덱스(Index)** 설계, 보안 및 편의성을 위한 **뷰(View)** 정의, FK 제약조건 설정을 통한 무결성 보장.

```
[논리: 슈퍼/서브타입 관계] 
          ▼ 변환 (물리적 요구 성능에 따름)
1. One Table (Single Table): 하나의 테이블로 통합 (조회 속도 빠름, 공간 낭비 가능)
2. Two Table (Table per Child): 서브타입별로 각각 테이블 생성
```

### 👀 핵심 요약

- **성능 중심**: 논리적 완성도보다는 인덱스, 반정규화, 저장 공간 등 실질적인 데이터베이스 처리 속도에 집중.
- **제한적 변경**: 물리적 최적화를 수행하더라도 기존 논리 모델 구조의 변화는 **10% 이내**로 통제하는 것이 원칙.
