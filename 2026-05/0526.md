# chap01 - Spring Bean Configuration

---

# ⚪ Section 00 — Plain Java

▎ Spring 없이 `new` 키워드로 직접 객체를 생성하는 전통적인 방식.

## 특징

- 객체 생성·소멸·재사용을 개발자가 직접 관리
- 클래스 간 결합도가 높아 유지보수 어려움
- IoC 이전의 baseline

```java
MemberDAO memberDAO = new MemberDAO();          // 직접 생성

MemberDTO member = memberDAO.selectMember(1);
MemberDTO newMember = new MemberDTO(3, "user03", "pass03", "봉미선");
memberDAO.insertMember(newMember);
```

---

# 🟢 Section 01 — XML Configuration

▎ XML 파일에 `<bean>` 태그로 객체를 선언하면 컨테이너가 생성·관리.

## 특징

- `id` 미지정 시 클래스명 첫 글자를 소문자로 변환한 이름 자동 부여
- `<constructor-arg>`로 생성자 주입
    - `index`(순서)
    - `name`(파라미터명)

---

## spring-context.xml

```xml
<bean id="member" class="com.ohgiraffers.common.MemberDTO">
    <constructor-arg index="0" value="1"/>
    <constructor-arg name="id" value="user01"/>
    <constructor-arg index="2">
        <value>pass01</value>
    </constructor-arg>
    <constructor-arg name="name">
        <value>홍길동</value>
    </constructor-arg>
</bean>
```

---

## Application.java

```java
ApplicationContext context =
    new GenericXmlApplicationContext(
        "section01/xmlconfig/spring-context.xml"
    );

// id + 타입 함께 전달 → 캐스팅 불필요
MemberDTO member = context.getBean("member", MemberDTO.class);
```

---

# 🟡 Section 02 — Java Configuration

▎ XML 대신 Java 클래스(`@Configuration`)로 Bean을 명시적으로 등록.

## 특징

- `@Configuration`

  → 해당 클래스가 Bean 설정 클래스임을 선언

- `@Bean`

  → 메서드 반환값을 컨테이너에 Bean으로 등록

- `name` 속성 생략 시 메서드명이 Bean ID로 자동 사용

---

## ContextConfiguration.java

```java
@Configuration
public class ContextConfiguration {

    @Bean(name = "member")
    public MemberDTO getMember() {
        return new MemberDTO(1, "user01", "pass01", "신짱구");
    }
}
```

---

## Application.java

```java
// 설정 클래스의 메타 정보를 생성자에 전달
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        ContextConfiguration.class
    );

MemberDTO member = context.getBean("member", MemberDTO.class);
```

---

# 🔴 Section 03 — Annotation Configuration

▎ `@ComponentScan`으로 패키지를 자동 탐색하여 Bean 등록.

## 특징

- `basePackages` 하위의
    - `@Component`
    - `@Service`
    - `@Repository`
    - `@Controller`

  클래스를 자동 스캔

- 설정 클래스에 `@Bean` 메서드 없이도 Bean 등록 완료
- `getBeanDefinitionNames()`

  → 컨테이너에 등록된 모든 Bean 이름을 배열로 반환


---

## ContextConfiguration.java

```java
@Configuration
@ComponentScan(basePackages = "com.ohgiraffers.common")
public class ContextConfiguration {

    // @Bean 메서드 불필요
    // 자동 스캔으로 대체
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        ContextConfiguration.class
    );

// 등록된 Bean 목록 확인
String[] beanNames = context.getBeanDefinitionNames();

for (String beanName : beanNames) {
    System.out.println(beanName);
    // memberDAO 등 자동 등록된 Bean 출력
}

MemberDAO memberDAO =
    context.getBean("memberDAO", MemberDAO.class);

memberDAO.selectMember(1);

memberDAO.insertMember(
    new MemberDTO(3, "user03", "pass03", "봉미선")
);
```

# ✅ 핵심 요약

| 항목 | 내용 |
| --- | --- |
| IoC | 객체 제어권을 개발자 → 컨테이너로 이전 |
| Bean | Spring 컨테이너가 관리하는 객체 단위 |
| XML 방식 | 설정이 외부 파일에 분리 — 변경 시 재컴파일 불필요 |
| Java Config | 타입 안전, IDE 자동완성 지원 |
| Annotation | 코드 최소화, 자동 탐색으로 대규모 프로젝트에 적합 |

# ⚠️ Bean 조회 시 주의

▎ `getBean("id")` 단독 사용 시 `Object` 반환 → 캐스팅 필요.

```java
Object obj = context.getBean("member");
MemberDTO member = (MemberDTO) obj;
```

---

▎ `getBean("id", 클래스.class)` 형태 사용 시 캐스팅 없이 안전하게 사용 가능.

```java
MemberDTO member =
    context.getBean("member", MemberDTO.class);
```

# chap02 - Spring Dependency Injection (DI)

---

# ⚪ Section 00 — Plain Java

▎ Spring 없이 `new` 키워드로 직접 객체를 생성하고 의존 관계를 직접 연결하는 방식.

## 특징

- 객체 생성과 조립을 개발자가 전부 담당
- `MemberDTO`가 `Account(interface)`에 의존

  → 구현 클래스를 직접 선택해서 주입

- IoC/DI 이전의 baseline

---

## 공통 모델 구조

```
Account (interface)
    └── PersonalAccount (구현체)  ←── MemberDTO 가 보유
```

---

```java
// Account 구현체를 직접 생성
Account account =
    new PersonalAccount(20, "110-234-567890");

// MemberDTO 에 setter 로 직접 주입
MemberDTO member = new MemberDTO();

member.setSequence(1);
member.setName("신짱구");
member.setPersonalAccount(account);   // 개발자가 직접 조립

System.out.println(
    member.getPersonalAccount().deposit(10000)
);

System.out.println(
    member.getPersonalAccount().getBalance()
);

System.out.println(
    member.getPersonalAccount().withDraw(5000)
);
```

# 🟢 Section 01 — XML Configuration

▎ XML 파일에 `<bean>` 태그로 객체를 선언하면 컨테이너가 생성·조립.

## 특징

- 구현 클래스(`class` 속성)는 인터페이스가 아닌 실제 구현 클래스로 작성
- 다른 Bean을 주입할 때는 `value` 대신 `ref` 속성 사용
- 생성자 주입

  → `<constructor-arg name="..." value="..."/>`

- Setter 주입

  → `<property name="..." value="..."/>`


---

## spring-context.xml

```xml
<!-- PersonalAccount Bean 등록 -->
<bean id="account"
      class="com.ohgiraffers.common.PersonalAccount">

    <constructor-arg index="0" value="20"/>
    <constructor-arg index="1" value="110-234-567890"/>
</bean>

<!-- 1. 생성자 주입 -->
<bean id="member"
      class="com.ohgiraffers.common.MemberDTO">

    <constructor-arg name="sequence" value="1"/>
    <constructor-arg name="name" value="홍길동"/>

    <constructor-arg name="personalAccount">
        <ref bean="account"/>
    </constructor-arg>

</bean>

<!-- 2. Setter 주입 -->
<!--
<bean id="member"
      class="com.ohgiraffers.common.MemberDTO">

    <property name="sequence" value="1"/>
    <property name="name" value="홍길동"/>
    <property name="personalAccount" ref="account"/>

</bean>
-->
```

---

## Application.java

```java
ApplicationContext context =
    new GenericXmlApplicationContext(
        "section01/xmlconfig/spring-context.xml"
    );

// id + 타입 함께 전달 → 캐스팅 불필요
MemberDTO member =
    context.getBean("member", MemberDTO.class);

System.out.println(
    member.getPersonalAccount().deposit(10000)
);

System.out.println(
    member.getPersonalAccount().getBalance()
);

System.out.println(
    member.getPersonalAccount().withDraw(5000)
);
```

# 🟡 Section 02 — Java Configuration

▎ XML 대신 Java 클래스(`@Configuration`)로 Bean을 명시적으로 등록하고 메서드 호출로 의존성 주입.

## 특징

- `@Configuration`

  → Bean 설정 클래스 선언

- `@Bean`

  → 메서드 반환값을 컨테이너에 Bean으로 등록

- `name` 생략 시 메서드명이 Bean ID로 자동 사용
- 다른 Bean 주입 시 해당 `@Bean` 메서드를 직접 호출

  → Spring이 싱글톤 보장


---

## ContextConfiguration.java

```java
@Configuration
public class ContextConfiguration {

    @Bean
    public Account accountGenerator() {

        return new PersonalAccount(
            20,
            "110-234-567890"
        );
    }

    @Bean
    public MemberDTO memberGenerator() {

        // 생성자 주입
        // accountGenerator() 호출로 Bean 조립
        return new MemberDTO(
            1,
            "신짱구",
            accountGenerator()
        );
    }
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        ContextConfiguration.class
    );

// 타입만으로 Bean 조회
// 등록된 Bean이 하나일 때 가능
MemberDTO member =
    context.getBean(MemberDTO.class);

System.out.println(
    member.getPersonalAccount().deposit(10000)
);

System.out.println(
    member.getPersonalAccount().getBalance()
);

System.out.println(
    member.getPersonalAccount().withDraw(5000)
);
```

# 🔴 Section 03 — Layered Architecture

▎ 3계층 구조(Controller → Service → Repository)에 Spring DI를 적용한 실전 패턴.

---

## 계층 구조

```
MemoController
    └── MemoService
            └── MemoRepository (interface)
                    └── MemoryMemoRepository (구현체)
```

---

## 특징

- 각 계층은 인터페이스에만 의존

  → 구현체 교체 시 상위 계층 코드 변경 불필요

- 모든 의존성을 생성자 주입으로 처리
- `ContextConfiguration`에서 Bean 조립 순서
    - Repository
    - Service
    - Controller

---

## ContextConfiguration.java

```java
@Configuration
public class ContextConfiguration {

    @Bean
    public MemoRepository memoRepository() {

        // 구현체 선택은 여기서만
        return new MemoryMemoRepository();
    }

    @Bean
    public MemoService memoService() {

        // 생성자 주입
        return new MemoService(
            memoRepository()
        );
    }

    @Bean
    public MemoController memoController() {

        // 생성자 주입
        return new MemoController(
            memoService()
        );
    }
}
```

---

## 각 계층 클래스

### Repository 인터페이스

```java
public interface MemoRepository {

    List<MemoDTO> findAll();

    MemoDTO save(String content);
}
```

---

### Repository 구현체

```java
public class MemoryMemoRepository
        implements MemoRepository {

    private final List<MemoDTO> memos =
        new ArrayList<>();

    private int sequence = 1;

    public MemoryMemoRepository() {

        save("Servlet에서 직접 만들던 객체 흐름 복습");
        save("Spring Container가 객체를 대신 조립");
    }

    // findAll(), save() 구현 ...
}
```

---

### Service 계층

```java
public class MemoService {

    private final MemoRepository memoRepository;

    public MemoService(
        MemoRepository memoRepository
    ) {

        // 생성자 주입
        this.memoRepository = memoRepository;
    }

    public List<MemoDTO> selectMemos() {

        return memoRepository.findAll();
    }

    public MemoDTO registMemo(String content) {

        return memoRepository.save(content);
    }
}
```

---

### Controller 계층

```java
public class MemoController {

    private final MemoService memoService;

    public MemoController(
        MemoService memoService
    ) {

        // 생성자 주입
        this.memoService = memoService;
    }

    public List<MemoDTO> findMemos() {

        return memoService.selectMemos();
    }

    public MemoDTO createMemo(String content) {

        return memoService.registMemo(content);
    }
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        ContextConfiguration.class
    );

// 최상위 계층만 꺼내서 사용
MemoController memoController =
    context.getBean(MemoController.class);

System.out.println(
    memoController.findMemos()
);

// [
//   MemoDTO{id=1, content='Servlet에서 직접 만들던 객체 흐름 복습'},
//   MemoDTO{id=2, content='Spring Container가 객체를 대신 조립'}
// ]
```

# ✅ 핵심 요약

| 항목 | 내용 |
| --- | --- |
| DI | 의존 객체를 직접 생성하지 않고 외부에서 주입 받음 |
| 생성자 주입 | 필드를 `final`로 선언 가능, 불변성·테스트 용이성 확보 |
| Setter 주입 | 선택적 의존성에 사용, XML `<property>` 태그 |
| XML 방식 | 설정이 외부 파일에 분리 — 재컴파일 없이 Bean 교체 가능 |
| Java Config | 타입 안전, IDE 자동완성, `@Bean` 메서드 직접 호출로 조립 |
| Layered Architecture | 인터페이스 기반 계층 분리 → 구현체 교체 시 상위 계층 무변경 |

## ⚠️ Bean 주입 시 주의

## ▎ ref vs value

▎ 다른 Bean을 참조할 때는 반드시 `ref` 사용.

---

### ❌ 잘못된 예

```xml
<constructor-arg
    name="personalAccount"
    value="account"/>
```

---

### ✅ 올바른 예

```xml
<constructor-arg name="personalAccount">
    <ref bean="account"/>
</constructor-arg>
```

---

## ▎ getBean()

▎ 타입 파라미터 함께 전달하면 캐스팅 불필요.

---

### id 단독 사용

```java
MemberDTO m =
    (MemberDTO) context.getBean("member");
```

---

### id + 타입 사용 (권장)

```java
MemberDTO m =
    context.getBean("member", MemberDTO.class);
```

---

### 타입만 사용

```java
// 등록된 Bean 이 1개일 때만 가능
MemberDTO m =
    context.getBean(MemberDTO.class);
```

---

## BookDTO.java

```java
public class BookDTO {

    private int sequence;
    private int isbn;
    private String title;

    // 기본 생성자
    // 전체 생성자
    // getter · setter · toString
}
```

---

## BookDAO.java (인터페이스)

```java
public interface BookDAO {

    List<BookDTO> selectBookList();

```

---

## BookDAOImpl.java

```java
@Repository("bookDAO")   // Bean 이름을 "bookDAO"로 명시
public class BookDAOImpl implements BookDAO {

    private Map<Integer, BookDTO> bookList =
        new HashMap<>();

    public BookDAOImpl() {

        // 샘플 데이터 초기화
        bookList.put(
            1,
            new BookDTO(1, 123456, "자바의 정석")
        );

        bookList.put(
            2,
            new BookDTO(2, 654321, "짱구는 못말려")
        );
    }

    @Override
    public List<BookDTO> selectBookList() {

        return new ArrayList<>(bookList.values());
    }

    @Override
    public BookDTO selectOneBook(int sequence) {

        return bookList.get(sequence);
    }
}
```

# 🟢 Section 01-1 — Constructor Injection (생성자 주입)

▎ 의존 객체를 생성자 파라미터로 전달받아 주입하는 방식. Spring 공식 권장.

## 특징

- 생성자가 하나뿐이면 `@Autowired` 생략 가능 (`Spring 4.3+`)
- 주입받은 필드를 `final`로 선언

  → 불변성 보장

- 객체 생성 시점에 의존성이 확정

  → NPE 방지


---

## BookService.java

```java
@Service("bookServiceConstructor")
public class BookService {

    private final BookDAO bookDAO;

    // 생성자가 하나뿐이므로
    // @Autowired 생략 가능
    public BookService(BookDAO bookDAO) {

        this.bookDAO = bookDAO;
    }

    public List<BookDTO> selectAllBooks() {

        return bookDAO.selectBookList();
    }

    public BookDTO searchBookBySequence(int sequence) {

        return bookDAO.selectOneBook(sequence);
    }
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        "com.ohgiraffers"
    );

BookService bookService =
    context.getBean(
        "bookServiceConstructor",
        BookService.class
    );

bookService.selectAllBooks()
           .forEach(System.out::println);
```

---

# 🟡 Section 01-2 — Field Injection (필드 주입)

▎ 필드에 `@Autowired`를 선언하면 컨테이너가 직접 값을 주입.

## 특징

- 코드가 가장 짧고 단순
- `final` 선언 불가

  → 불변성 보장 안 됨

- Spring 컨테이너 없이 테스트할 때 주입 수단이 없음

  → 테스트 어려움

- 실무에서는 권장하지 않음

---

## BookService.java

```java
@Service("bookServiceField")
public class BookService {

    @Autowired
    private BookDAO bookDAO;

    public List<BookDTO> selectAllBooks() {

        return bookDAO.selectBookList();
    }

    public BookDTO searchBookBySequence(int sequence) {

        return bookDAO.selectOneBook(sequence);
    }
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        "com.ohgiraffers"
    );

BookService bookService =
    context.getBean(
        "bookServiceField",
        BookService.class
    );

bookService.selectAllBooks()
           .forEach(System.out::println);
```

# 🔴 Section 01-3 — Setter Injection (세터 주입)

▎ `@Autowired`가 붙은 setter 메서드를 통해 의존성을 주입.

## 특징

- 의존성이 선택적(optional)일 때 유용
- 주입 전 메서드 호출 시 NPE 위험 존재
- 런타임에 의존 객체를 교체할 수 있는 유연성 제공

---

## BookService.java

```java
@Service("bookServiceSetter")
public class BookService {

    private BookDAO bookDAO;

    @Autowired
    public void setBookDAO(BookDAO bookDAO) {

        this.bookDAO = bookDAO;
    }

    public List<BookDTO> selectAllBooks() {

        return bookDAO.selectBookList();
    }

    public BookDTO searchBookBySequence(int sequence) {

        return bookDAO.selectOneBook(sequence);
    }
}
```

---

## Application.java

```java
ApplicationContext context =
    new AnnotationConfigApplicationContext(
        "com.ohgiraffers"
    );

BookService bookService =
    context.getBean(
        "bookServiceSetter",
        BookService.class
    );

bookService.selectAllBooks()
           .forEach(System.out::println);

System.out.println(
    bookService.searchBookBySequence(2)
);
```

| 생성자 주입 | 생성자 | ✅ | ⭐ 강력 권장 |
| 필드 주입 | 필드 | ❌ | ⚠️ 비권장 |
| 세터 주입 | setter 메서드 | ❌ | 선택적 의존성에만 |

---

## ⚠️ 스테레오타입 어노테이션 비교

▎ 모두 `@Component`의 하위 어노테이션 — 계층별 역할을 명확히 표현.

```java
@Component      // 범용 (특정 계층 없음)

@Repository     // 데이터 접근 계층 (DAO)

@Service        // 비즈니스 로직 계층

@Controller     // 프레젠테이션 계층 (Web MVC)
```

---

▎ `AnnotationConfigApplicationContext("com.ohgiraffers")`

▎ 패키지 스캔 시 위 네 가지 어노테이션이 붙은 클래스를 모두 Bean으로 자동 등록.

---

## ⭐ 생성자 주입이 권장되는 이유

- `final` 필드 사용 가능

  → 불변성 보장

- 객체 생성 시 의존성 강제

  → NPE 예방

- Spring 없이도 단위 테스트 가능

```java
BookService service =
    new BookService(new BookDAOImpl());
```

→ 순수 Java 환경에서도 테스트 가능하여 유지보수성과 테스트 용이성이 높음.
